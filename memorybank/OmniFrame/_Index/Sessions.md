- [[2026-05-29]] — **Cycle-count simultaneous-claim aisle/zone thrash (diagnosis)** ([[Debug/Investigate-Cycle-Count-Simultaneous-Claim-Aisle-Thrash-2026-05-29]]). Two operators hitting "Pull Next Count" at the same time get **swapped into each other's aisle**. Root cause: `claim_next_cycle_count` takes **no claim-time serializing lock**, and `FOR UPDATE SKIP LOCKED LIMIT 200` hands concurrent claimers disjoint windows ~200 serpentine positions apart; the live head SH(1–111)+SI(112–133)+SM(134–330) means one 200-row lock spans **3 aisles**, so the losing racer lands in SM (pos 201), jumping past SH+SI. The per-aisle cap + `one_counter_per_zone` can't block it — occupancy is heartbeat-derived (`worker_heartbeats` 5-min freshness + committed `in_progress`), blind to the other's uncommitted in-flight claim (READ COMMITTED); the zone advisory lock lives in the trigger and fires too late to steer selection. Evidence: near-simul (≤3 s) claims switch aisle **50.8% vs 11.8%** (4.3×); 131 near-simul pairs → **55 (42%) clean swaps, 57 (44%) both-linear** (coin-flip); named e.g. 17:29:08 William SH→SM / Marvin SM→SH (1.72 s), 16:44:03 David SI→SN / William SN→SI (0.02 s). `sticky_zone=false` (Phase-2 stickiness is a dead branch). Fix direction: per-org/per-zone `pg_advisory_xact_lock` *before* the Phase-2 scan; detect in-flight claims via `assigned_to`/`reservation_started_at` not heartbeat freshness; add real Phase-2 per-worker aisle stickiness. Diagnosis only — no code/migrations; server-side fix touches no `supabase.channel(...)`. **Railway deploy-log corroboration** (running deployment `81783c80`, not the SKIPPED `ecfcd99a`): both racing claims logged **2.65 ms apart** (David CC-4944 / William CC-4370 @16:44:05; Marvin CC-3990 @17:29:08); the Phase-2 `FOR UPDATE SKIP LOCKED LIMIT 200` scan logs `rows_affected=200 elapsed≈1.01s` as a slow-statement WARN on **every** pull-claim (→ ~1 s unserialized race window — confidence raised to very high); **zero `ZONE_LOCKED`** (displacement, not abort). Compounding stressors: `2× PoolTimedOut` on `get_fleet` right after the 16:44 swap + WS `work_ws_lagged` drops; incidental out-of-scope `trigger_evaluator: bad NOTIFY … missing field row_id` ERROR stream on the SAP `sap_agent_jobs` pipeline. **Fix implemented** (build green, migration applied, **not deployed**): per-org `pg_advisory_xact_lock('cyclecount_claim:<org>')` at the start of Phase 2 (serializes claims; deadlock-free since it's the outermost lock and the trigger's zone-lock is always acquired after); `OCCUPIED_AISLES_SQL` rewritten off committed assignment state (no heartbeat lag); real Phase-2 aisle stickiness via `get_sticky_aisle_tx` + a `$3` sticky tiebreaker in BOTH the SQL ORDER BY and the Rust `sort_by`, enabled for the org by **migration 332** (`sticky_zone=true`). `cargo build` green + 4 new `db::queries::tests` pass.
