# Fix SQCDP Metric Editor — Uncaught ZodError on Save

## Purpose / Context

Users reported that clicking **Create metric** in the SQCDP board's editor
dialog produced an uncaught `ZodError: [{ ... "path": ["title"], "message":
"Title is required" }]` in the browser console and the form never reached
the submit path — the new metric was never created. The same crash also
blocked every other react-hook-form + Zod form in the app (e.g. user role
change, labor-management dialogs, settings panels) whenever validation
failed.

The bundle name in the stack (`feature-admin-permissions-DwgbBbPB.js`) was
misleading — Vite happened to ship the shared `zodResolver` runtime in that
chunk, but the bug applied to **every** call site of
`zodResolver()` in the codebase.

## Root cause

Version incompatibility between two pinned dependencies:

- `@hookform/resolvers@3.10.0` (installed) catches `ZodError` instances by
  testing `Array.isArray(error.errors)` (Zod v3 shape).
- `zod@4.2.1` (installed) renamed the `errors` getter on `ZodError` to
  `issues` and removed `errors` entirely. So `error.errors === undefined`
  for every Zod v4 error.

When the resolver's `try/catch` ran:

```js
function isZod3Error(r) { return Array.isArray(r?.errors); }
// ...
try { schema.parseAsync(values) }
catch (r) {
  if (isZod3Error(r)) return { values: {}, errors: ... };
  throw r;  // <- Zod v4 error fell through here
}
```

The `ZodError` was re-thrown as an uncaught promise rejection, react-hook-form
never received an `errors` object, the form silently failed, and the
`<FormMessage>` slots never rendered inline messages.

Reproduction (Node REPL):

```js
const { z } = require('zod');           // 4.2.1
const { zodResolver } = require('@hookform/resolvers/zod');  // 3.10.0
const schema = z.object({ title: z.string().min(1) });
zodResolver(schema)({ title: '' }, undefined, { shouldUseNativeValidation: false, criteriaMode: 'firstError' })
  .catch(e => console.log('UNCAUGHT:', e.name)); // -> UNCAUGHT: ZodError
```

## Fix

1. Bumped `@hookform/resolvers` from `^3.10.0` to `^5.4.0` in
   `package.json`. v5 ships a Zod v4-aware codepath
   (`isZod4Error = error instanceof z4.$ZodError`) and reads `error.issues`
   instead of `error.errors`.
2. The v5 resolver's static type is stricter — it returns
   `Resolver<z4.input<T>, Context, z4.output<T>>` instead of v3's
   `Resolver<T, any, T>`. For schemas with `.default(...)` (e.g.
   `z.boolean().default(true)`), `z.input<T>` ≠ `z.output<T>` and the
   existing `useForm<z.infer<typeof schema>>(...)` calls no longer match
   the resolver's signature.
   - 25 form files exhibited this typecheck error. Added
     `as never` cast on the resolver expression in each (matches the
     existing convention from `sqcdp-editor-dialog.tsx`, which had already
     carried the cast for a previous related reason). The runtime is fixed;
     the cast is only there to keep `tsc -b` green without rewriting every
     `useForm` generic to `useForm<z.input<typeof schema>, any, z.output<typeof schema>>`.
3. Removed the now-unnecessary `as never` cast in
   `src/features/shift-productivity/production-boards/boards/sqcdp/components/sqcdp-editor-dialog.tsx`
   line 903 because the SQCDP `metricSchema` has no `.default(...)` calls
   (input = output), so the resolver type matches `useForm<MetricFormValues>`
   cleanly.

## Follow-ups

- **Type-cast cleanup.** The `as never` cast hides the resolver's true
  signature. The canonical Zod v4 + react-hook-form pattern is
  `useForm<z.input<typeof schema>, any, z.output<typeof schema>>(...)`.
  Migrating each of the 25 files this way is the "correct" fix but cascades
  through `Control<...>` typing on subcomponents — best done as a focused
  follow-up PR per feature area.
- **Pre-existing bundle budget breach.** `warehouse-location-map`,
  `feature-admin`, and `feature-rf-interface` already exceed 500 KB on
  `main`. Not introduced by this fix but worth chasing separately.
- **Lockfile drift.** Local `.tmp/users_const.ts` lint error is unrelated
  and pre-existing (not tracked in git); ignore.
