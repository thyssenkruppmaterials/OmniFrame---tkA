# Fix — OmniBelt rollout exposed missing `<SearchProvider>` after `__root.tsx` regressed to scaffold

## Problem

Immediately after the OmniBelt MVP rollout (P0–P9) landed, the dev
server began throwing a runtime error on every authenticated route:

```
Error: useSearch has to be used within <SearchContext.Provider>
    at useSearch (search-context.tsx:41:11)
    at Search (search.tsx:13:23)
```

`<Search>` (`src/components/search.tsx`) is the global ⌘K launcher
component imported by ~42 page modules across the app
(`src/routes/_authenticated/**/*.tsx`, `src/features/admin/**`,
`src/components/*-management.tsx`, etc.). It calls `useSearch()` on
mount, which throws if no `SearchContext.Provider` is present in the
tree above it. The error was caught by TanStack Router's
`CatchBoundaryImpl`, which is why the rest of the app stayed
visually responsive but every page hit the boundary.

## Root cause

**`src/routes/__root.tsx` had been reverted to a TanStack Router
scaffold placeholder** — a one-component `<React.Fragment>` with a
`<div>Hello "__root"!</div>` + `<Outlet />`. The reversion stripped:

- `ToastSettingsProvider`
- `SearchProvider` ← *this caused the reported error*
- `AppUpdateBanner` / `NavigationProgress`
- `<Toaster />`
- `<OmniBeltHost />` (added in P3)
- `errorComponent` / `notFoundComponent` registrations
- `ReactQueryDevtools` / `TanStackRouterDevtools`

Likely cause: an editor / file-routing scaffold tool overwrote the
file at ~09:14 AM on 2026-05-24 (one of the HMR updates in the
dev-server log mentions `__root.tsx` followed by a server restart).
The OmniBelt P0–P9 rollout did **not** cause this — it had carefully
extended the canonical provider stack to mount `<OmniBeltHost />`
between `<Outlet />` and `<Toaster />` **inside** `<SearchProvider>`,
exactly as the design spec requires. The rollout simply *surfaced*
the regression because the user navigated to the new
`/admin/omnibelt` route immediately after the file reverted, which
walked them into the first `<Search>`-using page they had not yet
loaded.

The displayed stack trace was a red herring: line 41 of
`search-context.tsx` is the `throw` itself, line 13 of `search.tsx`
is the `useSearch()` call. Neither file changed during the rollout.

## Fix

Restored `src/routes/__root.tsx` to the HEAD-committed provider
stack and additively re-applied the P3 `<OmniBeltHost />` mount.
Diff vs. HEAD is a 4-line additive insertion (mount + a comment
explaining the placement constraint):

```tsx
<SearchProvider>
  <AppUpdateBanner />
  <NavigationProgress />
  <Outlet />
  {/* P3 — OmniBelt site-wide launcher. Mounted INSIDE
      <SearchProvider> so any tool shell can read the
      command-menu context, and as a sibling of <Outlet />
      so it survives route transitions. */}
  <OmniBeltHost />
  <Toaster />
  {import.meta.env.MODE === 'development' && (
    <>
      <ReactQueryDevtools buttonPosition='bottom-left' />
      <TanStackRouterDevtools position='bottom-right' />
    </>
  )}
</SearchProvider>
```

This matches the canonical provider hierarchy already documented in
[[AppProviders - Provider Stack]] (the `__root.tsx` block at the
top of the file).

## Verification

```bash
pnpm tsc -b                         # ✅ clean (exit 0)
pnpm vitest run \
  src/features/omnibelt \
  src/features/admin/omnibelt-dashboard \
  src/context/                      # ✅ 380/380 passed
pnpm build                          # ✅ built in 10.85s, PWA precache OK
```

No new linter errors on `src/routes/__root.tsx`. Bundle-size
warnings on the build are pre-existing and unrelated
(`warehouse-location-map`, `feature-admin`, `exceljs`).

## Side effects checked

- **Provider order** — restored exact HEAD ordering
  (`ToastSettingsProvider` → `SearchProvider`), no changes to
  upstream `main.tsx` providers (`QueryClientProvider` →
  `ThemeProvider` → `FontProvider` → `UnifiedAuthProvider` →
  `RouterProvider`). Auth flow unaffected.
- **`OmniBeltHost` placement** — sibling of `<Outlet />`, child of
  `<SearchProvider>`. `useOmnibeltConfigInvalidator` (depends on
  `workServiceWs` + auth context) and `useUnifiedAuth` (lives in
  `main.tsx`) both have their providers above this point.
- **Toast settings + Toaster** — `ToastSettingsProvider` still
  wraps everything, `<Toaster />` still sits inside `SearchProvider`
  as before.
- **Error / NotFound components** — `errorComponent` and
  `notFoundComponent` re-registered (they were also dropped by the
  scaffold revert).
- **Route tree** — `src/routeTree.gen.ts` is auto-generated by
  TanStack Router's Vite plugin; no manual changes needed and
  none made.
- **OmniBelt tests** — all 380 OmniBelt + admin-dashboard +
  `src/context/` vitest cases pass (the `OmniBeltHost.test.tsx`
  suite mocks the host body, so it doesn't depend on the root
  provider stack).
- **Realtime / Supabase channels** — no new `supabase.channel(...)`
  callsites introduced; unchanged.
