# Omni-Agent (Headless SAP Agent)

## Purpose
A lightweight, **install-free** Windows console process that lets the OmniFrame web app (any Chrome/Edge browser in a Citrix desktop) drive SAP GUI automation via COM.

Designed for **Tier 4 Citrix** — non-persistent sessions with no IT help and no profile persistence. Users download a ZIP, extract, run the `.exe`, and delete when done. **Nothing is installed.**

## Architecture
```
Chrome on Citrix  ──HTTPS─→  OmniFrame (Railway FastAPI)
       │
       │  fetch('http://127.0.0.1:8765/...')
       ▼
OmniFrame_Agent.exe (console window)
       │
       ▼
SAP GUI (via COM Scripting Engine)
```

## Folder: `omni_agent/`
| File | Purpose |
|---|---|
| `agent.py` | FastAPI app: all endpoints, Citrix detection, SAP COM automation, live progress tracking |
| `requirements.txt` | fastapi, uvicorn[standard], pywin32, requests, pydantic, pyinstaller |
| `build_exe.bat` | Windows PyInstaller script — builds with `--console` (not `--windowed`) for visibility |
| `README.md` | Architecture + distribution docs |

Output: `dist/OmniFrame_Agent.exe` (~20 MB self-contained binary)

**Multi-session companion (Phase G):** `build_exe.bat` also produces `dist/OmniFrame_AgentMaster.exe` — a windowed CustomTkinter supervisor that spawns up to six worker `OmniFrame_Agent.exe` processes on ports 8765–8770. Legacy operators can still run the worker EXE alone; see [[Implement-Phase-G-Packaging-DualExe]].

**End-user Connect edition (Phase H.4):** `dist/OmniFrame_Connect.exe` — windowed floating widget supervising one worker on port 8765 with self-update via `connect_manifest.json`; shipped in `OmniFrame_Connect.zip`. See [[Implement-Phase-H4-Connect-Self-Update-Packaging]].

## Run Model: No Install
1. User downloads `OmniFrame_Agent.zip` from the download link
2. Extracts (Windows built-in) to any folder (Downloads works fine)
3. Double-clicks `OmniFrame_Agent.exe` — a **console window** opens showing live logs
4. Leaves window open for session
5. Closes window to stop; deletes `.exe` to uninstall

No `%LOCALAPPDATA%` directory, no Startup shortcut, no registry entries. This was a deliberate choice after the earlier auto-install approach was rejected (Citrix wipes everything anyway).

## REST API (localhost:8765)
| Method | Path | Purpose |
|---|---|---|
| GET | `/health` | Lightweight detection probe (polled by UI every 3s) |
| GET | `/status` | Detailed agent state + Citrix info |
| POST | `/sap/connect` | Attach to SAP GUI + auto-pick first valid session |
| POST | `/sap/disconnect` | Clear `state.sap_connected` flag |
| GET | `/sap/sessions` | Enumerate all SAP connections + sessions |
| POST | `/sap/session` | Manually select which `(conn_idx, sess_idx)` to use (validates first) |
| GET | `/sap/shipment-progress` | Live progress dict for the running shipment (polled every 1s during run) |
| POST | `/supabase/login` | Authenticate for transaction logging |
| POST | `/sap/confirm-to` | LT12 TO confirmation (single) |
| POST | `/sap/process-shipment` | Full 6-step shipment (long-running, synchronous) |
| POST | `/shutdown` | Gracefully terminate |

## Browser Security Layers (all four must pass)
1. **CSP `connect-src`** — [`api/main.py`](api/main.py) must include `http://127.0.0.1:*` and `http://localhost:*`
2. **Mixed-content** — Chrome exempts loopback automatically (no code needed)
3. **Private Network Access (PNA)** — `add_private_network_headers` middleware returns `Access-Control-Allow-Private-Network: true` on preflight
4. **CORS** — `CORSMiddleware` + custom OPTIONS handler responding to all origins

Each layer silently fails with no browser error beyond "Failed to fetch" if misconfigured.

## Citrix Detection
`detect_citrix()` reads `SESSIONNAME` (ICA-* / RDP-*), `ICAROOT`, `CITRIX_HDX_ENABLED`, `CLIENTNAME`. Result is included in `/health` and `/status` so the UI can show Citrix-aware messaging (e.g. "you'll need to re-download next session").

## SAP Session Auto-Selection
`_auto_select_valid_session()` walks every connection in `app.Children`, finds the first one with a session that has a usable `wnd[0]`, and returns `(conn_idx, sess_idx)`. Called:
- On `/sap/connect` when current selection is invalid
- On every `_get_sap_session()` call as a fallback

Fixes the common "error 614 collection enumerator" when the SAP Logon Pad itself is `Children(0)` with 0 sessions.

## Web App Detection (One Click Ship tab)
**File:** `src/features/admin/sap-testing/components/one-click-ship-tab.tsx`

2. Zip: `zip -j OmniFrame_Agent.zip dist/OmniFrame_Agent.exe`
3. Temporarily add anon INSERT/UPDATE policies via Supabase MCP

5. Drop the temp policies
6. No redeploy needed — Supabase serves the new binary immediately

## Live Progress Tracking
- Module-level `_shipment_progress` dict protected by `_progress_lock`
- `_set_step()`, `_append_step_result()`, `_finalize_progress()` helpers
- `GET /sap/shipment-progress` returns a snapshot each poll
- UI polls every 1 second while `isProcessing` is true, stops on completion
- Two HTTP requests run concurrently via FastAPI thread pool:
  - Long-running `POST /sap/process-shipment` (the worker)
  - Fast `GET /sap/shipment-progress` (the poller)

## Console Window
Built with `--console` (not `--windowed`) per user request for visibility during testing. Prints:
- Startup banner with Citrix/user/computer info
- Full SAP connection+session diagnostic on `/sap/connect`
- Per-step status during shipment processing
- Request logs via uvicorn

To switch to silent production: change `--console` to `--windowed` in `build_exe.bat`. The stdio redirect at module top (`os.devnull`) already handles the windowed case without crashing uvicorn's logging.

- **Agent-internal dual-patcher race fix** (2026-05-01, v1.6.8): see [[Debug/Fix-Agent-Dual-Patcher-Race]] + [[Patterns/Agent-Self-Attribution]] (new "Two-step overlay pattern" section). After v1.6.6 + v1.6.7 shipped honest agent attribution, every agent-confirmed `rf_putaway_operations` row STILL had `confirmed_by_label`, `confirmed_by_agent_id`, `confirmed_source` = NULL. Root cause was *agent-internal*, not browser/agent: TWO patchers race on every agent-side TO confirm. (1) `_update_putaway_status` (called from inside `confirm_transfer_order` after a successful LT12) PATCHes the LEGACY 3 fields first — `to_status='TO Confirmed'`, `confirmed_at`, `confirmed_by`. (2) `_apply_trigger_post_patch` (called by the job poller AFTER the handler returns) PATCHed with the FULL body (legacy 3 + attribution 3) plus a `skip_if = {to_status: 'TO Confirmed'}` filter encoded as `&to_status=neq.TO%20Confirmed`. Step 1 had already flipped `to_status='TO Confirmed'`, so the filter ALWAYS matched 0 rows. PostgREST returned 200 OK with empty body, agent logged "applied", attribution columns silently stayed NULL forever. The browser-side path didn't have this bug (no `_update_putaway_status` call there) — only manifested under fleet-routing once production agent-side triggers actually started firing this morning. Fix: `_apply_trigger_post_patch` now applies ONLY the OVERLAY (attribution) fields and drops the `skip_if` filter entirely; there's no double-write risk because we're not touching `to_status`/`confirmed_at`/`confirmed_by` here, the legacy patcher owns those. New `Prefer: return=representation` lets us count rows; `rows_affected == 0` now logs a WARN line with the field values for diagnosability instead of silently logging "applied". v1.6.7 self-healing `_TRIGGER_DROP_AGENT_ATTRIBUTION` cooldown is unchanged — still applies to the label + agent_id columns when migration 251 isn't yet visible to PostgREST. Documentation comment added to `_update_putaway_status` (no logic change there) calling out the OVERLAY pattern so future engineers reading either function understand they cooperate. New log line format: `[triggers] post-success PATCH rf_putaway_operations/<row_id> applied — overlay fields: source=agent_trigger_direct, label='Omni Agent', agent_id=USINDPR-Console-jsingh (rows_affected=1) (job <job_id>)`.


- **Backfill poller for missed Realtime events + bounded TTL dedup cache** (2026-05-01, v1.6.9): see [[Debug/Fix-Missed-Realtime-Events-Backfill]]. Production caught 17 `rf_putaway_operations` rows at `to_status='Completed'` with `confirmed_at=NULL` from the past 4 hours that the agent never picked up despite the same agent correctly auto-confirming 5 OTHER rows in the same window — pure missed Realtime events, not a logic bug. The agent's `rf_putaway_operations` Realtime channel is at-most-once: agent restart, EXE upgrade, WebSocket reconnect blip, Supabase Realtime publication lag, Citrix VDA hibernation, and pg_cron interrupting Realtime can all silently drop events forever, and pre-1.6.9 there was no recovery path. Three additive changes — NO existing handler touched, NO trigger semantics changed, NO frontend logic changed beyond the version-string bump. (1) **Backfill poller daemon thread** `_start_trigger_backfill_poller` wakes every 60s (first poll 10s after boot to give Realtime a head start), runs a bounded PostgREST query per `_HARDCODED_TRIGGERS` entry (`select=*&{trigger.backfill_filter}&organization_id=eq.<org>&created_at=gte.<24h-ago>&order=created_at.asc&limit=50`), defensively re-runs `_hardcoded_trigger_match` on each row, then feeds matches through the same `_enqueue_trigger_job` path the Realtime callback uses. Bounded scope (max 50 rows/poll, 24h lookback, 1 query per trigger per poll) prevents runaway flooding of `sap_agent_jobs`. Quiet by design — `[backfill] poll: N matched, M queued, K skipped (dedup)` only logs when M > 0, otherwise steady-state silent. New `backfill_filter` key on each `_HARDCODED_TRIGGERS` entry holds the per-trigger PostgREST filter so future triggers add their own without code changes. (2) **Bounded TTL dedup cache** `_recently_queued_rows: OrderedDict[str, float]` with 5-min TTL + 1000-entry LRU eviction. Realtime callback and backfill poller both consult `_is_recently_queued(row_id)` before the HTTP call to `sap_agent_jobs`, and both call `_mark_recently_queued(row_id)` on enqueue success or 409 (DB-level idempotency_key still wins as final guard, this just saves the round-trip and quiets the console). Critical correctness invariants: a row that was queued and FAILED can be re-tried later (entries age out, no permanent poison) AND memory is bounded (long-running agent processing 10k rows/day doesn't accumulate 10k OrderedDict entries). ## 2026-05-10 — LX25 Inventory Completion (cross-warehouse cycle-count fan-out)

New capability `lx25-inventory-completion` advertised in `/health.capabilities` (additive — AGENT_VERSION stays at `2.0.0`). Backs the new **Inventory Completion** entry in the SAP Testing → Inventory Management Query Library WAREHOUSE category. See [[Implement-LX25-Inventory-Completion]] for the FE side.

**Endpoint:** `POST /sap/lx25/inventory-completion`

**Request:**
```json
{
  "warehouses": [
    {"warehouse": "WH5", "variant": "TKAWH5"},
    {"warehouse": "WH8", "variant": "TKAWH8"},
    {"warehouse": "JSM", "variant": "TKAJSM"},
    {"warehouse": "JSF", "variant": "TKAJSF"},
    {"warehouse": "PDC", "variant": "TKAPDC"}
  ]
}
```
All fields optional; the agent defaults to a hardcoded `LX25_WAREHOUSES` list (the same 5 entries shown above) so the FE can fire-and-forget.

**Response (success):**
```json
{
  "ok": true,
  "warehouses": [
    {"ok": true, "warehouse": "WH5", "variant": "TKAWH5",
     "warehouse_code": "WH5", "warehouse_name": "Indianapolis Plt 5 Stores",
     "storage_types": [{"storage_type": "110", "total_bins": 6133, "executed": 4302, ...}],
     "total_bins": 6133, "executed": 4302, "completion_pct": 70.15, "elapsed_sec": 8.3},
    ...
  ],
  "totals": {"warehouses_succeeded": 5, "warehouses_failed": 0, "total_bins": 24500, "executed": 17600, "completion_pct": 71.84},
  "meta": {"transaction": "LX25", "started_at": "...", "elapsed_sec": 42.6, "warehouse_count": 5}
}
```

Per-warehouse failures (missing variant, SAP error) are captured INLINE — they DO NOT abort the rest of the fan-out. The aggregate `totals` block sums only the SUCCESSFUL warehouses so a missing variant doesn't drag the cross-warehouse % to zero.

**Flow** (mirrors the recorded `MacWindowsBridge/LX25data.vbs`, sequential per warehouse):
1. `/nLX25` → selection screen
2. Open Get Variant dialog (`tbar[1]/btn[17]`) → type variant in `txtV-LOW` → clear `txtENAME-LOW` → Execute (`wnd[1]/tbar[0]/btn[8]`)
3. F8 (`tbar[1]/btn[8]`) → run report
4. Menu → List → Save → File... (`mbar/menu[0]/menu[1]/menu[2]`) → Unconverted radio → OK
5. Set filename in `ctxtDY_FILENAME` → Save (Enter / btn[11] / sendVKey 11)
6. Read text file from `%TEMP%`, parse with custom LX25 parser (the agent's generic multi-format parser would mis-classify the storage-type-marker rows). Custom parser walks lines, tracks active warehouse + active storage type, matches metric rows by prefix.
7. Back (`tbar[0]/btn[3]`) → next warehouse

**Implementation:** `omni_agent/lx25_inventory_completion.py` — sibling module mounted via the same lazy `app.include_router` pattern as `zmm60_lookup.py` / `material_master_read.py` / `lt22_import.py`. Pydantic model registered in `_JOB_ENDPOINT_MODELS` so fleet-mode queue claims build the model from the JSON body.

**Recording artifact note:** the recorded VBS writes `ctxtP_VARI = "JSINGH"` after the variant load (the recording author's personal variant left over from a previous run). We deliberately do NOT replicate it — the Get Variant dialog already populates P_VARI with the warehouse variant when btn[8] in the dialog is pressed.

**Lease budget caveat:** the 5-warehouse loop typically takes 30-60s end-to-end (~6-12s per warehouse for variant load + execute + export). The agent's `stuck-job-watchdog` default is 120s — enough for the typical case but tight if SAP is slow. Documented as a follow-up in [[Implement-LX25-Inventory-Completion]] § Open follow-ups.

**Mirror status:** `MacWindowsBridge/Omni-Agent/agent.py` and `MacWindowsBridge/Omni-Agent/lx25_inventory_completion.py` updated + byte-equal `cmp` verified for the next EXE rebuild. AST-validated against the `omni_agent/` source.
