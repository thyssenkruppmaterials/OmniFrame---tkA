## Fix: SQCDP metric edit RLS / "Cannot coerce" error

User report: a non-superadmin curator hitting **Save** in the SQCDP metric editor saw `Failed to update metric: Cannot coerce the result to a single JSON object`. Superadmin (`admin@j.ai` / Jai Singh) saved cleanly against the same metric. The `<BoardEditToggle>` and per-card pencil were visible to the affected user (gated by `useCanEditBoards` → `authService.checkPermission`), so the JS gate believed they had `production_boards:edit`.

### Diagnosis path

1. **`Cannot coerce…`** is supabase-js's translation of PostgREST `PGRST116` — `.single()` requires exactly one row and got zero. With `.update().select().single()` that means RLS filtered the mutation to zero affected rows.
2. Read every RLS policy on `sqcdp_metrics`, `sqcdp_metric_history`, and the new `production_board_sqcdp_categories` table ([[Implement-SQCDP-Editable-Categories]] / migration 306) via Supabase MCP `pg_policy`. SELECT policies are org-scoped only — ruled out the FK-eval hypothesis. The single common gate on every mutate path: `public.has_permission('production_boards','edit')`.
3. Read `has_permission(text, text)` itself. Pre-fix body joined `role_permissions` on `rp.role = get_user_role()` — the **legacy `user_role` enum** column. Migration 295 inserted the `production_boards.edit` row for `tka_supervisors` with `rp.role = 'viewer'::user_role` (a placeholder; the real lookup is `rp.role_id`).
4. Live data audit: `tka_supervisors` users split into two cohorts by their `user_profiles.role` enum value (`'viewer'` vs `'tka_associate'`). Cohort A matched the placeholder → RLS approved → save worked. Cohort B did NOT match → RLS denied → UPDATE affected zero rows → PGRST116. Meanwhile `authService.checkPermission` JOINs on `role_id` and approved both cohorts — hence the disagreement between the JS gate (open editor) and the SQL gate (reject save).
5. Reproduction via Supabase MCP `execute_sql` + `SET LOCAL ROLE authenticated` + `set_config('request.jwt.claims', …)` for four personas confirmed the split exactly: Galvez (cohort B) got `affected_rows = 0`; Brumbaugh (cohort A), Jai Singh, and even a pure `viewer` user (latent over-grant) got `1`.

### Fix

- **DB:** `supabase/migrations/308_fix_has_permission_role_id.sql` rewrites `has_permission(text, text)` to JOIN through `user_profiles.role_id → role_permissions.role_id` (canonical, modern path) and preserves the per-user `user_permissions` override branch. Applied via Supabase MCP `apply_migration`. The single-function patch automatically benefits every RLS policy that already calls `has_permission(text, text)` — no policy bodies needed editing. Side-effect (intentional): closes the latent over-grant where pure `viewer` users were accidentally approved by the legacy-enum collision; none of those users could actually mutate via the UI (the JS gate hid the editor for them) so net regression = zero.
- **Client:** `src/features/.../sqcdp/hooks/use-sqcdp-metrics.ts` — `updateMetric` and `createMetric` swap `.single()` → `.maybeSingle()`. When `data === null` and there's no Supabase error, throw a permission-flavoured error (`Update didn't return a row. Your role might be missing the production_boards:edit permission — contact an administrator.`) so a future RLS misalignment can't resurface as the raw PGRST116 string.
- **Test:** new `use-sqcdp-metrics.test.tsx` mocks `maybeSingle()` to resolve `{ data: null, error: null }` and asserts the friendly-error contract (no `"cannot coerce"`, contains `production_boards:edit`, contains `administrator`). Second test pins the Supabase-error pass-through path.

**Note-worthy finding** from the escalator-code audit that materially reshaped Option B's framing: the 60-min escalator (`rust-work-service/src/scheduler/mod.rs::escalate_stale_reservations` → `public.escalate_stale_zone_reservations(60)`, runs every 5 min on `tokio_cron_scheduler "30 */5 * * * *"`) **already reaps `pending+assigned`** rows older than the threshold. B2 closes today's exact incident shape (online-but-idle operator's stale claim) without breaking the "actively-counting" protection; B3 closes the config-drift bug surfaced by the audit; C surfaces in-flight cascades immediately so admins don't have to discover them via operator complaints. Defer D (FE `beforeunload`) unless reap-surprise complaints arise after B ships.

## Fix — Hard DELETE on rr_cyclecount_data Leaves Orphan work_tasks (caused 2026-05-19 import failure)

Morning of 2026-05-19, user's bulk Inventory Counts import failed at every row with `duplicate key value violates unique constraint "work_tasks_org_type_number_uniq"`.

### Root cause
The forward projection trigger from migrations 257 / 265 only handles INSERT and UPDATE on `rr_cyclecount_data` — there is no DELETE branch. Yesterday's hard-delete of 505 pending cycle counts (from the [[Fix-Inventory-Counts-Total-Mismatch|previous session]]) left 505 orphan `work_tasks` rows with task numbers `CC-20260519-0001 … 0505`. The unique index `work_tasks_org_type_number_uniq` then blocked every freshly-generated `count_number`, and the projection error rolled back the source insert. `generate_count_number()` queries `rr_cyclecount_data` only so it kept handing out `CC-20260519-0001` forever; the dialog's "last 5 errors" view made it look like only row 64 was failing.

A side observation: `generate_count_number()` uses `CURRENT_DATE` (UTC), not EDT, which is why rows the user thought of as "yesterday" were stamped `CC-20260519-*` even though they were created the evening of 2026-05-18 EDT.

### Changes
- One-off SQL cleanup: hard-deleted the 505 orphan `work_tasks` rows (after confirming 0 `work_events` and 0 `task_artifacts` referenced them — the `work_events` FK is `RESTRICT`).
- **Migration `316_cycle_count_to_work_tasks_delete_projection.sql`** — new `AFTER DELETE` trigger `trg_sync_cycle_count_delete_to_work_task` on `rr_cyclecount_data`. Mirrors the existing forward trigger: gated by per-org `work_tasks_shadow_write` flag, sets `app.skip_sync` + `app.work_zone_lock_bypass`, targets `(source_table='rr_cyclecount_data', source_id=OLD.id, task_type='cycle_count', organization_id=OLD.organization_id)` so it can never touch a `work_tasks` row that came from a different pipeline.

### Verification
- `apply_migration` succeeded; `pg_trigger` confirms the new trigger is bound for AFTER DELETE.
- Post-cleanup orphan count = 0; `MAX(count_number)` and `MAX(task_number)` both NULL for today's prefix.
- User unblocked: next `generate_count_number()` call resets to `CC-20260519-0001` cleanly.
