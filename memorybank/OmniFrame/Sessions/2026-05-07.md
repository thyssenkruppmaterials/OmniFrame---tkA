## Key decisions

- **Whitelist-only DSL** with 12 operators (`all`, `any`, `not`, `eq`, `neq`, `in`, `gt`, `gte`, `lt`, `lte`, `is_null`, `is_not_null`) plus the `{}` always-match shorthand. Parser rejects unknown operators, multi-key bodies, extra keys, non-numeric values for numeric ops, dangerous field-path syntax (SQL injection, array indexing, double-dots).
- **Source-table allowlist** of 4 tables: `rf_putaway_operations`, `sap_agent_jobs`, `work_tasks`, `shipment_queue`. Adding a table requires a Rust release + a NOTIFY-trigger migration.
- **Target-endpoint allowlist** of 6 endpoints (`/sap/confirm-to`, `/sap/process-shipment`, `/sap/lt12`, `/sap/import-lt22`, `/sap/material-master-bin`, `/sap/material-master-storage-types`). Explicitly EXCLUDES `/sap/connect`, `/supabase/login`, `/agent-token/rotate`, `/shutdown` so the trigger evaluator can never escalate to agent-control endpoints.
- **WsEvent::TriggerFired** carries ONLY metadata (trigger_id, source_row_id, target_endpoint, job_id, organization_id) — no row payload, addressing the row-leak concern from the prior ADR.
- **Idempotency key** matches the legacy agent path (`trig:<id>:<row>:<unix-day>`) so a row whose first enqueue failed retries tomorrow without permanent 409 lockout.

# Session 2026-05-07 (continued) — Rust Work Service Phase 10: agent identity v2 (service-key auth)

## Goal

Ship Phase 10 of the comprehensive [[plans/rust_work_service_full_integration_5b88165d.plan]]: replace the agent's reliance on inherited user JWTs with first-class service-key authentication. Plaintext `omni_sk_*` keys (Argon2id-hashed at rest) traded for 15-min `kind: "agent"` JWTs that the work service verifies locally; admin can revoke a key without touching `auth.users`.

## Approach

Six workstreams landed serially (no parallel collisions because the prior phases consolidated the seams):

1. **Schema** — migration 283 (table + RLS, admin-only via `role IN ('admin','superadmin')`). Applied via Supabase MCP and verified via `information_schema.columns` (12 cols) + `pg_policy` (2 admin policies).
2. **Rust JWT helper** — NEW `agent_jwt.rs` module with HS256 issue/verify against `WORK_SERVICE_AGENT_JWT_SECRET`, plus a `looks_like_agent_jwt()` peek that decides whether to use the local-verify path or fall through to `rust-core-service` for legacy Supabase user JWTs.
3. **Rust routes** — NEW `api/routes/agent_identity.rs` with four endpoints (admin-only `/register` mints + reveals plaintext ONCE, public `/exchange` Argon2id-verifies + issues JWT, admin-only `/revoke` sets `revoked_at`, admin-only `/list`). Argon2id parameters memory_cost=64 MiB / time_cost=3 / parallelism=4 per the ADR; rate-limited at 5 failures / hour / agent_id in Redis.
4. **Middleware extension** — NEW `AuthIdentity { User { … } | Agent { … } }` enum injected alongside the legacy `AuthenticatedUser` shape. Agent JWT path: peek `kind` claim → verify locally → revocation check (Redis 60s cache, DB fallback, fail-closed) → inject identity. Existing handlers pulling `Extension<AuthenticatedUser>` keep working unchanged because we synthesise an `AuthenticatedUser { user_id: agent_id, role: "agent", … }` for `Agent { … }` variants.
5. **Agent first-boot flow** — `omni_agent/agent.py` reads the on-disk service key (default path `~/.omniframe/agent_service_key.txt`, env-overridable), exchanges for a JWT at boot, refreshes via a 60s daemon thread (~60s before expiry). `_work_service_request` (Phase 7) and `_start_work_service_ws_client` (Phase 4) both prefer the agent JWT over the legacy user JWT when present. NO existing handler logic touched, AGENT_VERSION INTENTIONALLY left at 1.9.0 per plan directive.
6. **Admin Setup page** — NEW "Agent Setup" tab in SAP Testing. Lists active keys with last_used_at + creator email, "Register new agent" dialog (free-text agent_id + optional label), one-time plaintext reveal dialog with copy-to-clipboard + "I have saved this key" toggle gate before the close button activates, revoke dialog with optional reason. NEW typed REST client in `src/lib/work-service/agent-identity-client.ts`.

Phase 10.6 polish: dashboard `service_capabilities` field on `/api/v1/sap-testing/dashboard` advertises `["agent-identity-v2"]` so the FE can gate admin UIs and per-agent badges in future polish.

## Key decisions

- **Argon2id parameters: memory_cost=64 MiB, time_cost=3, parallelism=4.** OWASP-baseline tier; ~50ms per verify. Cheap enough for the boot path (one verify per `/exchange` call, 5/hour rate-limit budget); expensive enough to deter offline brute-force on the hash. Encoded into the PHC string so a regression test (`argon2_uses_phase10_parameters`) catches accidental drops to defaults.
- **JWT TTL: 15 minutes.** Short enough that a leaked JWT is bounded; long enough that the agent's 60s pre-expiry refresh thread doesn't burn the work service. Not env-tunable to keep the runtime invariant simple.
- **Revocation cache TTL: 60 s.** Acceptable security/perf tradeoff documented in the ADR. Going lower would 10× the DB hit rate; going higher would let a leaked key keep firing too long. The `/revoke` route ALSO primes the cache with `"1"` immediately so the next agent call is rejected on the very first round-trip in the happy path; the 60 s is the worst case (Redis was unavailable when admin revoked).
- **`AuthIdentity` enum + synthesis pattern** lets us add the new shape without touching any of the 13+ existing route handlers. Phase 11 owns the per-handler audit pass that decides which routes should reject `AuthIdentity::Agent { … }` (admin-only) vs accept either.
- **`kind` claim is the discriminator.** Supabase user JWTs have no `kind` claim → fall through to the legacy `rust-core-service` validation path unchanged. The middleware's `looks_like_agent_jwt()` is a cheap peek that DOES NOT trust the unverified payload for anything beyond routing; `verify()` re-checks the signature against our local secret immediately after.
- **Process-wide test mutex** (`AGENT_JWT_ENV_LOCK`) serialises every test that mutates `WORK_SERVICE_AGENT_JWT_SECRET`. Without it, parallel tests race the env var (issue reads one secret, verify reads another) and surface as flaky `InvalidSignature`. The lock is `pub` so the sibling `agent_identity::tests::jwt_issued_by_register_via_helper_verifies_locally` reaches in for the same serialisation.

## End-to-end flow demonstrated

Documented in [[Implements/Implement-Rust-Work-Service-Phase10]]:
1. Admin registers agent → receives plaintext key (shown ONCE).
2. Agent exchanges key → receives 15-min JWT.
3. Agent calls `/api/v1/sap-agents/jobs/heartbeat` with new JWT → success.
4. Admin revokes → next agent call fails 401 within ~60s.
5. Backward-compat: an agent without a service key continues running on the legacy user-JWT path unchanged.

## Production hotfix — RF Putaway `row_to_jsonb` trigger crash

**Symptom:** RF putaway operators couldn't submit. Browser console: `function row_to_jsonb(rf_putaway_operations) does not exist` (Postgres 42883). Every submit rolled back the entire `complete_putaway_and_clear_cart` RPC transaction.

**Root cause:** Migration 276 (`276_notify_rf_putaway_changed.sql`, applied 03:25 UTC today as part of the rust-work-service Phase 4 ship) had a typo in `notify_rf_putaway_changed()` — it called `row_to_jsonb(NEW)` which **does not exist as a Postgres built-in**. The actual built-ins are `row_to_json(record)→json` and `to_jsonb(anyelement)→jsonb`. PL/pgSQL only resolves call-site overloads at execution time, so the buggy `CREATE FUNCTION` succeeded; the failure first surfaced when the AFTER trigger fired against a real warehouse putaway. **Not a `search_path` bug** — incident report initially framed it that way, but `pg_catalog` is always implicitly searched first regardless of `search_path`, and the other built-ins in the same body resolved fine.

**Fix:** New migration [`supabase/migrations/285_fix_notify_rf_putaway_row_to_jsonb.sql`](../../../supabase/migrations/285_fix_notify_rf_putaway_row_to_jsonb.sql) — minimal `CREATE OR REPLACE FUNCTION` swapping the single offending line `row_to_jsonb(NEW)` → `to_jsonb(NEW)`. All other attributes preserved verbatim (SECURITY DEFINER, search_path, language, returns). Applied via Supabase MCP `apply_migration`. Wire-format compatible — both produce a jsonb object keyed by column name, and `rust-work-service/src/rf_putaway_listener.rs` deserializes `new` as a loose `serde_json::Value` so consumers see no observable difference.

**Verification:** Transactional `DO $$ … INSERT … RAISE EXCEPTION 'rollback_after_verify' … $$;` against the live table fired the trigger cleanly (pre-fix would have raised 42883 before the explicit rollback). Leak check confirmed zero test rows persisted; `pg_notify` is suppressed by transaction rollback so the rust-work-service WS bus never saw the test event. Latent-sibling audit: only `notify_rf_putaway_changed` calls `row_to_jsonb` in the entire database — no other functions need the fix.

**Why this is unrelated to the realtime/presence sprint:** The presence-UI work landed today did NOT touch any backend trigger, RPC, or migration. The bug is owned by Phase 4 of the rust-work-service plan, not the presence sprint. Future post-mortems should attribute the regression accordingly.

**Related:** [[Fix-RF-Putaway-RowToJsonb-Search-Path]] — full debug note with diagnostic SQL, fix migration, verification procedure, and manual-verification steps for the user.

## Pooler regression triage — rust-work-service v0.1.34 hotfix (2026-05-07 22:30 UTC)

After the operator set `WORK_SERVICE_DATABASE_POOLER_URL` to the Supavisor **transaction-pool** URL (port 6543) at ~22:00 UTC, production immediately started 5xx-ing on every endpoint that hits the general-purpose pool — `/api/v1/sap-testing/dashboard`, `/api/v1/agent-identity/list`, `/api/v1/sap-agents/fleet`, plus all `scheduler::*` background tasks.

### Symptom

```
Database(PgDatabaseError { code: "42P05", message: "prepared statement \"sqlx_s_17\" already exists" })
Database(PgDatabaseError { code: "26000", message: "prepared statement \"sqlx_s_42\" does not exist" })
Database(PgDatabaseError { code: "08P01", message: "bind message supplies 3 parameters, but prepared statement \"sqlx_s_36\" requires 0" })
```

All three error classes are textbook **sqlx + Supavisor transaction-pool incompatibility**. sqlx 0.7 hard-codes named prepared statements `sqlx_s_<n>` in `sqlx_postgres::connection::executor::prepare()`. The statement name comes from a per-connection counter (`PgConnection::next_statement_id`, starts at `Oid(1)`), so two app connections that get multiplexed onto the same upstream backend at COMMIT boundaries will both PARSE statements named `sqlx_s_1`, `sqlx_s_2`, … — collision is a matter of when, not if.

### Why Option 1 (`statement-cache-capacity=0`) was insufficient

Verified against `~/.cargo/registry/src/.../sqlx-postgres-0.7.4/src/connection/executor.rs`. The cache only short-circuits *re-preparing* the same SQL on the same connection; it does NOT switch sqlx to use the unnamed prepared-statement path. Every prepare still allocates a fresh `sqlx_s_<n>` name regardless of cache capacity. With cache=0 the deploy still wedged immediately — `/health/detailed` came back 503 with `prepared statement "sqlx_s_18" already exists` on the first DB hit.

sqlx URL parameter naming, while we're here: hyphenated only. `~/.cargo/registry/src/.../sqlx-postgres-0.7.4/src/options/parse.rs` matches `"statement-cache-capacity"`. The underscore form `statement_cache_capacity` is silently ignored by `from_url`.

### Fix (Option 2 — session-mode pooler)

Final env-var value:

```
WORK_SERVICE_DATABASE_POOLER_URL=postgresql://postgres.wncpqxwmbxjgxvrpcake:{PASSWORD}@aws-1-us-east-2.pooler.supabase.com:5432/postgres?sslmode=require
```

Key delta vs. the failed setting: **port 6543 → 5432** (transaction-mode → session-mode). Session-mode preserves backend affinity for the entire client connection lifetime, so `sqlx_s_<n>` never collides across the pool. Drop the `pgbouncer=true` flag — that's only meaningful for Supavisor's txn-pool path.

## ZMM60 extraction strategy fix (same-day follow-up)

Follow-up to the [[Implement-Inventory-Adjustment-Workflow]] ship earlier today. The first cut of `omni_agent/zmm60_lookup.py` reused the LT10 / LT22 `_extract_via_pc_export()` helper for the SAP-side data extraction. That FAILED in the user's manual test with:
