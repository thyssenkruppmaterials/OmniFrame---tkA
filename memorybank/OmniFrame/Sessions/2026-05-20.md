## Putaway confirms — 41 stuck rows; auto-recovery dead-end + silent agent patch skip

User flagged the Putaway Log Search panel's Pending Confirms card showing
**41 stuck (oldest 1371 min) — auto-recovery runs every 5 min** for org
`c9d89a74…`. Investigation surfaced **three** independent compounding bugs;
fixed in a 2-migration sequence. End-to-end drain went from 41 → 1 in
≈30 minutes (the remaining 1 is a real SAP data error, not
auto-recoverable).

### Bug 1 — Backfill claim_count cumulative dead-end

Migration 289's `backfill_pending_putaway_confirms(...)` resets
`attempts = 0` per cycle but NOT `claim_count`. `claim_sap_agent_job(...)`
(migration 291) increments BOTH together. So `attempts` cycles
0→1→0→1… across backfill ticks while `claim_count` accumulates monotonically.
The protective `claim_count < 8` cap fires after ≈35 min of continuous
SAP outage and silently skips the row forever. 40/41 rows hit this from a
single SAPGUI logout window between 15:35–17:06 UTC on 2026-05-19. Manual
`SELECT public.backfill_pending_putaway_confirms()` returned `(0, 0, 1374)`
while 41 rows sat visibly stuck.

### Bug 2 — Backfill leaves `assigned_agent_id` pinned to offline agents

After migration 321's first apply rescued the rows, all 40 still sat in
`status='queued'` because `assigned_agent_id` was still pinned to
`USINDPR-CXA103V-Console-U8206556` (offline since 06:23 UTC). Online agent
`CXA102V` couldn't claim them because `claim_sap_agent_job(...)`'s
predicate is `(assigned_agent_id IS NULL OR assigned_agent_id = p_agent_id)`.
Migration 321 was extended to clear `assigned_agent_id = NULL` on requeue
and the rescue widened to also unpin currently-queued rows.

### Bug 3 — Agent silently skips source-row patch when supabase_token missing

After unpin landed, agent drained 40/40 jobs to `status='completed'`
(SAP returned `"Transfer order 0003684XXX confirmed"` for each). But FE
still showed 21 stuck. Root cause: `omni_agent/agent.py:6827`
(`_apply_trigger_post_patch`) and `:7349` (`_update_putaway_status`) both
short-circuit when `state.supabase_token` is empty. The operator's
Supabase session had expired but the agent's job-control path
(rust-work-service service-key auth) was unaffected, so it confirmed
in SAP but never patched `rf_putaway_operations.confirmed_source`.

### Fix — Migrations 321 + 322

- **`supabase/migrations/321_backfill_resets_claim_count.sql`** —
  CREATE OR REPLACE of `backfill_pending_putaway_confirms` to also reset
  `claim_count = 0` and `assigned_agent_id = NULL` on requeue. One-time
  rescue + unpin pass at the bottom processes 40 stuck rows.
- **`supabase/migrations/322_server_side_trigger_post_success_patch.sql`**
  — `AFTER UPDATE OF status ON sap_agent_jobs WHEN (NEW.status = 'completed')`
  trigger that reads `payload.__omni_trigger_meta.post_success_patch` and
  applies the allowlisted columns (`confirmed_source`, `confirmed_by_label`,
  `confirmed_by_agent_id`) directly in SQL, plus flips
  `to_status='Completed' → 'TO Confirmed'` and sets `confirmed_at = now()`
  for the FE filter. Idempotent (gated on `confirmed_source IS NULL`).
  Failure-suppressing. One-time reconcile pass at the bottom patched the
  21-row backlog.

Lifts the dependency on the agent's expiring user token entirely. No
agent-code change. No Rust change. Function signatures preserved — no FE
or Rust client breakage.

## Path-engine root-cause fix (~21:00–21:30 EST)

Diagnosed and shipped the fix for the silent path-engine regression discovered earlier today.

- **Root cause:** `auto_resolve_cycle_count_location()` (added in migration 204) used `IF v_result IS NOT NULL` instead of `IF FOUND`. In plpgsql, `<record> IS NOT NULL` is FALSE unless every column is non-NULL, and the resolver legitimately returns `mapping_id = NULL` for regex-resolved rows. So 95% of rows (17,202 of 18,112) landed with `resolution_source='unresolved'`, deprioritising them in every `rust-work-service` pull-claim ORDER BY.
- **Fix:** Migration 323 swaps `IF v_result IS NOT NULL` for `IF FOUND` and backfills every poisoned row via direct UPDATE of only the `resolved_*` columns (BEFORE trigger is scoped `OF location, warehouse`, so it doesn't refire; `app.skip_sync` and `app.cycle_count_zone_lock_bypass` GUCs silence the sync + zone-exclusivity triggers).
- **Verification:** 17,099 rows backfilled. Live INSERT test for `RM-69-B-02` / `SP-22-A-04` / `K2-15-03-08` / `GIBBERISH` all stamp the correct zone/aisle/sequence (or correct fallback). The next 15 tasks the pull-claim queue will serve walk a clean serpentine `RK-64-D-01 → E-01 → F-01 → D-02 → E-02 …`.
- 103 pending rows remain `unresolved` — they use non-standard formats like `RM-75A`, `RP-52-B-1A` not covered by the existing three regex rules. Captured as follow-up in [[Debug/Fix-Cycle-Count-Resolve-FOUND-Check-2026-05-20]].
