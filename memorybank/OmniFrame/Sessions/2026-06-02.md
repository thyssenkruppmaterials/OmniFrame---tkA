# Session 2026-06-02

## Done — Print Cover Sheet button on the Kit Build Audit Trail
- Added a **Print Cover Sheet** button to the sticky footer of the **Kit Build Audit Trail** dialog (`KitProductionTrackerDialog`), beside **Delete Kit**, so operators can quickly **reprint** a kit's cover sheet (the `KitBuildSheet`) without leaving the audit trail.
- **Multi-kit-per-PO correctness:** enhanced `KitBuildSheet` with an optional `kitSerialNumber` prop — when set it loads via `getKitBuildPlanDetailsBySerialNumber(serial)` instead of the PO-resolving `getKitBuildPlanDetails(po)`, so a PO covering sibling kits reprints the exact kit the trail is scoped to (per [[Kit-Serial-Scoping]]). The "load-once" ref + open-effect now key on `kitSerialNumber ?? kitPoNumber`. Backward-compatible — the kanban Start-Kit flow still passes PO-only.
- **Files:** `kit-production-tracker.tsx` (footer button group + `isCoverSheetOpen` state + `<KitBuildSheet>` render passing PO + serial; `Printer` icon import), `kit-build-sheet.tsx` (optional `kitSerialNumber` prop + serial-scoped load).
- **Verified:** `tsc -b` clean; ESLint clean (only a pre-existing `flex-shrink-0` warning, not added by this change).
- **Deploy:** FE-only — ships with the next frontend deploy.
- See [[Implementations/Print-Cover-Sheet-From-Audit-Trail]].

## Fixed — Cover sheet showed the wrong kit on a shared PO
- Bug: printing a kit on a multi-kit PO rendered a **sibling kit's** cover sheet (PO `2009708223`: printing `KIT-20260526-002` showed `KIT-20260526-001`), and the printed **QR scanned as "already picked"** (QR encodes the loaded — wrong — kit's serial).
- Root cause: the kanban **Start-Kit** print path still passed **PO-only** to `KitBuildSheet`, which resolves a PO to the first kit. Sibling bug: `markKitAsPrinted(po)` flipped **every** kit on the PO to `printed`.
- Fix: threaded `kitSerialNumber` through the Start-Kit → cover-sheet flow (`pendingStartKitSerialNumber` / `buildSheetKitSerialNumber`, captured from `task.kitSerialNumber`), and added serial-scoped `markKitAsPrintedBySerialNumber`. The audit-trail reprint button already passed the serial, so both print entry points are now serial-scoped (fixes rendered data **and** QR payload). New `markKitAsPrintedBySerialNumber` regression test in `kit-serial-scoping.test.ts`.
- **Verified:** `tsc -b` + ESLint clean; 26 scoping tests pass (the lone failure is the pre-existing date-bomb).
- See [[Debug/Fix-Cover-Sheet-Wrong-Kit-Shared-PO]].
