# Session 2026-05-14

## Production: RF Cycle Count stuck on "waiting for next count"

Operator **David Simmons** (`cbe23c27-…`) signed in to the RF and
tapped Cycle Count this morning (8:24 AM EDT) and got stuck on the
"Pull Next Count" landing / "Loading next count..." flash forever,
even though 2,087 cycle counts are pending org-wide.

Root cause: the **2026-05-07 capacity-gate bug** that was documented
in [[Investigate-Work-Tasks-Capacity-Gate-Returning-Existing-Task]]
but left explicitly unresolved. With `capacity_per_worker = 1` for
cycle_count and David already holding one in-progress
`rr_cyclecount_data` row (`CC-20260421-3323` at `RL-57-C-03`,
claimed 12:03 UTC today), `resolve_effective_capacity` returned 0,
`claim_next_task` short-circuited to `Ok(None)`, and the operator
could neither resume his own in-flight row nor claim a new one.

Shipped fix in [[Fix-RF-Cycle-Count-Stuck-Waiting]]:

- New `phase0_already_assigned_cycle_count` helper in
  `rust-work-service/src/db/queries.rs` runs the already-assigned
  read BEFORE the capacity gate. Same SELECT/zone-collision filter
  as Phase 1 of `claim_next_cycle_count`, read-only on the pool.
- `claim_next_task` calls Phase 0 first for `task_type='cycle_count'`;
  if it returns Some, return immediately. Otherwise fall through to
  the existing capacity-gate → claim_next_cycle_count flow unchanged.
- New regression test
  `tests/dispatcher_phase1.rs::phase0_bypasses_capacity_gate_for_already_assigned_row`
  seeds both rr_cyclecount_data and work_tasks rows + asserts the
  call resolves to the seeded row.

any other operator hits the same disconnect-mid-count sequence.

### Why the morning deploy failed (root cause)

`railway up` built the image, the OLD container kept serving, the new container tried to come up and crash-looped 10× on:

```
thread 'main' panicked at src/main.rs:224:6:
Failed to create general-purpose PostgreSQL pool:
  Database(... "(EMAXCONNSESSION) max clients reached in session mode
                - max clients are limited to pool_size: 16" ...)
```

`WORK_SERVICE_DATABASE_POOLER_URL` is configured against Supavisor's **session-mode** endpoint (`aws-1-us-east-2.pooler.supabase.com:5432`) with a `pool_size = 16` ceiling. During a zero-downtime rolling deploy the OLD container still holds its 20-connection share, so the NEW container's **eager** `connect_with` was rejected immediately, panicked at `src/main.rs:224:6`, and Railway flipped to FAILED after `restartPolicyMaxRetries=10`. Old container kept running → no Phase 0 in prod → James (and the rest of the disconnect-mid-count class) was still vulnerable.

### 13:24Z — Pooler-mode flip ADR drafted

Drafted [[ADR-Pooler-Mode-Flip-Session-To-Transaction]] capturing the
decision space for flipping `rust-work-service`'s general pool from
Supavisor session-mode (port 5432, `pool_size = 16`) to transaction-mode
(port 6543). Documentation-only deliverable — no code changes, no env
changes, no deploys, no DB writes.

Four options considered (status quo + lazy pools, tx-mode flip with
per-tx GUC injection, hybrid two-pool, raise session-mode `pool_size`).
Recommendation: **Option 1 (status quo)** for now — both deploy-crash
classes are CLOSED via the AM general-pool + PM listener-pool lazy
fixes, the residual ~7 s 5xx blip during rolling deploys is bounded
and self-clearing, and the tx-mode flip is **blocked on sqlx 0.7's
hardcoded named prepared statements** (proven in the 2026-05-07
production incident at [[Fix-Sqlx-Supavisor-Txn-Pool-Prepared-Statement-Collision]]).

Note-worthy finding (top of ADR): the day's debug notes implied the
tx-mode flip's main migration cost was the `work_engine.flag_overrides`
refactor. After auditing the code, the **GUC scope is small** —
1 Rust setter (`pool_setup.rs::pool_options_with_hooks::after_connect`),
1 plpgsql reader (`work_engine_feature_flag(uuid, text)`), and ~4 trigger
callers in migrations 257 + 265. FastAPI doesn't use the GUC pattern
at all (zero `set_config` callsites under `api/`). The **real blocker**
is the sqlx 0.7 → 0.8 upgrade or a pgcat / pgbouncer-rs swap, which
makes Option 2 a multi-quarter project, not a 1-line env-var flip.

Also captures explicit decision triggers (rolling-deploy 5xx > 30s
sustained, p99 acquire wait > 1s, concurrent worker count growth, sqlx
upgrade landing for unrelated reasons, Supabase pool_size raise becomes
free, SLA tightening) and open questions (Supabase tier confirmation,
tier-upgrade cost, sqlx 0.8 audit appetite, listener-pool direct-host
migration). Status: `proposed`.
