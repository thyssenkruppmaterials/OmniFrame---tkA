# Session 2026-05-04 — Defer-by-user history on the Inventory Counts dashboard

## Goal

Surface end-to-end visibility for *who deferred / skipped a cycle count, when, and why* on the **Inventory Counts** dashboard — including historical (cleared) defers — plus a dashboard filter "Deferred by [user]" and a search predicate that maps an operator's name to every count they've deferred.

Motivated by the ergonomic gap exposed during the 2026-05-01 [[Per-Operator-vs-Global-Defer-Scope]] / [[Fix-Critical-Hidden-By-Global-Defer-Filter]] forensics: per-operator defer state is powerful but invisible to supervisors auditing why a count was sitting unassigned.

## Decision — Option A (view) over Option B (cache + trigger)

The brief offered two implementations:

- **A.** Postgres view joining `cycle_count_operator_deferred_counts` with `user_profiles` + `rr_cyclecount_data`, lazy-loaded by the frontend.
- **B.** Cache `last_deferred_by` / `last_deferred_at` / `last_defer_reason` columns on `rr_cyclecount_data` maintained by a trigger.

Chose A. Three reasons:

1. The defer table is already source-of-truth; B duplicates state.
2. B keeps only "last" — we want full audit history ("4 operators skipped this in 2 days").
3. RLS on the underlying table already enforces org scope; `security_invoker = true` on the view passes it through transitively, no new policy.

B's perf advantage doesn't matter because we lazy-load — the bulk dashboard fetch is unchanged.

### Lazy-load discipline

The brief's perf constraint dictated the `enabled` gating. The org-wide `useDeferHistoryForOrg` and `useDistinctDeferUsers` hooks are enabled only when:

- the "Deferred by" filter has selections, OR
- the dropdown is open, OR
- the search query is at least 2 characters.

The per-count `useDeferHistoryForCount` is enabled only when the popover is open or `EditCountModal` is mounted. Default dashboard load with no filter and no search costs **0 extra roundtrips**.

## Evening — LT22 import parser hardening (v1.8.2)

User reported their LT22 PDC import via the Omni-Agent failing with a `409 duplicate key value violates unique constraint sap_outbound_to_imports_unique_per_batch` after the agent reported parsing `561 rows, 2 columns` for a file that should have been ~573 rows × 22 columns. Diagnosed two cascading bugs:

1. The v1.7.7 `_parse_attempt_b_tab_delimited` smart-header pass uses a `non_empty < 3` floor to skip SAP banner lines. The user's PDC banner (`Warehouse No.\t\t\tPDC\tIndianapolis PDC`) ships EXACTLY 3 non-empty cells — it slipped past the floor. While the real header on this specific file (19 non-empty) still wins by raw `non_empty`, a future SAP variant with a 4-5 non-empty banner on a sparse layout would trip the parser.
2. Even with a perfect parser, the bulk-INSERT path had no defense against duplicate / null `to_number` values. SAP can legitimately ship duplicate TO numbers in a single LT22 pull (split deliveries) and a single 409 from PostgREST aborts the entire chunk.

### Three layered defenses

**A. Multi-factor header scorer.** New `_score_header_candidate(non_empty, total_cells, subordinate_data_rows)` helper in `omni_agent/agent.py` blends three factors: base `non_empty * 10`, `+ min(subordinate_data_rows, 20) * 5` bonus for lines whose siblings are subordinate (≤ in BOTH `non_empty` AND `total_cells`), and a `-50` penalty when `fill_ratio < 0.3 AND non_empty < 5` (the SAP banner shape). Penalty is intentionally larger than any banner could earn from raw `non_empty` alone (3 × 10 = 30), so banners are reliably outranked even when the bonus is zero for both candidates. Threshold drops from `non_empty < 3` to `non_empty < 2`. Applied to both Format B (tab-delimited) and Format C (fixed-width); Formats A, D, E untouched.

First attempt used a "following lines with similar shape (±2)" bonus per the brief, but that bonus also fires for data rows (they see hundreds of data-row siblings), so a 9-non-empty data row would score 190 (90 + 100) and beat the LT10 17-non-empty real header (170 + 0). The CI-caught regression on `test_lt10export_smart_header.py` drove the refinement to **subordinate** lines (≤ in both metrics) instead of similar-shaped — a data row only wins the bonus if every later candidate is strictly less populated, which by definition is what makes a line a header.

**B. Per-batch dedup defense in `omni_agent/lt22_import.py`.** New `_dedupe_lt22_rows(...)` runs after `normalize_lt22_row` and before the bulk INSERT: drops rows with empty / null `to_number` (single warn-summary log line), deduplicates by `to_number` keeping the first occurrence (split deliveries can legitimately produce duplicate TO numbers in the SAP export). The bulk POST switches from `Prefer: return=minimal` to `Prefer: return=minimal,resolution=ignore-duplicates` so a partial-success run that re-executes against rows already inserted no longer 409-aborts.

**C. Parse-validation gate.** New `_validate_lt22_parse(...)` runs BEFORE the bulk INSERT. If the parsed columns array has no "TO Number"-shaped header OR every row's TO Number is empty, `_save_lt22_parse_failure_snapshot(...)` writes a JSON snapshot to `%TEMP%/omniframe_lt22_parse_failure_<UTC_ts>.json` (best-effort, never blocks the original error) and a specific error is raised: `LT22 parsed but TO Number column not found / values empty — likely parser misidentified header. Diagnostic file saved to <path>`. Triagers can grab the diagnostic and we add a Format F parser if a future SAP variant slips past the multi-factor scorer.

### Verification

New regression test `omni_agent/tests/test_lt22_smart_header.py` mirrors the v1.7.7 `test_lt10export_smart_header.py` self-contained namespace pattern. Three assertions:

- `test_lt22_delivery_data_format` — against the user's actual file: ≥19 cols, ≥500 rows, a "TO Number" column present, non-empty first-row TO number, `header_y ≥ 3`, positive `header_score`.
- `test_lt22_format_b_outranks_banner_with_score` — direct probe of `_score_header_candidate`. Pins the formula via three asserts including a regression-guard for the LT10 data-row-vs-header case.
