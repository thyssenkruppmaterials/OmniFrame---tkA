## Kitting: similar parts cross-linked across kits sharing a Kit PO (afternoon, review-only)

User reported on the **Kit Build Plans** screen that two `In Progress` kits
for the same Kit PO `2010102615` (`KIT-20260512-001` C47E/4 Gear Box 1, TO
`7287809`; `KIT-20260512-002` C47E/4 Gear Box 2, TOs `7287810` + `7287871`)
have their similar parts tied together even though SAP issued separate TOs
for each kit.

### Root cause

The data model uses `kit_serial_number` as kit identity, but the service /
kanban / picking / flag code paths key on `kit_po_number` alone. The C47E/4
dual-gearbox flow is the first production case where one PO maps to multiple
kit serials, exposing the cross-link.

Confirmed via Supabase MCP that `kit_kanban_tasks` for `KIT-20260512-002`
shows `to_lines_picked = 35 / total_to_lines = 49`, which is exactly the
**sum** of both kits' rows (27 + 8 picked, 31 + 18 lines) —
`syncKitProgressFromData(kitPoNumber)` aggregated across both serials and
wrote the combined progress to whichever single task
`getTaskByKitPoNumber` returned first. KIT-001's kanban card is stranded at
`0 / 31`. The `kanban_task_id` on KIT-001's `RR_Kitting_DATA` rows was
also clobbered when KIT-002 was created — `createKitBuildPlan` line ~429
updates `.eq('kit_po_number', input.kitPoNumber)` rather than filtering by
the newly generated serial.

Full callsite inventory + symptom-to-line mapping captured in
[[Fix-Kit-Build-Cross-Linked-Parts]].

### Surfaces affected

- `verifyKitForPicking`, `updateKitStatusToInProgress`,
  `checkAndUpdateKitPickingStatus`, legacy `addPurpleHatFlag` in
  `rf-kitting-picking.service.ts` — all `.eq('kit_po_number', …)`.
- `syncKitProgressFromData`, `getTaskByKitPoNumber` in
  `kit-kanban.service.ts` — same.
- `createKitBuildPlan` line ~429, `appendTOsToKit`, `recheckBomCoverage`,
  `addFlag` / `clearFlagByType`, the three auto-Black-Hat callsites in
  `rr-kitting-data.service.ts` — same.
- `kit_build_flags` is keyed by `kit_po_number`; the active-flag rule does
  not include `kit_serial_number`, so one kit's Black Hat blocks the other.
- `rf-kitting-picking-form.tsx` asks only for a Kit PO at the Scan step —
  no kit-serial disambiguator when one PO covers multiple kits.

The rows in `RR_Kitting_DATA` themselves are clean — each kit serial has
its own rows with its own TO. The line-level mutations
(`markLinePicked`, `reportMissingPart`) update by row `id`, so those are
line-safe. The corruption is in the kit-level fan-out around them.

### Proposal

Review-only — no application code touched. Concrete fix outlined in
[[Fix-Kit-Build-Cross-Linked-Parts]]: rescope every kit-level service
operation to `kit_serial_number`, add a kit-serial picker on the RF Scan
Kit step when verify returns multiple kits, optionally enforce in a
`303_kit_build_flags_serial_scope` forward migration. Sibling primitives
(`addFlagBySerialNumber`, `updateTaskPriorityBySerialNumber`,
`getKitBuildPlanDetailsBySerialNumber`) already exist — the call graph
just doesn't reach them from the bug-affected paths.

## Shift Productivity follow-up #2 — PostgREST 1000-row truncation (late evening)

User reported David Simmons's Gantt was STILL empty after migration 304
shipped, even though his row total correctly updated to 48 tasks.
Investigated end-to-end (row gate, enhancement loop, buildDailyTimeline,
buildActivityBlocks, shift clamping, break splitting) and built a Node
simulation that produced 17 activity blocks for David's events.
Everything pointed at the data NOT arriving at the FE despite the SQL
being correct.

**Root cause:** PostgREST's default 1,000-row page limit silently
truncates the `get_team_activity_events` RPC response. The events RPC
returns 1,651 rows org-wide for today (`ORDER BY user_id,
event_timestamp`); David's UUID `cbe23c27-…` sorts at row 1,343 of
1,651, so his entire 48-event tail falls beyond the cutoff and never
reaches the client. The counts RPC returns 1 row per user (74 ≪ 1,000)
so `total_tasks = 48` was correct — it just had no companion events.
Detric Hardin's UUID `08921ea2-…` sorts inside the first 1,000 rows,
which is why his Gantt rendered normally.

**Fix** (5 LoC in `src/lib/supabase/team-performance.service.ts`):
appended `.range(0, 49999)` to the `.rpc('get_team_activity_events',
…)` chain. Patches both the main `getTeamProductivity` path (associate
row Gantt) and `getActivityEventsForDate` (Production Boards hourly
grid) via the shared private method. No SQL change.

## Shift Productivity follow-up #3 — PostgREST pagination semantics (late evening)

User confirmed `railway up` had shipped the previous `.range(0, 49999)`
fix (verified by grepping the deployed chunk
`feature-shift-productivity-BwNnx6Jw.js` on
`onebox-ai-logistics-production.up.railway.app`), but David's Gantt was
still empty. Live `curl` probes against a temporary `_tmp_range_probe()`
function surfaced TWO PostgREST behaviours I'd missed:

1. **`db-max-rows = 1000` is a hard server-side cap.** `Range: 0-49999`
   returned the SAME 1000-row body (`Content-Range: 0-999/1651`) as no
   Range header at all.
2. **`POST /rpc/…` ignores Range OFFSET.** Range `1000-1999` on POST
   returned rows 0–999 (offset silently ignored). Only `GET /rpc/…`
   honours Range as offset+limit (probe confirmed Range `1000-1999`
   on GET returned rows 1000–1650 of 1651).

Real fix: switch the events RPC to `{ get: true, count: 'exact' }` and
page client-side. First page learns the total via `Content-Range`;
remaining pages fan out in parallel. 50,000-row safety cap with
logger.warn.

Validation: tsc clean; eslint clean on touched file;
`pnpm vitest run src/lib/supabase` 36/37 (the 1 fail is the pre-existing
`rf-cycle-count.service.test.ts` jsdom issue). `pnpm build` succeeds;
new chunk `feature-shift-productivity-kbvwkVvn.js` contains
`B.rpc("get_team_activity_events",{…},{get:!0,count:"exact"}).range(0,999)`
followed by a `Promise.all` over additional `.range(…)` calls.

Not committed. User needs to run `railway up` once more (the Dockerfile
rebuilds the frontend inside the builder stage) and hard-refresh once.

Captured the full POST-vs-GET probe table + bundle grep evidence in the
"2026-05-12 follow-up #3 — Deploy / cache resolution" section of
[[Investigate-Shift-Productivity-Cycle-Counts-Hidden]] and a matching
"Pagination via GET + count:'exact'" sub-section in
[[Fix-Shift-Productivity-Cycle-Count-Filter]].

### Cross-cutting reminder

`supabase.rpc(…)` defaults to POST, which does NOT honour Range
OFFSETs. Any RPC that may return >1000 rows must either pass
`{ get: true }` and paginate via `.range()`, or expose explicit
`p_limit` / `p_offset` body parameters. Worth folding into
`memorybank/.../Patterns/` next session.
