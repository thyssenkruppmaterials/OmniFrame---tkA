## Railway deploy failure — smartsheet-python-sdk 4.0.0 breaks API startup

`onebox-ai-logistics` deploys were failing. Looked like a build error but the
image built fine; the failure was at **healthcheck** — the API aborted startup
with `cannot import name 'Smartsheet' from 'smartsheet' (unknown location)` →
`Critical router import failure`. Production stayed up on the prior deploy
throughout.

Root cause: `api/requirements.txt` pinned `smartsheet-python-sdk>=3.0.0`; a
cache-busted clean install resolved to **4.0.0**, which restructured the package
so `from smartsheet import Smartsheet` no longer resolves (namespace package →
"unknown location"). Known-good line is 3.7.2. The same-named local package
`api/services/smartsheet/` was a red herring (ruled out via local repro + a
build-stage check that runs before `api/` is copied).

Fixes:
- `api/requirements.txt`: `smartsheet-python-sdk>=3.0.0` → `>=3.0.0,<4.0.0`.
- `Dockerfile`: removed the silent `|| (grep -v pyrfc …)` install fallback that
  masked failures; added a post-install `from smartsheet import Smartsheet`
  verification so a broken SDK fails the **build**, not the deploy.

Verified: hardened build first **failed loudly** at the verification step
(confirming the cause), then with the `<4.0.0` pin the redeploy went green —
`[2/2] Healthcheck succeeded!`, service `● Online · 2/2 running`, and public
`/health` → `HTTP 200 {"status":"healthy"}`.

Deploy mechanism note: this service deploys via `railway up` (working-tree
upload), not GitHub auto-deploy (`source.repo` null).
See [[Fix-Smartsheet-SDK-4.0.0-Breaks-Startup-Deploy-2026-06-08]].

## Fix: completed kit stuck in Open Work tab

`KIT-20260519-001` showed "Completed" in the grid but sat in Open Work. Cause:
the Status column uses the derived `kit_stage_status` (on dock = done) while
the tab split used raw `kit_build_status` — and this kit is on dock but stored
`'printed'` (likely a cover-sheet reprint regressed the status). Fixed
`isCompletedRow` to use `(kit_stage_status ?? kit_build_status) ===
'completed'`, and made `getStatistics` on-dock-aware (Completed card counts
on-dock OR 'completed'; Pending/In Progress exclude completed) so the card
matches the tab. 1 kit affected org-wide. Offered follow-ups: backfill that
kit + guard `markKitAsPrinted*` against status regression. `tsc` + ESLint
clean; FE+service only. See
[[Debug/Fix-Completed-Kit-Stuck-In-Open-Work-OnDock-Stale-Status]].

## Location tab 3D scene crash — CSP blocks drei HDRI + troika font/worker

The Location tab's isometric `WarehouseScene3D` blanked with `Could not load
lebombo_1k.hdr: Failed to fetch` → error boundary → `THREE.WebGLRenderer:
Context Lost`. Root cause was a **CSP block, not a network failure** — the prod
CSP in `api/main.py` didn't whitelist the three external/blob requests the 3D
scene makes: drei `<Environment preset>` HDRI from `raw.githack.com` (the
crash), troika `<Text>` font resolution from `cdn.jsdelivr.net`, and troika's
SDF `blob:` worker (the latter two were latent — masked by the HDRI crash).

Fix: whitelist `raw.githack.com` + `cdn.jsdelivr.net` in `connect-src` and add
`worker-src 'self' blob:` in **both** CSP branches (IBL HDRI is the documented
ADR design intent, so whitelist rather than rip out). Plus a frontend resilience
guard — wrapped `<Environment>` in its own `<Suspense>` + an `IBLBoundary` error
boundary so a future blocked/failed HDRI degrades to direct lighting instead of
crashing the `<Canvas>` and dropping the WebGL context. Reviewed-but-unchanged:
the `rust-work-service` notifications 404 (intentional soft-fail, global, not
Location-specific) and the `THREE.Clock` deprecation warn (three/drei internal).
Needs a main-app redeploy for the CSP header to ship. ESLint + TS clean.
See [[Debug/Fix-Location-3D-Scene-CSP-HDRI-Font-Crash]].
