## Summary

- SQCDP TV-mode layout pass. The board was leaving roughly half a card's worth of dead space above each primary value (e.g. "848 Days" floated mid-card) and the grid itself didn't fill `<TvFrame>`'s viewport, so blank space sat below the secondary row.

## Fix: Post composer dialog centering

The newly-built `<PostComposerDialog>` (see [[Implementations/Implement-Production-Boards-Post-Composer]]) opened anchored to the top-left of the viewport instead of being centered.

**Root cause** — single-class oversight in the composer's `DialogContent` override. The shadcn base `<DialogContent>` carries `w-full max-w-[calc(100%-2rem)] sm:max-w-lg`. The composer passed `className='gap-0 overflow-hidden bg-transparent p-0 shadow-none sm:max-w-none border-0'` — which overrode the `max-w-*` cap but left `w-full` intact. Result: `DialogContent` stretched to fill the viewport, and the 1240×780 `<ComposerResizableShell>` (which has its own `style={{ width, height }}`) ended up anchored to the top-left of that viewport-sized grid container. Radix's `translate-x-[-50%] translate-y-[-50%]` was still centering — but it was centering the huge wrapper, not the visible shell.

**Fix** — added `w-auto max-w-none` to the composer's `DialogContent` className so the container shrinks to the shell's intrinsic size, and Radix's centering then centers what the user actually sees. Inline comment documents the trap so a future contributor doesn't strip it.

```diff
-        className='gap-0 overflow-hidden bg-transparent p-0 shadow-none sm:max-w-none border-0'
+        className='w-auto max-w-none gap-0 overflow-hidden bg-transparent p-0 shadow-none sm:max-w-none border-0'
```

**Non-issues we verified along the way** (worth recording so the next debugger doesn't re-investigate):

- `ComposerResizableShell` already only persists `{ width, height }` to localStorage — never `x`/`y`. No stale-position migration needed.
- The shell has no move handle (only a bottom-right resize handle), so there is no drag-to-move state to reset on each open.
- The shell already clamps width/height to `window.innerWidth/innerHeight - 32` on resize. With the dialog now shell-sized + Radix-centered, viewport resize automatically keeps it on-screen.

**Regression guard** — added a one-shot test in `post-composer-dialog.test.tsx` that asserts the rendered `[data-slot="dialog-content"]` element keeps `w-auto`, drops `w-full`, and retains `top-[50%] / left-[50%] / translate-x-[-50%] / translate-y-[-50%]`. If anyone re-introduces `w-full` (or a future shadcn upgrade flips the base classes), this test fails fast instead of silently regressing the visible layout.

## Fix: Composer publish NOT NULL on published_at

**Symptom.** Clicking **Publish** (or **Save draft**) on a new post in `<PostComposerDialog>` without first picking a schedule time produced a Postgres toast:

> `null value in column "published_at" of relation "production_board_posts" violates not-null constraint`

Matching error path for jobs would surface on `posted_at` of `production_board_job_postings`.

**Root cause.** Both `production_board_posts.published_at` and `production_board_job_postings.posted_at` are `NOT NULL` (with a DB-side `DEFAULT now()` per migration 295, verified via `information_schema.columns`). The composer's payload builders did `publishedAt: values.publishAt ?? undefined` / `postedAt: values.publishAt ?? undefined`, expecting the omitted key to let the DB default fire. When `values.publishAt` was `null` (the default, no schedule picked), the chain `null ?? undefined → JSON.stringify` *should* have dropped the key, but the row ended up reaching Postgres with `null` — most likely a supabase-js path that keeps `undefined` properties and serialises them to JSON `null`, or an intermediate `?? null` step somewhere on the wire. The cleanest fix is to never leave the boundary with anything but a non-null timestamp.

**Fix.** In `src/features/shift-productivity/production-boards/components/post-composer-dialog.tsx`, default `publishedAt` (and the job-flavoured `postedAt`) to `new Date().toISOString()` inside the shared payload builders `persistPost` and `persistJob` — the single funnel point for Publish / Save draft / Scheduled / Edit flows. No DB-side change, no migration churn.

```ts
// persistPost
const publishedAt = values.publishAt ?? new Date().toISOString()
// persistJob
const postedAt = values.publishAt ?? new Date().toISOString()
```

Lifecycle semantics now explicit at the boundary:

- **Publish-now** → `publishAt = null` → defaults to `now`, `isPublished = true` → live immediately.
- **Save draft** → `publishAt = null` → defaults to `now`, `isPublished = false` → hidden by `v_active_board_posts` (which AND's `is_published = TRUE` and `published_at <= now()`).
- **Scheduled publish** → `publishAt` = user-picked future ISO → passes through, `isPublished = true` → hidden by `v_active_board_posts` (`published_at <= now()` is false) until the time arrives.

**Why at the persist helpers, not the hook layer.** `persistPost` / `persistJob` are the only callers of the four mutation hooks (`createPost`, `updatePost`, `createJob`, `updateJob`), and they're already the per-kind boundary. Putting the default here keeps the rule in one place, is trivially testable through the existing dialog test (which mocks the mutation), and avoids changing the hook's external contract.

1. `src/features/shift-productivity/production-boards/boards/jobs/hooks/use-job-postings.ts` `createJob` mutationFn omits `posted_at`, `is_published`, `priority`, `attachments`, `kind_data` from its INSERT row — the composer's payload reaches the hook but those keys are silently dropped before they hit Supabase. Scheduled jobs, draft jobs, priority, attachments, and kind_data therefore don't persist on create today. `updateJob` has the same gap. Worth a separate small PR to widen the row builder (does not affect the NOT-NULL fix; `posted_at` has a DB default of `now()`).
2. Even with the DB-side `DEFAULT now()` on `published_at`, defaulting client-side is the cleaner long-term posture — it lets us drop the DB default in a future migration if we ever want to enforce "caller must specify" semantics, and it makes test assertions concrete.

## SQCDP hero auto-fit typography

Same-day follow-on to the morning's [[Implement-SQCDP-TV-Viewport-Fit]] work. After the v1 viewport-fit gave each card the full row-stretched height, single-line hero values like "475" still looked small relative to that newly-tall card — leaving cosmetic dead space between the colored header and the chart strip.

User on the actual TV display:

> The CDP board is on display for the TV. The top cards have tons of empty space. Is there a way that the text inside of those stack cards dynamically increases to help fill up the space and make it look much nicer?

### Mechanism — CSS container queries (`cqh`)

Picked container queries over a hand-rolled `useFitText` hook + `ResizeObserver`. Pure CSS, zero JS, no flash of un-fitted text on first paint. Recharts' `<ResponsiveContainer>` lives in a sibling element OUTSIDE the new container boundary so it's unaffected.

Three static literal class strings live at the top of `sqcdp-card.tsx`:

```
TV_AUTOFIT_PRIMARY     = 'text-[clamp(6rem,28cqh,16rem)] leading-[0.95] overflow-hidden'
TV_AUTOFIT_SUB         = 'text-[clamp(3rem,30cqh,9rem)] leading-[1] overflow-hidden'
TV_AUTOFIT_TREND_ICON  = 'h-[20cqh] w-[20cqh] min-h-[2.5rem] min-w-[2.5rem] max-h-[5rem] max-w-[5rem]'
```

Static literals (not template-composed) so Tailwind v4's JIT can see the arbitrary class — the same gotcha [[Per-Field-Style-Overrides]] documents.

The single-mode wrapper (and each `<SubMetricBlock>` in stacked mode) sets `style={{ containerType: 'size' }}` in TV mode, which establishes the container query boundary. The value div picks up the auto-fit class via `cn()` — appended LAST so twMerge's dedup picks the clamp expression over the static `text-{N}xl` baseline. Curator overrides win: when `styleConfig.primary.size` is pinned, the gate `useAutoFitPrimary` is false, the auto-fit class is omitted, and twMerge picks the curator's `text-{N}xl`.

Per-block `flex-1 min-h-0` added to each `<SubMetricBlock>` in TV mode so 1 / 2 / 3 sub-metrics distribute the wrapper's height evenly — combined with each block's own `container-type: size`, the value text scales proportionally across the stack.

Min/max of the clamp ladder were tuned so a 1080p TV stays at-or-above the previous `text-7xl` default (96 px floor on primary, vs 72 px before — strict improvement) and a 4K TV gets meaningfully bigger numbers (≤ 256 px max).

### Editor UX

`<FieldStyleRow>` now renders a tiny `text-[10px]` hint under the Primary value Size select:

- **Size unset** → `"TV mode: auto-fit scales the value to the card height (default)."`
- **Size pinned** → `"TV mode: pinned size wins — auto-fit disabled. Use the Reset button to re-enable."`

The existing row-level Reset button covers re-enabling; we didn't add a per-key reset (no other field has one).

## SQCDP uniform hero typography (v15.1)

Same-day correction to the morning's [[Implement-SQCDP-Hero-Autofit-Typography]] (v15). User reviewed a 1080p TV screenshot and reported that despite v15's identical class chain, the rendered glyph sizes drifted noticeably across the five primary cards (35 QNs ~110 px vs. 848 Days ~80–90 px lines). User's exact ask:

> "What I would like to see is pure consistency across the board, where everything resizes in the correct ratio as the screen gets larger. This way, it is an even design across the board that is dynamic and fluid."

### Why v15 drifted

v15 used `text-[clamp(6rem, 28cqh, 16rem)]` with `style={{ containerType: 'size' }}` on each card's value-block wrapper. CSS container queries (`cqh`) resolve against the OWNING container's height — and each card's wrapper is `flex-1` inside the body, with subtitle / comparison / `<TrendIndicator>` competing for residual space. Subtle per-card differences (text wrap, comparison width, trend arrow presence) drove subtly different wrapper heights → subtly different `cqh` values → subtly different font-sizes. **Container queries are the wrong tool for cross-component uniformity.**

### Mechanism — viewport-relative `vh`

Replace `cqh` with `vh`. `vh` resolves against the viewport (every card sees the same value) so siblings render uniformly. Still fluid: 1 vh on 1080p = 10.8 px, on 4K = 21.6 px — same clamp expression resolves to the same px in every card on a given viewport, but smoothly tracks screen size. No `containerType: 'size'` needed → drop the inline style entirely. Constants renamed `TV_AUTOFIT_*` → `TV_UNIFORM_*` so future readers find a name that reflects the actual mental model.

```ts
const TV_UNIFORM_PRIMARY =
  'text-[clamp(4.5rem,10.5vh,12rem)] leading-[0.95] overflow-hidden'
const TV_UNIFORM_SUB =
  'text-[clamp(2.25rem,5.5vh,5.5rem)] leading-[1] overflow-hidden'
const TV_UNIFORM_TREND_ICON =
  'h-[7vh] w-[7vh] min-h-[2rem] min-w-[2rem] max-h-[4.5rem] max-w-[4.5rem]'
```

### Visual QA matrix (computed from clamp + vh inputs)

| Viewport | 1 vh | Primary | Sub-metric | Trend icon |
|----------|------|---------|------------|------------|
| 1024×768 (small preview) | 7.68 | 80 px | 42 px | 54 px |
| 1920×1080 (canonical TV) | 10.80 | 113 px | 59 px | 72 px (max) |
| 2560×1440 (1440p) | 14.40 | 151 px | 79 px | 72 px (max) |
| 3840×2160 (4K TV) | 21.60 | 192 px (max) | 88 px (max) | 72 px (max) |

On 1080p all five primary cards (475 / 99.7% / 8% OT / 35 QNs / 848 Days) render their hero number at 113 px. The user's reported drift is closed — the size-resolution input is the viewport, not the per-card container.

### Alternatives considered (in the ADR)

- **Container queries with `cqw` (width).** `grid-cols-5` + `auto-rows-fr` → uniform width too, but containerType still needed; vh is strictly simpler.
- **JS-driven `useUniformHeroSize` hook.** Perfect consistency AND perfect fit, but adds JS, ResizeObserver, first-paint flash, sibling coupling. Held in reserve for content extremes vh can't handle.
- **Single fixed vh value** (no clamp). Unreadable on small previews, comically large on 8K.
- **CSS subgrid / grid-row tricks.** Overkill — `auto-rows-fr` already gives every card identical row-stretched heights; the inconsistency was in font-size resolution against that height, not in the height itself.

## RF Kit Picking — Floor Bin Visual Confirmation

### Summary

In the RF Kit Picking flow the `Confirm Location` step (internal `currentStep === 'scan_location'`) was asking the operator to scan a bin barcode for **every** pick. But **floor bins** (`K…` / `S…`) have **no printed barcodes** — only **rack bins** (`R…`) do. Floor pickers were stuck staring at an input that could never be satisfied; the flow was soft-blocked.

The `Confirm Location` step now branches by bin prefix:

- **Floor (K/S)** — renders a new visual-confirm UI: giant `text-5xl sm:text-6xl` bin label inside a bordered `Card`, `Part` / `Qty` mini-grid below, then a new `HoldToConfirmButton` (~800ms press-and-hold) at the bottom. The press-and-hold gesture IS the verification.
- **Rack (R)** — unchanged. Same scan input, same auto-advance on match, same `validateLocation` gate, same `Back` button.

Internal step name `'scan_location'` preserved on both branches so any downstream telemetry that ever reads the step value (none today per [[Fix-Worker-Heartbeats-Stale-Task-Type]]) sees no regression.

### Chosen confirmation pattern — Press-and-hold (~800ms)

The task brief offered three candidates. Decision matrix:

| Candidate | Verdict | Why |
|---|---|---|
| Typed-confirm (last 2 digits) | rejected | Painful on RF terminals' software keyboards; amplified by floor picks dominating kit picking. |
| Brief tap-debounce (~400ms) | rejected | Operators read a sluggish button as broken and tap multiple times; doesn't actually block accidental taps, just delays them. |
| **Press-and-hold (~800ms)** | **chosen** | Single-handed; visually fed-back via fill bar; hard to misfire on glove-knock / accidental scanner trigger; matches modern RF UX (Zebra / Honeywell). Reuses the existing `autoAdvanceDelay = 800` constant — the form has one "deliberate action" tempo across the whole flow. |

It's also the natural escalation of the form's existing no-barcode pattern (`visuallyVerifiedPart` Checkbox + button on the part step) — same intent ("operator attests no-barcode action"), stronger commit gesture because the location step has no second source-of-truth to cross-check against.

## SQCDP measured hero typography (v15.2)

Third same-day pass on the SQCDP TV hero typography. After v15 (`cqh`, [[Implement-SQCDP-Hero-Autofit-Typography]]) and v15.1 (`vh`, [[Implement-SQCDP-Uniform-Hero-Typography]]) both shipped CSS-only recipes that the user reported as broken on the actual TV display, v15.2 takes the JS measurement route. The user's third request, verbatim:

> "It looks like it is currently not displaying as expected. The sizes are still different, and the percent is cut off. Please review comprehensively and fix these issues."

Fault map from the v15.1 screenshot the user attached: Delivery's `99.7%` glyph clipped right-edge; Safety `848 Days` + Quality `35 QNs` wrapped to 2 lines; Cost `8 % OT` looked smaller than wrapped neighbours; People `475` looked oversized; Shipping `500 Orders` + Announcement `73 DAYS FOR PHYSICAL` rendered at the primary clamp and overflowed worse (the v15.1 gate didn't split tiers). Pure-CSS approaches can't deliver "every card the same px AND no card overflows" without measuring what actually fits.

v15.2 introduces:

- `hooks/use-uniform-hero-fit.ts` (new) — the hook + context + registry. Three independent tier registries (`primary` / `sub` / `secondarySingle`), shared `ResizeObserver`, debounced (100 ms) window resize, RAF-collapsed recompute, natural-width cache keyed by `(text, font properties, refPx)`. Compute per tier: `tierSize = min(maxFit across tier)`, clamped by `viewportHeight * viewportCeilingVh[tier] / 100`, floored at `floorPx[tier]`, rounded to 4 px. Overflow fallback: if `min(maxFit) < floorPx`, demote the snuggest entry to `whitespace-normal` + `line-clamp-2` and pick tier size from the survivors. Bounded by entry count.
- `components/sqcdp-hero-fit-provider.tsx` (new) — mounts the registry once at the SQCDP board root for TV mode. Inert in normal density.
- `components/sqcdp-card.tsx` (modified) — calls `useUniformHeroFit` BEFORE early returns (Rules of Hooks). The registered element is a new `min-w-0 flex-1` wrapper around just the value text (not the outer flex row) so the measurement's available-width reading factors in the trend icon sitting as a flex sibling. Trend icon size now derived from the measured hero px (`iconPx = clamp(28, heroPx * 0.5, 80)`) and applied via inline `style` on `<TrendIndicator>`. Curator override path unchanged — when `metric.styleConfig.primary.size` is pinned, `enabled: false` is passed to the hook and the static `text-{N}xl` chain survives.
- `sqcdp-board.tsx` (modified) — wraps the TV branch in `<SqcdpHeroFitProvider enabled>`.

## Fix: Build Kit completion check was PO-keyed → blocked multi-kit POs

Direct follow-up to this morning's [[RF-Build-Kit-By-Serial-Number]] ship. That note explicitly flagged the Build Kit downstream mutations (`startKitBuild`, `kitMaterial`, `completeKitBuild`, `markLineAsKitted`, `unmarkLineAsKitted`) as still PO-keyed and called the multi-kit-per-PO risk an open follow-up. The risk bit the floor a few hours later: operator on RF Build Kit scanned kit serial `KIT-20260515-002` (the 31-line gear-box on PO `2010102616`), kitted all 31 lines (review screen correctly showed **Materials Kitted: 31 / 31** in green), tapped **Complete Kit Build** → toast **"Cannot complete kit: 18 lines still need to be kitted"**. The PO also covers the unstarted 18-line sibling, and `completeKitBuild`'s aggregate check + final UPDATE were scoped to `kit_po_number` alone, so it saw 49 rows and rejected.

### Root cause

`RRKittingDataService.completeKitBuild(kitPoNumber)` ran its pre-check (`SELECT kit_to_line_kitted_date_time FROM RR_Kitting_DATA WHERE kit_po_number = $1`) and final UPDATE (`SET kit_build_status = 'kit_built' WHERE kit_po_number = $1`) at PO scope. With two kits on the PO and one fully kitted but the other unstarted, the unkitted-row count came from BOTH kits. Same class of multi-kit-per-PO cross-link bug as [[Fix-Kit-Build-Cross-Linked-Parts]] but on the Build Kit service paths rather than picking.

### Fix (Option A from the task brief: additive optional `kitSerialNumber`)

- `RRKittingDataService.completeKitBuild(kitPoNumber, kitSerialNumber?)`, `.startKitBuild(…, kitSerialNumber?)`, `.kitMaterial(…, kitSerialNumber?)` — when the caller supplies a serial, every internal query adds `.eq('kit_serial_number', serial)`. PO-only behaviour preserved verbatim when the serial is absent (backward-compat for single-kit POs and pre-serial legacy rows).
- `markLineAsKitted` / `unmarkLineAsKitted` — still line-id-keyed (inherently safe from the cross-link bug). Now read the row's `kit_serial_number` alongside its PO and route the kanban sync through `KitKanbanService.syncKitProgressFromSerial(serial)` instead of the PO-fanout helper.
- `assembleBuildKitPayload` — now exposes `kitSerialNumber: firstRecord.kit_serial_number ?? null` on the returned `kitData`. Single source of truth used by both `verifyKitForBuild` and `verifyKitForBuildBySerialNumber`. A new exported `BuildKitVerifyResult` type captures the shared shape so both entry points and the form stay in lock-step.
- `RRKittingDataRecord` interface — added the missing `kit_serial_number?: string | null` field. (The DB column always existed; only the type was missing.)
- `useBuildKitTool` (`src/hooks/use-build-kit.ts`) — `startBuildMutation` and `completeKitMutation` now accept either a bare string (legacy) or `{ kitPoNumber, kitSerialNumber? }`; `kitMaterialMutation` input gained an optional `kitSerialNumber`.
- `RFBuildKitForm` (`src/components/ui/rf-build-kit-form.tsx`) — `KitData` interface gained `kitSerialNumber: string | null`; `handleKitPoValidation`, `handleQuantityConfirm`, and `handleCompleteKit` all now pass `{ kitPoNumber, kitSerialNumber }` instead of just the PO. Directly fixes the user-visible toast.
- Legacy desktop `BuildKitForm` (`src/components/kitting/build-kit-form.tsx`) — same parity update on the desktop surface.

### Inspection-bypass exclusion (the decision call)

The skip-inspection branch in `completeKitBuild` stamps the inspection columns to preserve the production-tracker audit trail (see [[Optional-Kit-Inspection-Toggle]]). Those rows are not productive inspection work — nobody actually inspected. The chosen pattern is an **org-level guard** rather than a per-row comparison (there is no `kit_built_by_user` column, and `kit_to_line_kitted_by_user` is N:1 per kit):

```sql
AND (
  SELECT NOT EXISTS (
    SELECT 1 FROM kitting_workflow_settings kws
    WHERE kws.organization_id = p_organization_id
      AND kws.kit_inspection_required = FALSE
  )
)
```

Default-true via `NOT EXISTS` so orgs without a settings row keep legacy three-stage behaviour. Verified live on j.AI OneBox (currently `kit_inspection_required = FALSE`) — the smoke check returned `kit_inspection: 0 events` even though `RR_Kitting_DATA` carries two distinct kit serials with `kit_inspection_completion_date_time` set.

## Fix: v15.2 hero-fit overflow on secondary single-mode cards

User pinged ~30 minutes after the [[Implementations/Implement-SQCDP-Measured-Hero-Typography]] v15.2 ship: secondary single-mode cards were STILL clipping on the TV display. Screenshot evidence (secondary tier row, TV mode):

- **Shipping** (`500 Orders Shipped`) — rendering as `500 Orders Shipp`, the `ed` clipped off the right edge of the card. Single-line, no wrap fallback fired.
- **Announcement** (`73 DAYS FOR PHYSICAL`) — wrapping to 2 lines `73 DAYS FOR / PHYSICAL`. Wrap DID fire here, with a stray `R` visible below `PHYSICAL` (most likely the start of the `Rolling 4 Weeks` period chip on a tightly-packed card, but could also be a 3rd line that `line-clamp-2` is barely catching on the narrow viewport).

Two failure modes, two cards, one root cause.

### Root cause — sole-survivor demote bug in the v15.2 compute loop

`useUniformHeroFitRegistry.compute()` in the hook runs an iterative demote loop on each tier. When an entry's one-line max-fit falls below the tier floor it gets added to the overflow set (renders with `whitespace-normal` + `line-clamp-2`) and the tier-size is recomputed from the survivors. The v15.2 ship's loop:

```ts
if (minFit < options.floorPx[tier] && working.length > 1) {
  const culprit = working.reduce(...)
  overflow.add(entryKey(tier, culprit.id))
  working = working.filter(...)
  continue
}
chosen = candidate
break
```

The `working.length > 1` guard is correct for the filter step (you can't demote the last entry out of `working` — the loop would never pick a size). But it's WRONG for the `overflow.add` call. When two secondary-single entries both have `maxFit < floor`:

1. Iter 1: snuggest entry (Announcement, `maxFit ≈ 29 px`) demoted. `working = [Shipping]`, `overflow = {Announcement}`.
2. Iter 2: `minFit = 32 px < floor 48 px`, but `working.length === 1` ⇒ guard fails ⇒ Shipping is **not** added to overflow ⇒ falls through to `chosen = 48 px`, break.

Result: tier size = 48 px floor; Shipping renders single-line at 48 px with `whitespace-nowrap` (the non-overflow class chain). By construction `maxFit < floor`, so the floor doesn't fit either ⇒ visible horizontal clipping. Exact match for the screenshot.

The comment in the original code said *"only one entry remains (we accept its overflow)"* — but the code only **accepted** the overflow visually; it never **marked** the entry as `overflow: true`, so the card kept the nowrap chain instead of the wrap chain. Off-by-one in the bookkeeping.

### Measurement math (canonical 1080p TV)

Layout: TvFrame `<main p-10>` → 80 px horizontal padding; secondary row `grid-cols-4 gap-5` → each card `(1920 − 80 − 60) / 4 = 445 px`; card body `p-7` = 56 px total padding ⇒ body inner ≈ 389 px. The hook measures the inner `min-w-0 flex-1` wrapper around the value text, whose `clientWidth` is 389 px by construction. Available = 389 − 16 (inline safety) = 373 px.

| Card | text | naturalRef @ 100 px (Geist `font-black tabular-nums`) | maxFit | vs. floor 48 |
|------|------|--------------------------------------------------------|--------|--------------|
| Shipping | `500 Orders Shipped` (18 chars) | ~1170 px | ~32 px | below |
| Announcement | `73 DAYS FOR PHYSICAL` (20 chars) | ~1300 px | ~29 px | below |

Both below floor at canonical TV ⇒ both *should* be demoted. The bug left Shipping out of the overflow set.

### Padding theory in the task spec was wrong

The task spec ranked "padding accounting wrong" as the most likely cause. I disproved it: the registered element (the inner `min-w-0 flex-1` div) sits **inside** the body's `p-7`. Its `clientWidth` excludes its own padding by definition, and the div has no padding of its own, so `clientWidth` IS the post-`p-7` inner width. No double-counting. The card-body padding is correctly accounted for.

### Fix shape

Two small, additive changes to `hooks/use-uniform-hero-fit.ts`:

1. **Sole-survivor demote.** Move `overflow.add(entryKey(tier, culprit.id))` outside the `working.length > 1` guard. The filter+continue stays gated (can't filter the last entry); the bookkeeping flips unconditionally. After the fix, both Shipping and Announcement land in the overflow set; tier size = floor (48 px); both wrap to 2 lines `500 Orders` / `Shipped` and `73 DAYS FOR` / `PHYSICAL` at 1080p TV — no horizontal clipping.

```diff
-        if (minFit < options.floorPx[tier] && working.length > 1) {
+        if (minFit < options.floorPx[tier]) {
           const culprit = working.reduce((acc, e) =>
             e.maxFit < acc.maxFit ? e : acc
           )
           overflow.add(entryKey(tier, culprit.id))
-          working = working.filter((e) => e.id !== culprit.id)
-          continue
+          if (working.length > 1) {
+            working = working.filter((e) => e.id !== culprit.id)
+            continue
+          }
         }
```

2. **Font-load re-measure.** Geist is loaded from Google Fonts with `display=swap`. The first compute pass runs against the system fallback (`sans-serif`) — narrower per-glyph than Geist — so the initial natural-width measurement is undersized and the chosen tier size comes out larger than what actually fits once Geist swaps in. Added a `document.fonts.ready` listener that clears the natural-width cache and re-schedules a compute. Defensive against the latent race; doesn't change the immediate visible bug because both fallback and Geist measurements put the user's exact values below the floor.

```ts
useEffect(() => {
  if (!enabled) return
  if (typeof document === 'undefined') return
  const fonts = (document as Document & { fonts?: FontFaceSet }).fonts
  if (!fonts || typeof fonts.ready?.then !== 'function') return
  let cancelled = false
  fonts.ready.then(() => {
    if (cancelled) return
    measureCacheRef.current.clear()
    schedule()
  })
  return () => { cancelled = true }
}, [enabled, schedule])
```

### Fixes considered but NOT applied (and why)

- **`2 × maxFit` upper bound for overflow entries.** Spec'd as Fix B. Would let the tier size go above the floor when the wrap-fit math says it can. Tested it: for `73 DAYS FOR PHYSICAL` at 1080p TV, `2 × maxFit ≈ 57 px > floor 48 px`, so it pushes the chosen size up to 57. At 57 px the greedy word-boundary wrap goes to 3 lines (`73 DAYS` / `FOR` / `PHYSICAL`) because `73 DAYS FOR` at 57 px is wider than the card. The floor (48 px) is the correct answer — adding the bound makes it strictly worse. Skipped.
- **Re-measuring wrapped-text height to verify ≤ 2 lines.** Spec'd as Fix D. More robust but expensive (per-entry clone-and-measure per recompute), and unnecessary at canonical TV viewports because the floor wraps cleanly for both values once Fix #1 is in place. Held in reserve for embedded-preview viewports where the floor itself is too wide to fit the longest word.
