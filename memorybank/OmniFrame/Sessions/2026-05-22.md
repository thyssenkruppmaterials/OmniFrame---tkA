### Fix: Admin password reset returned `Authentication failed` (401)

User-management → Reset password kept failing in production with
`Failed to reset password: Authentication failed`. Affected every admin
endpoint guarded by `RequireAdmin` (also saw `/api/smartsheet/...` 401s
in the same window from the same IP). Root cause was the `RUST_CORE_PRIVATE_URL`
env var added yesterday in [[Fix-FastAPI-Rate-Limiter-500-Storm-2026-05-21]]
pointing at `http://rust-core-service.railway.internal:8010`, while
`rust-core-service` binds only to `0.0.0.0:8010` (IPv4) and Railway's
`*.railway.internal` DNS resolves to AAAA records (IPv6 only) on legacy
environments. Result: `httpcore.ConnectError: All connection attempts failed`
on every auth call.

Made it worse: `validate_token_with_profile` in `api/lib/rust_core/__init__.py`
only caught `httpx.HTTPStatusError`, so the raw transport exception escaped
to the generic `except Exception` arm of `get_current_user` and surfaced as
`401 Unauthorized` / `"Authentication failed"` — pointing everyone at JWT
state instead of infra.

**Fixes (this session):**
- `rust-core-service/src/main.rs` — bind `[::]:PORT` instead of `0.0.0.0:PORT`
  so the service is reachable over Railway's IPv6 private DNS.
  (Dual-stack via `bindv6only=0`, so the public proxy hop keeps working.)
- `api/lib/rust_core/__init__.py` — wrap `httpx.ConnectError`,
  `httpx.TimeoutException`, `httpx.TransportError` in
  `RustCoreConnectionError` inside both `validate_token` and
  `validate_token_with_profile`. The caller already maps that to a
  proper `503 Service Unavailable` with detail
  `"Authentication service temporarily unavailable"` — no more silent
  401s when the upstream is unreachable.
- **Railway env var** (`onebox-ai-logistics`): set
  `RUST_CORE_PRIVATE_URL=https://rust-core-service-production.up.railway.app`
  as an immediate unblock (auto-triggered redeploy
  `0b5df4c3-d310-4df3-b3d7-abc09063f027`, succeeded 14:03 UTC).

**Verified:** new deploy logs show
`✅ Rust Core Service connected (secure JWT validation enabled)` on all 7
workers; no more `Rust core connection failed` warnings; no further 401s
from the affected user IP after the redeploy.

Captured in [[Fix-Rust-Core-Private-URL-IPv6-401-2026-05-22]].

### Fix: SQCDP `New metric` save crashed with uncaught `ZodError`

User reported that the SQCDP board's `New metric` button opened the editor
but clicking **Create metric** silently failed — the console showed
`Uncaught (in promise) ZodError: [...{ path: ['title'], message: 'Title is required' }]`
and no inline form error rendered. The same crash blocked every
react-hook-form + Zod form in the app on validation failure.

Root cause: `@hookform/resolvers@3.10.0` (installed) reads `error.errors`
on Zod errors (the v3 shape), but `zod@4.2.1` (installed) removed `errors`
in favour of `error.issues`. The resolver's `isZod3Error` guard
(`Array.isArray(r?.errors)`) returned `false` for every Zod v4 error, so
the raw `ZodError` got re-thrown as an uncaught promise rejection.

**Fixes (this session):**
- `package.json` — bump `@hookform/resolvers` `^3.10.0` → `^5.4.0` (v5
  auto-detects Zod v3 vs v4 schemas and uses `error.issues` for v4).
- 25 form components — added `as never` cast on the resolver expression
  to keep `tsc -b` green. v5's stricter `Resolver<z4.input<T>, Context,
  z4.output<T>>` signature breaks the existing
  `useForm<z.infer<typeof schema>>` typing for any schema with
  `.default(...)` (where input ≠ output). The cast matches the existing
  convention from `sqcdp-editor-dialog.tsx` and keeps the diff surgical.
- `sqcdp-editor-dialog.tsx` — removed the now-unnecessary `as never` cast
  (its `metricSchema` has no `.default(...)`, so input = output and the
  resolver type matches `useForm<MetricFormValues>` cleanly).

**Verified:**
- `pnpm tsc -b` green (was 304 errors after the resolver bump alone, 0
  after the casts).
- `pnpm build` green; bundle delta +1.54 KB (pre-existing budget breach
  on `warehouse-location-map` etc. is unrelated).
- 386/386 targeted vitest tests pass across `shift-productivity`,
  `admin/onboarding`, `user-management`.
- Runtime smoke test in Node REPL confirms the resolver now returns
  `{ errors: { title: { message: 'Title is required' } } }` on invalid
  input instead of throwing.

Captured in [[Fix-SQCDP-Metric-Editor-Uncaught-ZodError-2026-05-22]].
