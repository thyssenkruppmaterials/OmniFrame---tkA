## Omni-Agent ŌĆö Tier 4 Citrix Headless Service (new)

### Motivation
Bridge requires launching a specific `.exe` per session. Agent lets users open **any Chrome browser** in Citrix and use One Click Ship directly. Bridge retained as fallback.

### Architecture
- **`omni_agent/agent.py`** ŌĆö FastAPI + uvicorn + pywin32 on `127.0.0.1:8765`
- **`build_exe.bat`** ŌĆö PyInstaller `--console` (visible window for observability)
- **No install** ŌĆö runs from wherever user extracts the ZIP; delete the `.exe` to uninstall
- **Auto-select valid SAP session** ŌĆö `_auto_select_valid_session()` walks all connections, skips SAP Logon Pad (0 sessions), picks first with a usable `wnd[0]`
- **`_wait_for_control()`** helper ŌĆö polls for specific SAP element before interacting, handles scripting-popup-induced delays
- **Citrix detection** ŌĆö `SESSIONNAME`, `ICAROOT`, `CITRIX_HDX_ENABLED`, `CLIENTNAME` env vars
- **Live progress tracking** ŌĆö `_shipment_progress` dict + `threading.Lock`, polled via `GET /sap/shipment-progress` every 1s during run

### REST Endpoints
`/health`, `/status`, `/sap/connect`, `/sap/disconnect`, `/sap/sessions`, `/sap/session`, `/sap/shipment-progress`, `/supabase/login`, `/sap/confirm-to`, `/sap/process-shipment`, `/shutdown`

## Agent Triggers ŌĆö Realtime Automation Engine (new)

### Why
After a deep-dive conversation about "what becomes possible when SAP is just another REST endpoint," built the next logical step: **database events as automatic SAP triggers**.

### What Shipped
- **New tab** (8th in SAP Testing): `Agent Triggers`
- **Runtime hook** `use-agent-trigger-runtime.ts`:
  - Per-trigger Supabase Realtime channel subscription
  - Global serialized fire queue (SAP is single-threaded per session)
  - 500ms gap between fires
  - Per-trigger rate limit (default 30 fires/min)
  - `${row.field}` payload interpolation
  - Session binding: auto-switches agent session before firing when `type: 'specific'`
  - Defense-in-depth client-side filter check
  - Agent-offline graceful skip
  - `testFire()` for synthetic events
- **Tab component** `agent-triggers-tab.tsx`:
  - Unified agent/session status bar
  - Stats overview (configured / enabled / recent events)
  - Trigger cards with live subscription status badges (`paused` / `subscribing` / `listening` / `subscribe error` / `closed` / `agent offline`)
  - Template picker (3 starter templates + blank)
  - Edit trigger dialog (source, action, session binding)
  - Test fire dialog with synthetic row builder
  - Event log with 5 status types (received / fired / success / error / skipped)
- **Persistence**: localStorage for v1 (TODO: Supabase-backed table for multi-device/team sharing)

### Endpoint Policy Clarified
Only the agent's **GUI-scripting** endpoints are wired (`/sap/confirm-to` and `/sap/process-shipment`). The Railway backend's pyrfc `/api/sap/confirm-to` is deliberately excluded since it's not integrated end-to-end.

## Inventory Management ŌĆö SAP Query Framework (new tab)

### Why
User framing: "look up a part in LX03 and return everything based on my layout, without me navigating through SAP screens." Generalize to ANY SAP transaction, not just LX03. Completes the read/write SAP story:
- **One Click Ship** = form ŌåÆ SAP write (manual)
- **Agent Triggers** = realtime event ŌåÆ SAP write (automated)
- **Inventory Management** = form ŌåÆ SAP read (query + table) ŌåÉ NEW

### Record vs. Hand-Written Question
User asked if I should design the script or they should record. Recommended **both**:
- Recording captures authoritative element IDs for their specific SAP
- Hand-written adds data extraction that recordings can't capture
- Split ownership: user records nav, engineer writes handler + extraction

For today I wrote 3 starter handlers (LX03, MB52, MMBE) based on standard SAP. If they don't match, user records and we iterate.

### Agent Changes
- New `QueryRequest` pydantic model
- Helpers: `_safe_get`, `_extract_alv_grid` (multi-shell-ID fallback), `_extract_table_control` (older scrolling tables)
- Handlers: `handler_lx03`, `handler_mb52`, `handler_mmbe`
- `QUERY_HANDLERS` registry dict
- `POST /sap/query` dispatcher endpoint
- `GET /sap/query-handlers` discovery endpoint
- Console logging: `[query]` prefix shows handler name, params, row count

### Frontend Changes
- New tab `inventory-management-tab.tsx` (~800 lines)

  - Row count + filtered count badges
