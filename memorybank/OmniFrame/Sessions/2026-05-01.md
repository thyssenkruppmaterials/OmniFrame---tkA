## Fleet-aware trigger suppression

### `agent-triggers-tab.tsx` (+25 LOC, -3 LOC)

Replaced:
```ts
const agentSideTriggersActive =
  agentStatus === 'connected' &&
  Array.isArray(agentHealth?.capabilities) &&
  agentHealth.capabilities.includes('agent-side-triggers')
```
with the fleet-aware variant:
```ts
const localAgentSideTriggers =
  agentStatus === 'connected' &&
  Array.isArray(agentHealth?.capabilities) &&
  agentHealth.capabilities.includes('agent-side-triggers')
const fleetAgentSideTriggers =
  detection.fleetHasCapability('agent-side-triggers')
const agentSideTriggersActive =
  localAgentSideTriggers || fleetAgentSideTriggers
const fleetAgentForTriggers =
  !localAgentSideTriggers && fleetAgentSideTriggers
    ? detection.fleet.agents.find((a) =>
        a.capabilities.includes('agent-side-triggers')
      ) ?? null
    : null
```

The `useAgentTriggerRuntime` hook is left untouched — its two `agentSideTriggersActive` gates (lines ~520 + ~594) already do the right thing once the input is correct (cede the `enqueueFire` call AND skip opening browser-side Realtime channels).

### Banner copy — mirrors SmartImportButton

When the LOCAL agent owns the cap → unchanged copy:
> **Agent-side triggers active**
> Agent v1.6.7 is subscribed to `rf_putaway_operations` directly and enqueues `sap_agent_jobs` rows itself — triggers fire even when this tab is closed. The browser-side runtime is dormant; manual Test Fire still works.

When ONLY the fleet has the cap → new fleet-routing copy:
> **Routing through fleet agent `USINDPR-CXA105V` · v1.6.7**
> That remote agent owns the `rf_putaway_operations` subscription and enqueues `sap_agent_jobs` rows itself — triggers fire even when this tab is closed. The browser-side runtime here is dormant to avoid overwriting the agent's "Omni Agent" attribution; manual Test Fire still works.

The naming + format intentionally mirrors the SmartImportButton's `via <hostname> · v<ver>` sub-label so users see consistent fleet-routing language across surfaces.

### Why this is the right fix
- The fleet agent's own `rf_putaway_operations` subscription is the source of truth for trigger work — same model as v1.6.4 introduced for the local case. There's no double-fire only when the BROWSER doesn't also fire.
- Hard-coding "v1.6.7" or "Citrix only" in the gate would break the moment another agent topology shows up. Capability-gated routing (per [[Patterns/Fleet-Aware-Smart-Routing]]) handles that automatically.
- Mirroring the SmartImportButton's pattern keeps the routing decision visible in the UI — never silently fan out without telling the user which agent is doing the work.

## Fix — Supabase MCP showing "No tools, prompts, or resources"

Unrelated infra hiccup the user noticed mid-session: their The IDE was showing the `supabase` MCP server entry as **"No tools, prompts, or resources"** even though `railway-mcp-server` and `obsidian-mcp` were listing tools normally and the toggle was green.

### Root cause

The `docs` feature in `@supabase/mcp-server-supabase` (default-enabled, present on every release ≥ v0.7.0 incl. v0.8.0 latest) calls Supabase's Content API at `https://supabase.com/docs/api/graphql` during `tools/list`. As of late April 2026 the API now returns a success-shaped `{ data: {...} }` body that fails the server's discriminated-union zod schema (which only allows error-shaped responses). Net effect: zod throws → `tools/list` errors with `-32603` → The host never receives ANY tool definition (not just docs ones) → UI shows "No tools". The `initialize` handshake itself succeeds (server reports `capabilities: { tools: {} }` and the version banner) which is why the toggle stays green. Confirmed by manual stdio handshake on both v0.7.0 and v0.8.0 — not a recent npm regression, a server-side API shape change.

### Fix

Updated `repo `mcp.json`` for the `supabase` entry: explicit `--features=database,functions,branching,debugging,development,storage` (whitelist excluding the broken `docs`), plus `--project-ref=wncpqxwmbxjgxvrpcake` for project scoping. Deliberately NOT adding `--read-only` — OmniFrame uses `apply_migration`/`execute_sql` heavily.

```json
"supabase": {
  "command": "/opt/homebrew/bin/npx",
  "args": [
    "-y",
    "@supabase/mcp-server-supabase@latest",
    "--project-ref=wncpqxwmbxjgxvrpcake",
    "--features=database,functions,branching,debugging,development,storage"
  ],
  "env": { "SUPABASE_ACCESS_TOKEN": "sbp_..." },
  "type": "stdio"
}
```

## v1.6.8 — agent-internal patcher race fix

After the morning's fleet-aware suppression rolled out, the user spot-checked `rf_putaway_operations` in production and discovered every recent agent-confirmed row STILL had `confirmed_by_label = NULL` / `confirmed_by_agent_id = NULL` / `confirmed_source = NULL`. The UI was falling back to `user_profiles.full_name` for the "Confirmed By" column on every agent-confirmed TO — looking exactly like the v1.6.6 attribution bug all over again.

The root cause is *not* the v1.6.6 bug returning. It's a different bug — agent-internal — that was hidden by the v1.6.6 work all along.

### Root cause (confirmed via SQL + code reading)

TWO patchers fight on every agent-side TO confirm:

1. **`_update_putaway_status`** (called from inside `confirm_transfer_order` after a successful LT12) PATCHes the LEGACY 3 fields first: `to_status='TO Confirmed'`, `confirmed_at`, `confirmed_by`. No source. No label. No agent_id.

2. **`_apply_trigger_post_patch`** (called by the job poller AFTER the handler returns) was supposed to overlay the v1.6.6 attribution columns. It PATCHed with the FULL body (legacy 3 + attribution 3) and a `skip_if = {to_status: 'TO Confirmed'}` filter — encoded as `&to_status=neq.TO%20Confirmed`. Step 1 had already flipped `to_status='TO Confirmed'`, so the filter ALWAYS matched 0 rows. PostgREST returned 200 OK with empty body. The agent logged "applied". Attribution columns silently stayed NULL forever.

The browser-side path doesn't have this bug — there's no `_update_putaway_status` call there, the browser owns all six fields. The bug only manifested under fleet-routing (yesterday's [[Implementations/Implement-Fleet-Aware-SmartImportButton]] + this morning's `agentSideTriggersActive` extension), which is exactly when production agent-side triggers started actually firing.

Full writeup: [[Debug/Fix-Agent-Dual-Patcher-Race]].

### The fix — OVERLAY-ONLY in `_apply_trigger_post_patch`

```python
body = {
    k: v for k, v in (post_patch.get("patch") or {}).items()
    if k in ("confirmed_source", "confirmed_by_label", "confirmed_by_agent_id")
}
if not table or not row_id or not body:
    return
# NO `skip_if` — the legacy `_update_putaway_status` already set
# `to_status='TO Confirmed'`. Re-applying a `&to_status=neq.TO%20Confirmed`
# filter here would always match 0 rows (the v1.6.7 race condition).
url = f"{state.supabase_url}/rest/v1/{table}?id=eq.{row_id}"
```

Five things changed in `_apply_trigger_post_patch`:
1. Body filtered to ONLY the three OVERLAY columns. Legacy 3 fields never sent.
2. `skip_if` dropped entirely (no double-write risk because we're not touching `to_status`).
3. `Prefer: return=minimal` → `Prefer: return=representation` so we can count rows.
4. New diagnostic log line prints field values + `rows_affected=N`. On `rows_affected == 0` the prefix becomes `WARN ` so the next regression of this kind shows up loud:
   ```
   [triggers] post-success PATCH rf_putaway_operations/<row_id> applied — overlay fields: source=agent_trigger_direct, label='Omni Agent', agent_id=USINDPR-Console-jsingh (rows_affected=1) (job <job_id>)
   ```
5. v1.6.7 self-healing `_TRIGGER_DROP_AGENT_ATTRIBUTION` cooldown preserved — still strips the label + agent_id columns when migration 251 isn't yet visible to PostgREST.

`_update_putaway_status` got a documentation comment (no logic change) calling out the OVERLAY pattern so future engineers reading either function understand they cooperate.

### Why this is the right fix (vs alternatives)

- **Drop `skip_if` only, keep the full body** — would have leaked a redundant UPDATE event onto the `rf_putaway_operations` Realtime channel, possibly re-triggering downstream subscribers. Overlay-only avoids the duplicate write.
- **Remove `_update_putaway_status` entirely** — would require plumbing trigger meta into every handler that today calls it, and breaks the manual-click `POST /sap/confirm-to` path which has no trigger meta.
- **Make `_update_putaway_status` set the attribution columns too** — same plumbing problem, same anti-pattern.

The overlay split keeps each function's responsibilities disjoint and mirrors how browser-side `applyPostSuccessPatch` already cooperates with the in-handler legacy patcher when running browser-mode.

### Live troubleshooting that motivated v1.7.0
After v1.6.9 install, the user reported the agent "stops for so long before doing the last TO" and the queue wasn't draining fast enough. Investigation:
- Found 2 jobs stuck in "running" state simultaneously (TO 1790022 running 1m 37s, TO 1790033 running 14s) — agent claimed a second job before the first cleared, indicating broken active-job tracking.
- DB query showed wait_secs of 200-672s for jobs created 20:22-20:43 — agent was processing work but with massive inter-job pauses.
- Console showed `[triggers] enqueue error: HTTPSConnectionPool... Read timed out. (read timeout=8)` — corporate proxy + Citrix latency exceeding 8s timeout.
- Console showed dozens of `[realtime] connected to wss://...` lines per minute — WebSocket churn.
- Manually reset the 2 stuck jobs to "failed" via SQL so the agent could move on; queue resumed draining.

### v1.7.0 fix (5 surgical changes)
1. **Drain-back-to-back job poller** — claims next job immediately after completing one, with 5→60s exponential idle backoff. 50-job chain cap. Logs `[jobs] Drain mode: <N> jobs claimed in last burst.` for bursts of 5+.
2. **Stuck-job watchdog** — daemon thread tick every 30s, marks any job running >120s as `failed` with reason "Watchdog: job exceeded 120s timeout — likely SAP session hang", then releases the lease. Configurable via `OMNIFRAME_JOB_WATCHDOG_TIMEOUT_SECONDS` env var.
3. **Active-job tracking** — `AgentState.active_job_id` + lock prevents claiming a new job while one is in-flight (closes the v1.6.9 race that allowed two simultaneous "running" rows).
4. **HTTP timeout normalization** — new `_supabase_request` helper applied to all 17 Supabase HTTP call sites: 30s timeout (was 4-10s spread), single retry on Timeout/ConnectionError after 2s. Eliminates spurious `enqueue error` lines.
5. **Stable Realtime singleton** — sticky `_realtime_started` flag prevents duplicate WebSocket subscriptions; reconnect floor raised 1s→5s. The library still handles transient drops; we just don't open parallel connections.

Three new capabilities: `'job-drain-mode'`, `'stuck-job-watchdog'`, `'realtime-singleton'`.

## v1.7.3 Rebuild + Upload (LT10/LT22 bulk-export hardening)

User reported LT10 query was paging down on the SAP GUI AFTER the `%pc` export saved successfully. Confirmed via code reading: `_extract_via_pc_export` was raising during the parse step (column-boundary detection mismatch on this LT10 variant), and `_extract_alv_grid`'s `except Exception` clause silently fell back to `_extract_sap_list_output` which paginates via Ctrl+PgDn. Result: user saw save-dialog complete → 16+ pagedowns → finally returns. v1.7.3 fixes the fallback semantics + restructures `handler_lt10` to skip the wasteful ALV probe.

### Fix breakdown
1. **Two-phase `_extract_via_pc_export`** — `_PcPreCommitError` (before file is saved, fallback-safe) vs `_PcPostCommitError` (file already on disk, fallback wrong, re-raise).
2. **`_extract_alv_grid` selective fallback** — only falls back to pagination on `_PcPreCommitError`. Post-commit failures or unknown errors propagate.
3. **`handler_lt10` skips ALV probe** for `storage_type='*'` — goes straight to `_extract_via_pc_export`. Saves COM round-trips walking the control tree on a screen LT10 never has an ALV grid on.
4. **Same pattern in `lt22_import.py`** — pre-commit/post-commit split, same selective fallback.
5. **New diagnostic prints** — `[query] Starting %pc bulk export — file will save to TEMP and be parsed in-place. No pagination needed.` and `[query] %pc bulk export complete: <N> row(s), <C> columns in <T>s. No GUI pagination performed.`
6. **New capability** — `'bulk-export-no-fallback'`.

## v1.6.9 — backfill poller + dedup hardening

After the v1.6.8 attribution fix shipped this morning, the user spot-checked production and found a different problem. Recent agent confirms WERE working correctly — DB query showed 5 fresh `rf_putaway_operations` rows with `confirmed_by_label = 'Omni Agent'`. BUT 17 OTHER rows from the same 4-hour window were sitting at `to_status='Completed'` with `confirmed_at = NULL`. Same agent. Same org. Same trigger config. The agent had simply never seen the Realtime event for those rows.

User-facing report: "It is not properly confirming the pending TOs or the completed putaways. It is also skipping them. There are issues still there, and not everything is getting confirmed properly."

This is **not** a regression of any prior bug. It's a previously-invisible failure mode — Realtime is at-most-once, not at-least-once — that became visible because v1.6.8 made it possible to reliably count "rows the agent should have processed".

### Root cause

The agent's `rf_putaway_operations` Realtime channel is the ONLY trigger source today. Every link in the WebSocket chain can drop events:

- Agent restart / EXE upgrade — bridge gap.
- WebSocket reconnect blip — the `_run_async` exponential-backoff loop spends 1-32s in retries; rows that arrive in that window are gone.
- Supabase Realtime publication lag during maintenance windows.
- Citrix VDA hibernation — close laptop → WebSocket dies → events dropped.
- pg_cron interrupting Realtime publications.

The user's 17-row backlog was a mix of WebSocket reconnect blips and Citrix hibernation overnight. Full writeup: [[Debug/Fix-Missed-Realtime-Events-Backfill]].

### The fix — three additive changes

**Pure additive.** No existing handler touched, no trigger semantics changed, no frontend logic changed beyond the version-string bump.

#### Fix 1 — Backfill poller daemon thread

New `_start_trigger_backfill_poller()`. Wakes every 60s; first poll runs 10s after boot to give Realtime a head start for the common case (no backlog). For each `_HARDCODED_TRIGGERS` entry, fires a bounded PostgREST query:

```
GET {supabase_url}/rest/v1/rf_putaway_operations
    ?select=*
    &to_status=eq.Completed
    &confirmed_at=is.null
    &is_mca_workflow=not.is.true
    &confirmed_source=is.null
    &organization_id=eq.{org_id}
    &created_at=gte.{(utcnow - 24h).isoformat()}Z
    &order=created_at.asc
    &limit=50
```

Each row → `_hardcoded_trigger_match` (defensive) → `_enqueue_trigger_job` (same path Realtime callback uses). The new `backfill_filter` key on each `_HARDCODED_TRIGGERS` entry holds the per-trigger PostgREST filter so future triggers add their own without code changes to the poller loop.

Bounded scope:
- `LIMIT 50` per poll → 17-row backlog drains in 1 poll, 1000-row backlog drains in 20 polls (20min). Prevents runaway flooding of `sap_agent_jobs`.
- 24h lookback → first boot after a long downtime doesn't try to backfill ancient rows.
- 1 query per trigger per poll → cheap by design.

Quiet by design — only logs `[backfill] poll: N matched, M queued, K skipped (dedup)` when M > 0. Steady-state silent.

#### Fix 2 — Bounded TTL dedup cache

New `_recently_queued_rows: OrderedDict[str, float]` with 5-min TTL + 1000-entry LRU eviction. Pre-1.6.9 the dedup was purely PostgREST `idempotency_key` UNIQUE → 409, which is correct but expensive (HTTP round-trip per double-fire) and noisy (one log line per 409).

```python
_DEDUP_TTL_SECONDS: float = 300.0
_DEDUP_MAX_ENTRIES: int = 1000

_recently_queued_rows: OrderedDict[str, float] = OrderedDict()
_dedup_lock = threading.Lock()


def _is_recently_queued(row_id: str) -> bool:
    if not row_id:
        return False
    with _dedup_lock:
        _purge_expired_dedup_entries(time.time())
        return row_id in _recently_queued_rows


def _mark_recently_queued(row_id: str) -> None:
    if not row_id:
        return
    with _dedup_lock:
        now = time.time()
        if row_id in _recently_queued_rows:
            _recently_queued_rows.move_to_end(row_id)
        _recently_queued_rows[row_id] = now
        while len(_recently_queued_rows) > _DEDUP_MAX_ENTRIES:
            _recently_queued_rows.popitem(last=False)
```

Three invariants:
1. A row that was queued and FAILED can be re-tried later (entries age out).
2. Memory is bounded (LRU + TTL purge).
3. Realtime double-fires are still suppressed within the 5-min window.

`_enqueue_trigger_job` short-circuits BEFORE the HTTP call when the row is in the cache, marks the row in the cache on enqueue success or 409, and does NOT mark on failure (so transient errors retry).

#### Fix 3 — Throttled dedup logging

Console pre-1.6.9 spammed `[triggers] dedup: ...` per Realtime double-fire — dozens of lines for one row. New `_should_log_dedup(row_id)` allows the FIRST hit per row per minute, suppresses subsequent hits silently for 60s. After 60s of quiet, the next hit logs again — so prolonged dedup storms remain visible.

### New boot prints

```

```


When the agent isn't logged in:
```

```

## v1.7.0 — throughput + watchdog

After v1.6.9 shipped the backfill poller at 16:36, the user checked the agent live in production and found it WAS working but with avoidable latency. Three console + DB observations drove a coordinated throughput pass:

1. **60-180s inter-job dwell** on back-to-back jobs even when queue depth > 0. The poller was sleeping 60s between claims when a Realtime wake-up was missed during a WebSocket reconnect blip.
2. **Stuck-job watchdog missing.** TO `1790022` claimed at 20:54:15 stuck `running` for 97+s; no in-agent mechanism to detect and release.
3. **HTTP enqueue timeouts** — `[triggers] enqueue error: HTTPSConnectionPool... Read timed out. (read timeout=8)` from corporate proxy + Citrix latency.
4. **Realtime WebSocket churn** — dozens of `[realtime] connected to wss://...` lines per minute from multiple reconnect loops racing.

Full root cause + fix writeup: [[Debug/Fix-Agent-Throughput-Latency]]. Pattern writeup: [[Patterns/Job-Queue-Drain-Mode]].

### Five surgical fixes

**(1) Drain-back-to-back job poller.** `_start_job_poller._loop` now chains claims until the queue returns empty (cap 50 jobs/burst) before falling through to a wait loop. Idle backoff exponentially ramps `_DRAIN_MIN_IDLE_SEC = 5s` → `_DRAIN_MAX_IDLE_SEC = 60s` on consecutive empty polls; resets to 5s on any claim hit (including Realtime wake-up). Bursts of 5+ jobs log `[jobs]   Drain mode: <N> jobs claimed in last burst.`. Pulled claim+dispatch+complete/fail logic into `_claim_and_dispatch_one()` which returns True on did-work, False on queue-empty.

**(2) Active-job tracking + stuck-job watchdog.** `AgentState` gained three fields: `active_job_id`, `active_job_started_at`, and `active_job_lock: threading.Lock`. Poller sets active-job before dispatch, clears in `finally:` after complete/fail. Legacy `_job_poller_state["current_job_id"]` still mirrored for `/status` + `_build_agent_registry_row` back-compat. Claim lease dropped 300s → 90s.

New `_start_job_watchdog_thread` daemon wakes every 30s, checks `state.active_job_id` + `state.active_job_started_at`; if > 120s (tunable via `OMNIFRAME_JOB_WATCHDOG_TIMEOUT_SECONDS` with 10s floor + fallback on parse error), logs `[jobs]   WATCHDOG: job <id> running >Ns — likely stuck. Marking failed and releasing.`, calls `jobs_fail()` with step=`watchdog`, clears active-job state under the lock, kicks the poller. Does NOT try to kill the hung COM call — Python can't safely — but frees DB state so the queue keeps draining while the user manually unsticks SAP.

**(3) HTTP hardening.** New `_supabase_request(method, url, **kwargs)` helper injects `timeout=30s` (was 4-10s spread across 17 call sites) + single retry on `requests.exceptions.Timeout` / `ConnectionError` after 2s sleep. Every `requests.post/patch/get` to Supabase in agent.py now routes through the helper. Retry prints `[http] transient <ExcType> on <METHOD> <url[:120]> — retrying once after 2s` so ops can see when the helper saved an otherwise-failed call. Does NOT swallow 4xx/5xx — caller still inspects the Response.

**(4) Stable Realtime singleton.** New sticky module-level `_realtime_started: bool` flag. `_start_realtime_subscription` bails immediately if already armed — prevents `/supabase/login` or `_on_startup` from spawning a second reconnect loop that races with the first. Never reset until process exits; if the thread really crashes, user sees `[realtime] thread crashed: ...` in log and restarts the agent. Reconnect backoff floor bumped 1s → 5s (dampens churn when `client.listen()` returns cleanly for library-internal reasons). New explicit `[realtime] listen() returned cleanly — socket closed without exception; reconnect in <N>s` log line distinguishes clean returns from exceptions.

**(5) Version bump + capabilities.** `AGENT_VERSION = "1.7.0"` with full throughput-pass banner. `LATEST_AGENT_VERSION = '1.7.0'` in `agent-fetch.ts`. Three new capabilities advertised in `/health.capabilities`: `job-drain-mode`, `stuck-job-watchdog`, `realtime-singleton`. Frontend doesn't gate on any of them (pure backend throughput improvements, no UX change).

## v1.7.1 — realtime crash-loop containment

User left v1.7.0 agent running on a Citrix VDA. ~25 min idle later: queue drain frozen, heartbeat stopped (sap_agents.last_seen_at stuck), no responses for 25 minutes. Console.txt (485 KB, ~8000 lines) flooded with `Task exception was never retrieved` traceback bursts.

Root cause: bug in `realtime>=2.x` Python library. WebSocket dropped (Citrix hibernate / corporate proxy idle close), library's internal `_listen_task` crashed inside `_on_connect_error → _reconnect()` calling `asyncio.wait()` on an empty pending-tasks set → `ValueError: Set of Tasks/Futures is empty.`. Library didn't catch it, so it escaped to asyncio's default exception handler which prints multi-line tracebacks to stderr. Each crash spawned a new listen task that died the same way. Stderr lock contention starved every other thread (heartbeat, job poller).

### Fix — four defensive layers, surgical containment only

NO handler touched. NO trigger semantics changed. NO frontend logic changed beyond `LATEST_AGENT_VERSION` bump. Files: `omni_agent/agent.py` (+~512 LOC), `omni_agent/requirements.txt` (+18 LOC), `src/features/admin/sap-testing/lib/agent-fetch.ts` (+~56 LOC, comment + version constant only).

**(A) asyncio loop exception handler** — `loop.set_exception_handler(_realtime_loop_exception_handler)` installed inside `_run_async` BEFORE the AsyncRealtimeClient is constructed. Suppresses known `ValueError('Set of Tasks/Futures is empty')` and `websockets.exceptions.ConnectionClosedError` / `ConnectionClosedOK` quietly. Catch-all branch logs once per exception class per minute via `_realtime_should_log(key)` so a never-before-seen failure mode is visible without flooding. Single change eliminates 99% of stderr.

**(B) Sliding-window circuit breaker** — new `_RealtimeCircuitBreaker` (deque-backed, thread-safe). Every suppressed exception increments a 60s window counter; at 20 errors the circuit trips. `_disable_realtime_subsystem(reason)` logs once with `[realtime] CIRCUIT BREAKER TRIPPED — ...`, sets `state.realtime_disabled = True`, signals the asyncio loop's stop_event so it exits cleanly on its next backoff cycle, and kicks the job poller so the new tightened backoff takes effect immediately. Tunables (`_REALTIME_ERROR_WINDOW_SECONDS = 60`, `_REALTIME_ERROR_THRESHOLD = 20`, `_REALTIME_CIRCUIT_RESET_INTERVAL_SEC = 300`, `_REALTIME_FALLBACK_POLL_MAX_IDLE_SEC = 15`) are module-level constants — deliberately not env-tunable, keeps the runtime simple.

**(C) Threading isolation** — confirmed each background thread has its own loop / no asyncio at all. Realtime in `sap-realtime-jobs` (only async one). Heartbeat in `sap-agent-heartbeat` (sync `requests.post()`). Job poller in `sap-job-poller` (sync). Watchdog in `sap-job-watchdog` (sync). Backfill in `sap-trigger-backfill` (sync). SAP auto-connect in `sap-auto-connect` (sync). New v1.7.1 reset loop in `sap-realtime-reset` (sync). A Realtime asyncio crash CANNOT directly wedge them. The actual coupling mechanism in v1.7.0 was the stderr lock contention; layers A+B+D bound the flood, so the threads are now truly independent.

**(D) Bound stderr noise** — `logging.getLogger("realtime").setLevel(logging.WARNING)`, same for `"websockets"`, and `"asyncio"` raised to ERROR. Defense-in-depth against unpredicted exception classes that bypass our handler.

**(E) Auto-recovery** — new `_start_realtime_circuit_reset_loop` daemon (`sap-realtime-reset` thread) wakes every 5min. If the breaker is tripped: reset it, flip `state.realtime_disabled = False`, clear the v1.7.0 sticky `_realtime_started` singleton flag, re-enter `_start_realtime_subscription()`. If network has recovered (Citrix unhibernated, proxy resumed) we're back on Realtime; if not, the breaker trips again and we cycle. No EXE restart required.

**(F) Job poller fallback ceiling** — new `_resolve_idle_max()` in the job poller's `_loop` returns `min(configured_max, 15s)` when `state.realtime_disabled = True`, else the normal 60s. Re-evaluated every iteration so a trip/reset takes effect on the next backoff cycle without restarting the poller. Polling-only mode keeps inter-job dwell low while the trigger backfill poller (60s, unchanged) carries the missed-event load.

**(G) Library pin** — `omni_agent/requirements.txt` floor bumped from `realtime>=2.0.0` to `realtime>=2.29.0,<3.0`. Latest as of 2026-04-24; `_reconnect()` was refactored away around 2.5+ so the upstream bug is fixed. Containment layers stay as defense-in-depth so a future 2.x regression is bounded.

## v1.7.3 — LT10 bulk export hardening

User bug report: running LT10 with `storage_type='*'` (warehouse-wide) showed the `%pc` Save-As dialog flash, the file land on disk, and then SAP visibly start paging down via Ctrl+PgDn for 5+ minutes. The bulk-export path that v1.6.3 introduced specifically to AVOID pagination was being silently undone by a too-greedy fallback chain.

### Root cause
`_extract_via_pc_export` raised plain `Exception` from BOTH the pre-Save dialog setup AND the post-Save file-read/parse phase. The caller in `_extract_alv_grid` caught everything with a single `except Exception` that fell through to `_extract_sap_list_output(sess)` which paginates. So a single quirk in the parsed file (extra header banner lines, read-only %TEMP%, unexpected dash-row pattern) silently turned into a 5-minute GUI walk over data we already had on disk. `handler_lt10` compounded this by probing 11+ candidate ALV shell IDs + walking the entire window tree + probing for a `GuiTableControl` before ever reaching the bulk-export path — LT10 always renders a classic list output, never an ALV grid, so all those probes were wasted COM round-trips.

### Three surgical fixes
1. **Two-phase error taxonomy** — new `_PcPreCommitError` (raised before pressing Save — fallback-safe) and `_PcPostCommitError` (raised after pressing Save — NOT fallback-safe). `_extract_via_pc_export` now wraps Steps 1-3 (typing %pc, picking Unconverted, opening Save-As) in a single try/except that re-raises as `_PcPreCommitError`; everything from pressing `btn[11]` onwards raises `_PcPostCommitError` on failure.
2. **`_extract_alv_grid` fallback chain narrowed** — `_PcPreCommitError` falls through to `_extract_sap_list_output` (safe — GUI is still on the source screen); `_PcPostCommitError` re-raises as a clean `Exception("Bulk export saved file but parse failed: ...")`; unknown error class re-raises (conservative, no double-burn).
3. **`handler_lt10` skips the ALV probe** — `storage_type == '*'` calls `_extract_via_pc_export(sess)` DIRECTLY with NO ALV/TableControl probe; falls back to `_extract_sap_list_output` ONLY on `_PcPreCommitError`. Specific storage_type calls `_extract_sap_list_output` directly (small result sets don't pay for bulk-export overhead).

Same pre/post-commit split applied to `omni_agent/lt22_import.py`. LT22 is even more sensitive to greedy fallback because re-paginating after a successful save can re-trigger the v1.6.2 SAP-COM crash (`(-2147023174, 'The RPC server is unavailable.', None, None)`).
