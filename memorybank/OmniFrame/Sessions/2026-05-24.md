the terminal with 16+ `ECONNREFUSED` errors over 7 minutes for

### Fix: OmniBelt kill-switch 401 in production

Root cause: three OmniBelt callsites used `credentials: 'include'` (cookies) instead of attaching the Supabase access token as `Authorization: Bearer`. FastAPI's auth dep reads JWT from the header, not cookies. Other features still have local copies — listed in the debug note for future opportunistic migration.

### Fix: OmniBelt kill-switch 500 — settings RLS + anon-key client

User tried the master kill-switch toggle in production; returned 500
with a Postgres 42501 RLS denial on `public.settings`. Two-bug fix:

1. **Backend was using `db.client` (anon-key singleton)** for every
   OmniBelt admin write. The user's JWT never propagated, so
   `auth.uid()` resolved to NULL inside RLS and every settings policy
   denied the request. Fixed in `api/routers/omnibelt.py` by adding a
   small `_user_scoped_client(request)` helper that mints a fresh
   `create_authenticated_supabase_client(token)` per request and
   wiring both admin handlers (`write_kill_switch`,
   `write_role_config`) through it. The same anon-client bug latently
   affects `write_prefs` + `write_events` — flagged as v1.5 follow-up.
2. **`settings` RLS didn't acknowledge `omnibelt.manage`.** Migration
   329 (`omnibelt_settings_rls`) applied via Supabase MCP — additive
   `settings_omnibelt_admin_rw` policy for `key LIKE 'system.omnibelt.%'`,
   gated on `has_permission('omnibelt','manage')`, accepting both
   global (`organization_id IS NULL`) and same-org rows. Existing
   settings policies left untouched.

Verified via Supabase MCP `execute_sql` simulation:
- `SET LOCAL ROLE anon; INSERT … settings;` → 42501 (matches prod
  500).
- `SET LOCAL ROLE authenticated; SET LOCAL request.jwt.claims … (admin
  user); INSERT …;` → succeeds.
- `(non-admin tka_associate)` → `has_permission('omnibelt','manage')` =
  `false`; new policy correctly denies.

log updated with a "Post-launch fixes" subsection in
[[Implement-OmniBelt-MVP]] capturing the dual root cause + the
latent prefs/events bug + a separate `settings:manage` over-grant
observation.

### Fix: OmniBelt Orb — interactivity + skin-picker escape hatch (PM)

Live user report: "When in orb mode, none of the buttons work, and
there is no way for me to change it back to another skin."

**Two bugs, one symptom:**

1. **Radial fan shell-tile clicks were silent no-ops.**
   `src/features/omnibelt/skins/orb/RadialFan.tsx::launch()` had a
   real `navigate({...})` path for `navigationUrl` tools but for the
   6 of 9 v1 tools that are shell-backed (`sap_status`,
   `inventory_lookup`, `background_jobs`, `agent_chat`,
   `quick_note`, `build_info`), it just called `close()` and did
   nothing visible. The P7 docstring's mitigation ("re-open into the
   Pill skin's Panel grid") never fires — the user is on the Orb
   skin, which suppresses the standard panel.

2. **No UI path back from the Orb skin.** `<PanelMenu>` (the `⋮`
   dropdown with the skin picker) lived only inside the standard
   `<OmniBeltPanel>` header. Orb suppresses that panel (see this
   morning's SkyStrip refactor — `SKINS_USING_SHARED_PANEL` was
   already `{ 'pill' }`). So picking Orb removed the only skin
   picker from the page — users got trapped until they wiped
   localStorage.

**Architectural lesson — codified into the pattern docs:**

> Any skin that opts out of the standard Panel MUST provide its own
> path to skin/visibility settings — otherwise users get trapped in
> that skin.

**Fix (Approach A from the brief — surgical):**

- Extract `<PanelMenuContent />` body from `<PanelMenu>` so the
  skin picker can mount under any custom trigger. Pill panel still
  uses the wrapping `<PanelMenu>` (default trigger inside Radix
  DropdownMenu); Orb mounts the same `<PanelMenuContent />` under
  its own trigger.
- Wrap the orb in a positioning anchor div. Add a small `⋮` overflow
  trigger as a SIBLING of the orb's `<button>` (HTML disallows
  nested buttons) at top-right. `e.stopPropagation()` on its
  `onClick` keeps the orb's toggle from firing alongside (the orb's
  hit area is the full bounding box rect, not the `rounded-full`
  silhouette, so the two buttons overlap by a few pixels at the
  corner). Status dot moved top-right → top-LEFT to leave the
  prime corner clear.
- New `<OrbShellPopover>` glass card anchored above the orb. When
  the radial fan launches a shell-backed tool, the orb mounts the
  popover with `<PanelContent initialActiveTool={tool}>` so the
  shell loads immediately, not after a redundant grid render. Esc +
  click-outside dismiss, mirrors `<OmniBeltPanel>` and `<RadialFan>`
  dismissal contract.
- `<RadialFan>` accepts optional `onLaunchShell`. Navigation tools
  still navigate + close as before. Shell tools delegate to the
  callback (orb opens popover); callers that omit the handler get
  the legacy silent close so future skins don't break.
- `<PanelContent>` gets optional `initialActiveTool?: ToolDef | null`
  prop. Pill panel and SkyStrip panel ignore it; orb popover uses
  it. Backward-compatible.

**Tests — 13 new across 3 files, all green:**

- `OmniBeltOrb.test.tsx`: wrap-structure regression (anchor div

### Fix: OmniBelt Agent Chat — instant-close on launch + cinematic motion polish (post-Orb fix, same evening)

User: *"When clicking into the agent chat, it instantly closes…
as we're opening each functionality, there should be some cinematic
animation motion built into it to make it look extremely fluid and
elegant when activating a function like agent chat or notes and all
kinds of stuff."*

Two coupled tasks: a layered runtime bug fix on the agent-chat
launch path, plus a motion-language extension so every tool open
across the OmniBelt feels cohesive.

**Bug — three compounding root causes:**

1. The launching click on `<ToolTile>` bubbled to the panel's
   window-level `pointerdown` capture-phase handler before
   `<AgentChatDialog>` finished mounting. The handler saw the
   click as "outside the panel" and collapsed both the panel and
   the just-mounted dialog.
2. Even after fix #1, every subsequent click *inside* the dialog
   was still treated as outside-the-panel because Radix renders
   the dialog at `document.body` (Portal), outside the panel's
   `[data-omnibelt-host]` subtree. The panel's outside-click
   helper `isOmnibeltOverlayPointerTarget` only knew about
   dropdown / tooltip portals, not tool-shell dialogs.
3. (Verified, not the issue.) Radix Dialog's `modal` default —
   pinned explicitly so a future change can't quietly flip it.

**Layered fix:**

- **A.** Extended `isOmnibeltOverlayPointerTarget` to also match
  `[data-omnibelt-overlay]`. `<AgentChatDialog>`'s `<DialogContent>`
  now carries `data-omnibelt-overlay='true'`. New
  `OMNIBELT_OVERLAY_DATA_ATTR` constant exported alongside the
  existing `OMNIBELT_OVERLAY_Z` so future shells (Sheet, Popover,
  AlertDialog, …) discoverably opt in.
- **B.** `<ToolTile>` shell-tool click handler now calls
  `e.preventDefault()` + `e.stopPropagation()`. Navigation tools
  keep the existing behaviour — the panel collapses naturally when
  the route changes anyway.
- **C.** `<Dialog modal>` set explicitly.
- **D.** New `AgentChatShell.regression.test.tsx` (+213 lines, 7
  tests) pins all four layers independently.

**Motion polish — four new constants in `lib/motion.ts`:**

- `TOOL_LAUNCH_SPRING` (stiffness 380 / damping 32 / mass 0.9 /
  restDelta 0.001). Used on the panel grid ↔ active-tool-shell
  swap (`<AnimatePresence mode='wait'>`), the agent chat dialog
  mount, the orb shell popover mount.
- `BACKDROP_FADE` (0.22 s + HOUSE_EASE). Reserved for future
  Dialog overlay overrides; documented as the canonical scrim
  transition.
- `CONTENT_STAGGER` (delayChildren 0.08 / staggerChildren 0.04).
  Applied to the dialog body so header → message list →
  composer cascades after the container settles.
- `TILE_PRESS_TRANSITION` (stiffness 600 / damping 30 / mass 0.5).
  Applied to `<ToolTile>` whileTap / whileHover so the press reads
  as a tactile tick instead of a soft squish.

`<MotionConfig reducedMotion='user'>` at the host already
collapses springs to instant; every motion node also reads
`useReducedMotion()` directly and flattens initial/animate/exit
variants under the user preference (belt + suspenders).
