User reported "typing issues and all kinds of stuff" in the Standard Work builder (`/apps/standard-work?tab=settings`, Templates & Settings tab ‚Üí Edit ‚Üí right-rail Properties panel).

### Root cause
Classic controlled-input vs. TanStack Query refetch race. The properties panel synced local editor state from props on every reference change; the parent rebuilt the `item` reference whenever the items query refetched (after every debounced auto-save). Mid-keystroke refetches stomped local state with stale server values.

## Outbound import ON CONFLICT regression (evening)

User reported "Import failed: there is no unique or exclusion constraint matching the ON CONFLICT specification" on the Outbound Data Manager's **Import from File** (clipboard TSV/CSV) path. Every clipboard import was 100% broken in production.

### Root cause

The perf-review fix earlier today flipped `bulkInsertOutboundData` to `.upsert(‚Ä¶, { onConflict: 'organization_id,delivery,transfer_order_number,material,batch,source_storage_bin', ignoreDuplicates: true })`. PostgREST emits `INSERT ‚Ä¶ ON CONFLICT (cols) DO NOTHING`, which requires a unique constraint/index on EXACTLY those plain columns.

The only matching unique index was `idx_outbound_to_data_unique_record` (from migration 047), built on `COALESCE(col, '')` **expressions** ‚Äî PostgreSQL's `ON CONFLICT (cols)` does not match expression indexes. So every import failed.

Fleet-Agent / LT22 path was unaffected (it writes to `sap_outbound_to_imports` staging).

### Fix ‚Äî Migration 320

File: [[supabase/migrations/320_outbound_to_data_unique_plain_columns.sql]]

1. Safety dedupe (`ROW_NUMBER() OVER (PARTITION BY ‚Ä¶) ORDER BY created_at DESC NULLS LAST, id DESC`).
2. `DROP INDEX idx_outbound_to_data_unique_record`.
3. `ALTER TABLE outbound_to_data ADD CONSTRAINT outbound_to_data_unique_record UNIQUE NULLS NOT DISTINCT (organization_id, delivery, transfer_order_number, material, batch, source_storage_bin)`.

`UNIQUE NULLS NOT DISTINCT` (PG 15+) preserves the NULL-equality semantic the original COALESCE-to-empty-string index was emulating, AND it works with `ON CONFLICT (cols)`.

Applied via Supabase MCP `apply_migration`. Smoke test (DO block running the exact `INSERT ‚Ä¶ ON CONFLICT (cols) DO NOTHING` with a `WHERE false` guard) parsed clean post-migration.

### Frontend tweak

Replaced the misleading comment in `bulkInsertOutboundData` that claimed PostgREST emits `ON CONFLICT (idx_outbound_to_data_unique_record)` (it doesn't, it emits a plain column list). The comment now points at migration 320 and the new debug note.

## Bulk Role Change Runtime Fix (18:55 EDT)

User hit `Failed to complete bulk action: this.updateUserRole is not a function` after the earlier bulk role change UX shipped. Root cause: `useUserManagement` passed the static method by reference ‚Äî `mutationFn: UserManagementService.bulkUpdateUsers` ‚Äî so TanStack Query rebound `this` to the mutation options object and every `this.X(...)` dispatch in the switch threw. Same trap had been latent for activate/deactivate/suspend/terminate/delete/send_invitation as well; nobody had clicked them.

Fixed in two places:
1. Hook: wrapped `mutationFn` in an arrow so the call site rebinds `this` correctly.
2. Service: replaced `this.X` with `UserManagementService.X` inside `bulkUpdateUsers` (and one stray `this.getUserById` in `getUserPermissions`) so future detached callers can‚Äôt reproduce the bug.
