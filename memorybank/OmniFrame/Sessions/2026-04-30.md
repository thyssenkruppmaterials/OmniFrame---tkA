## v1.6.4 â€” agent-side triggers + auth fixes

### Backend (omni_agent/agent.py, +260 LOC)
- **Mounted the missing routes**: `GET /supabase/session` (read-only login probe) and `POST /supabase/logout` (clears `state.supabase_token`/`org_id`/`agent_token`, tears down realtime). v1.6.3 had advertised these as capabilities and exempted the paths from the auth middleware, but the actual handlers were never written â€” hence 404 (no token sent) or 401 (stale token race) on every poll.
- **Fixed Realtime auth**: `realtime>=2.x` uses the `AsyncRealtimeClient(url, token=...)` constructor `token` arg as the `?apikey=` query param. v1.6.3 was passing `state.supabase_token` (user JWT) there, which Supabase rejected with HTTP 401 BEFORE accepting the WS upgrade. v1.6.4 passes `state.supabase_key` (anon key) to the constructor and the JWT via `set_auth()` per the Supabase Realtime protocol.
- **Agent-side trigger evaluator**: agent now subscribes to a SECOND Realtime channel â€” INSERT+UPDATE on `rf_putaway_operations` â€” and runs a hardcoded trigger list server-side. On match, it INSERTs `sap_agent_jobs` rows directly (idempotency_key=`trig:<trigger-id>:<row-id>`, priority=50) so the existing job poller picks the work up. The post-success DB patch lives in `payload.__omni_trigger_meta.post_success_patch` and is applied by the poller after a successful SAP dispatch. New capability `agent-side-triggers`. See [[Debug/Fix-Agent-Triggers-Browser-Dependency]] for the full root cause.

## v1.6.7 â€” self-healing schema-cache fallback

User flagged that v1.6.6's `_TRIGGER_DROP_AGENT_ATTRIBUTION` (and the v1.6.5 `_REGISTRY_DROP_PROCESS_STARTED_AT` it copied from) was correct in spirit but had a forever-disable bug: after migration 251 lands and PostgREST's schema cache is stale for the first ~2 minutes, the agent's first PATCH against `rf_putaway_operations` 400s with `"Could not find the 'confirmed_by_label' column ... in the schema cache"`, the agent flips the boolean to True, and every subsequent PATCH for the rest of the process lifetime strips those fields â€” even after PostgREST's cache has caught up. Net effect: an honest agent attribution feature that quietly stops working until the user thinks to restart the EXE. Same hazard with `process_started_at` (migration 250 lag).

### Backend (omni_agent/agent.py, +56 / -23 LOC)

- **`_SchemaFallbackFlag` helper class** â€” self-healing fallback primitive. Once tripped, suppresses a feature column for 5min, then auto-retries the FULL schema on the next call; success keeps the flag cleared, failure restarts the cooldown. Lifecycle prints `[schema-fallback] <label>: tripped (<reason>); will retry full schema in 300s.` / `cooldown expired, re-attempting full schema on next call.` / `cleared (full schema works again).` See [[Patterns/Self-Healing-Schema-Fallback]].
- **Migrated existing flags** â€” `_REGISTRY_DROP_PROCESS_STARTED_AT` (label `sap_agents.process_started_at`) and `_TRIGGER_DROP_AGENT_ATTRIBUTION` (label `rf_putaway_operations.agent_attribution`) both wrapped in `_SchemaFallbackFlag(â€¦, cooldown_seconds=300)`. Reads change from `if FLAG:` to `if FLAG.active:`. Writes change from `FLAG = True` to `FLAG.trip(reason)`. Successful patches with the full schema call `FLAG.clear()` (no-op if already clear, but covers the recovery case).
- **`global` declarations dropped** â€” we now mutate the `_SchemaFallbackFlag` instance instead of rebinding the name, so neither `_upsert_self_in_registry` nor `_apply_trigger_post_patch` need `global â€¦`.
- **New capability** `self-healing-schema-fallback`. Pure backend hardening â€” no frontend gating needed â€” but advertised so capability dashboards show it.
- **Banner bumped** â€” `AGENT_VERSION = "1.6.7"` with the prepended note: `Self-healing schema-cache fallback so a transient PostgREST cache miss doesn't permanently disable feature columns.` v1.6.6 carry-over text preserved.

### Frontend (src/features/admin/sap-testing/lib/agent-fetch.ts, +8 / -1 LOC)

- No frontend logic change â€” this fix is pure backend defensive hardening.
