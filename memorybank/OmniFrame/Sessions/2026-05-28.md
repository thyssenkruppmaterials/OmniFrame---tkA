## Putaway auto-confirm: 29 stuck → 4 (SAP 619 layout race + bad-warehouse data)

Investigation found 29 `rf_putaway_operations` rows wedged at `Pending TO Confirm` while the rest of the fleet auto-confirmed normally. Two root-cause buckets: **27** failed on the transient SAP GUI `(619) 'control could not be found by id.'` layout race (4 workers hammering one pinned session; `sap_audit_log` burst was 17 OK / 44 619-fails on the same idempotent LT12 path), and **2** on bad-warehouse data (`-03` = truncated tail of location `RO-60-B-03`; `H5N` = garbled `WH5`). They wedged permanently because the trigger evaluator enqueues `sap_agent_jobs` with the table-default `max_attempts=1`, and the backfill re-queue loop (mig 289) only covers a 24h lookback.

Full write-up: [[Fix-Putaway-Confirms-Stuck-At-29-Layout-Race]].

### Fix 1 — re-enqueue the 27 (Supabase write)
- `UPDATE rf_putaway_operations SET updated_at=now()` scoped to the exact 27 ids (not a blanket `status` filter). **23/27 flipped to `TO Confirmed`** within minutes.
- Hit the documented same-day idempotency-key collision (`trig:...:<unix_day>`): the 4 that failed again on the first bump can't be re-fired the same day (their `:20601` key already exists) — they need a next-day bump.

### Fix 2a — migration 331 (applied via Supabase MCP)
- `331_confirm_to_jobs_max_attempts.sql`: `BEFORE INSERT` trigger raising `max_attempts=3` for `/sap/confirm-to` ONLY (endpoint-scoped so long-running LT10 jobs keep `max_attempts=1` — avoids the mig-291 phantom-re-execution regression). **Verified live:** new confirm-to job `415e28fc` carried `max_attempts=3`.

### Fix 2b — agent.py 619-retryable classifier (workspace-only; REBUILD REQUIRED)
- `_is_retryable_sap_error(...)` classifier (terminal markers win over transient); `confirm_transfer_order` tags transient 619 as `{retryable, step:'sap_layout_race'}`; job poller leaves a retryable job `running` (instead of failed) when `attempts < max_attempts` so the mig-291 lease-expiry re-claim retries it (bounded). Test `omni_agent/tests/test_confirm_to_retryable_error.py` (12 cases). **Needs an agent EXE rebuild to take effect in production** — not done here.

### Fix 3 — corrected the 2 bad-warehouse rows
- Both `-03` (TO 7292240) and `H5N` (TO 7292943) → **`WH5`**, high confidence (bin history `RO-60-B-03`→19/19 WH5, `RN-55-A-01`→12/12 WH5; `729xxxx` TO range = WH5; driver same-day pattern). TO 7292240 **confirmed in SAP at 22:26:15**; TO 7292943 corrected, will confirm on next-day re-fire (same-day collision).
