# 2026-05-29

## Cycle-count: simultaneous "Pull Next" swaps operators' aisles (diagnosis)

Investigated the production report that two operators pressing "Pull Next Count" at
the same time get **swapped into each other's aisle / jump zones** instead of
sweeping their own aisle. Diagnosis-only (no code/migrations).

**Root cause:** `claim_next_cycle_count` takes **no claim-time serializing lock**, and
its `FOR UPDATE SKIP LOCKED LIMIT 200` candidate scan hands two concurrent claimers
**disjoint windows offset by ~200 serpentine positions**. The live queue head is
SH(1–111) + SI(112–133) + SM(134–330), so a single 200-row lock spans **three
aisles** — the losing racer's first lockable row (position 201) lands in SM, jumping
past SH and the 22-row SI aisle. The per-aisle cap (`max_counters_per_aisle=1`) and
`one_counter_per_zone` guards can't prevent it: occupancy is computed from
`worker_heartbeats` (5-min freshness, ~3–30 s cadence) joined to **committed**
`status='in_progress'` rows, so the other worker's uncommitted in-flight claim is
invisible (READ COMMITTED). The zone-exclusivity advisory lock is in the *trigger*
(fires at UPDATE/commit, after selection) so it can't steer selection — and since
SKIP-LOCKED already split the racers into different aisles, they grab different zone
locks and neither raises `ZONE_LOCKED`; both succeed into the swapped aisles.

**Evidence (live):**
- Config: 3 `serpentine_zone` rules, all `max_counters_per_aisle=1`;
  `one_counter_per_zone`, `zone_pattern=NULL` (zone = aisle prefix), **`sticky_zone=false`**.
- Near-simultaneous (≤3 s) claims switch aisle **50.8%** (127/250) vs **11.8%**
  (149/1258) for isolated claims — **4.3×**.
- 131 near-simul pairs → **55 (42%) clean aisle swaps** (`A:X→Y` while `B:Y→X`),
  **57 (44%) both-linear** ≈ coin-flip = unserialized-race signature.
- Named: 17:29:08 (1.72 s) William SH→SM / Marvin SM→SH; 16:44:03 (**0.02 s**)
  David SI→SN / William SN→SI; + ~14 more.
- No `ZONE_LOCKED` in Postgres logs (consistent with displacement, not abort).

**Hypotheses:** advisory-lock/isolation = root (H2); sticky-zone tiebreaker dead
(`sticky_zone=false`, and Phase-2 stickiness only checks held in_progress rows that
don't exist between counts; no heartbeat-stickiness in Phase 2) (H3); heartbeat-lagged
occupancy = why the cap is blind, secondary (H1/H4).

**Recommended fix direction (not applied):** `pg_advisory_xact_lock` per-org (or
per-meta-zone) at the **start of Phase 2 before the candidate scan** so claims
serialize and each sees the prior committed claim; detect in-flight claims via
`assigned_to`/`reservation_started_at` not heartbeat freshness; add real Phase-2
per-worker aisle stickiness (heartbeat `current_zone` / last-completed aisle) + enable
`sticky_zone`; optionally shrink the 200-row SKIP-LOCKED lock footprint. Server-side
only — touches no `supabase.channel(...)`, so the realtime hard-ban is not implicated.

## Cycle-count thrash fix — implemented (rust-work-service)

Implemented the fix (build green, migration applied; **not deployed**). All in
`rust-work-service/src/db/queries.rs` (+190/−31) + a 1-line `models.rs` doc + migration 332.

- **Fix 1 (primary) — serialize claims:** `pg_advisory_xact_lock(hashtextextended('cyclecount_claim:<org>',0))`
  at the **start of Phase 2**, before the `FOR UPDATE SKIP LOCKED LIMIT 200` scan, so
  two simultaneous claims no longer race the 200-row window (the second runs after the
  first commits). **Lock ordering is deadlock-free:** claim-lock is outermost and the
  only other advisory lock (the zone-exclusivity trigger's `cyclecount_zone:<org>:<zone>`)
  is always acquired *later* inside the UPDATE — no path takes them in reverse, so no
  cycle. Per-org (target aisle unknown until after selection; claims are sub-second so
  serialization is negligible).
- **Fix 2 — in-flight detection:** rewrote `OCCUPIED_AISLES_SQL` to count distinct
  other assigned workers per aisle directly from `rr_cyclecount_data`
  (`status IN ('in_progress','recount')`), dropping the `worker_heartbeats` 5-min
  freshness gate → a just-claimed aisle is visible to the per-aisle cap immediately
  (no 3–30 s lag), consistent with the zone pre-filter/trigger.
- **Fix 3 — real stickiness:** new `get_sticky_aisle_tx` (most-recent ≤30 min
  assigned/completed `resolved_aisle`, fallback live `current_zone`) feeds a `$3`
  sticky tiebreaker added to BOTH the SQL ORDER BY *and* the Rust `sort_by` (the latter
  is the source of truth for the pick and previously had no stickiness). Replaced the
  doubly-dead old branch (gated off + keyed on a held in_progress row that doesn't
  exist between counts).
- **Config decision:** kept the `sticky_zone` gate (preserves per-org opt-out) and
  **enabled it for this org via migration 332** instead of making stickiness
  unconditional.
- **LIMIT 200:** left as-is (Fix 1 makes the window harmless; shrinking risks
  regressing migration 252).
- **Build/test:** `cargo build` green; `cargo test … db::queries::tests` → 4 pure unit
  tests pass (priority + sticky_rank). Integration tests in `tests/` are `#[ignore]`
  scaffolds.
- **Deploy (NOT run):** `cd rust-work-service && railway up --service rust-work-service --environment production`.
- Full write-up: [[Debug/Investigate-Cycle-Count-Simultaneous-Claim-Aisle-Thrash-2026-05-29]] → "Implementation (2026-05-29)".
