# Session 2026-06-07

## Kit grid Status shows the real stage, not a flat "In Progress"

The grid Status column showed the coarse `kit_build_status` (stuck on "In
Progress" through picking + kitting). `getKitGridData` now aggregates per-kit
picked/kitted/inspected/on-dock progress while grouping by serial and derives
a granular stage (`picking` / `picking_complete` / `kitting` / `kit_built` /
`kit_inspected` / `completed`, else raw) into a new display-only
`kit_stage_status`. Raw `kit_build_status` untouched so tab/search logic is
unaffected. `StatusBadge` gained picking/picking_complete/kitting; Status
column renders the derived stage; search + CSV updated. `tsc` + ESLint clean;
FE+service only. See [[Kit-Grid-Derived-Stage-Status]].

## Variance approve no longer raises spurious "modified in another window"

Single-row approve (and assign) on the Inventory Counts page failed ~91% of
the time with "Count was modified in another window — refreshing" while
mass-approve worked. Root cause: the single-row CAS ran on `work_tasks`
(`.eq('id', countId).eq('updated_at', expected)`) but the UI reads
`rr_cyclecount_data` — most counts have no projection row (flag ON, only
72/746 matched on tenant `c9d89a74`) and `updated_at` is set by different
clocks, so `count===0` threw `ConcurrencyError` and the handler early-returned
before the real legacy write. Moved optimistic concurrency onto the
authoritative `rr_cyclecount_data` write (the table+timestamp the UI read);
`work_tasks` sync is now best-effort by `source_id` with no CAS, mirroring
`massApproveVariance`. `work.service` `approveVariance`/`updateAssignment`
return affected-count instead of throwing; unused `ConcurrencyError` import
dropped. `tsc -b` + ESLint clean, cycle-count hook tests pass.
See [[Fix-Variance-Approve-Spurious-Concurrency-Error]].
