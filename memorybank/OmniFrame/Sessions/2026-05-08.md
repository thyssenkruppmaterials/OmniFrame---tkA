## Subsection — worker_heartbeats stale `task_type` fix

Follow-up debug pass closing the two open items from yesterday's [[Fix-RF-Activity-Step-Source-Confusion]] (Phase 0(C) follow-up #1 + #2). The supervisor's `<LiveOperatorStatus>` Tab 1 was rendering every operator card with `cycle_count` as the work type — even when the operator was on Outbound Apps, Inbound Part Transfer, or any other RF view — because two work-engine-path defects compounded:

1. `<RFInterface>`'s `useWorkerHeartbeat({ taskType: 'cycle_count', … })` was hardcoded.
2. `<RFCycleCountUnified>` only fired `onTaskChange(null)` when its INTERNAL `currentTask` cleared, NOT on its unmount — so navigating cycle-count → home → another RF view with a still-claimed task left the parent's `currentTask` / `currentZone` set, which kept feeding stale data to the heartbeat.

**Phase 0 outcome:**
- `useWorkerHeartbeat` API: `{enabled, interval, taskId, taskType, zone, location}` → `workServiceClient.sendHeartbeat(…)` → `rust-work-service::queries::upsert_heartbeat` writes `worker_heartbeats.current_task_type` directly.
- `worker_heartbeats.current_task_type` is `VARCHAR(50)` with **no CHECK constraint** (verified via Supabase MCP `pg_constraint` query — only constraint on the table is `status_check`). Production currently has only `cycle_count` because of the hardcoding bug. Free-form, so we can write any descriptive snake_case label.
- This `task_type` is NOT a key into the dispatcher's `DispatchStrategyRegistry` (claim-time `task_type` is supplied separately by `claim_next`); purely descriptive for the supervisor card.
- Sibling RF form sweep — grep `onTaskChange|onActiveTaskChange|onTaskClaimed|onCurrentTaskChange|onZoneChange|onLocationChange` across `src/components/ui/`. Only `rf-cycle-count-unified.tsx` accepts a parent task callback. Phase 3 of the brief was a no-op for siblings (`rf-putaway-form.tsx`, `rf-inbound-part-transfer-form.tsx`, `rf-picking-form.tsx`, `rf-build-kit-form.tsx`, `rf-inspect-kit-form.tsx`, `rf-kitting-picking-form.tsx`, `rf-grs-cycle-count-form.tsx`, `rf-empty-location-material-dialog.tsx`, `rf-task-claim.tsx`, `rf-sap-migo-form.tsx`, etc.).

**What shipped:**

- `src/features/rf-interface/rf-interface.tsx` (~+47 LOC, ~−5 LOC). Module-level `RF_VIEW_TO_TASK_TYPE` lookup hoisted above `<RFInterface>`; replaces hardcoded `'cycle_count'`. ## Subsection — Auto-recovery for missed putaway confirms (rust-work-service v0.1.35 + migration 289)

The agent had recovered by 21:27+ UTC and was confirming fresher TOs cleanly, so the underlying SAP path was healthy — only the earlier window's failed jobs were orphaned. **Diagnosis:** Category 5 ("Agent claimed but failed silently"). Each stuck row had a `sap_agent_jobs` row in `status='failed', attempts=1, max_attempts=1` with errors like `(619, 'SAP Frontend Server', 'The control could not be found by id.')`, `No active SAP GUI session found`, or `Watchdog: job exceeded 120s timeout`. Once a job hit terminal `failed`, nothing in the stack ever revisited it. The earlier-day workaround of `UPDATE rf_putaway_operations SET updated_at = now()` was insufficient because the trigger evaluator's `INSERT ... ON CONFLICT (organization_id, idempotency_key) DO NOTHING` collides with the same-day `<unix-day>`-suffixed key.

**What shipped:**

Two branches: requeue stale `failed` jobs (clear claim/heartbeat fields, reset `attempts=0`, raise `max_attempts`, append breadcrumb to `error`) and replay NOTIFYs for true orphans. Trigger UUIDs not hardcoded — function joins through `agent_triggers` so future tenants light up automatically.

## Fix phantom "Unassigned" department in Customer Portal charts

User flagged that the **Avg Open Days by Department** chart on the
Customer Metrics tab was showing an `Unassigned` bar even though they
insisted no ticket was unassigned. Diagnosed as
`ticket.ilc_department || 'Unassigned'` — a frontend-only fallback in
`src/routes/_authenticated/apps/customer-portal.tsx` that bucketed any
ticket with a blank/whitespace ILC department into a synthetic
"Unassigned" group. Aggregating all 4,247 historical tickets surfaced
legacy rows from before the field was populated.

Fixed by switching to `ilc_department?.trim()` and skipping rows that
are blank instead of inventing a bucket. Applied to both
`AvgOpenDaysByDeptChart` and `OpenRequestsByDeptChart` (rollup +
dropdown). Status fallback label renamed from `Unassigned` to `Blank`
for accuracy. Did not touch `RequestsByTypeChart` (different field,
user didn't report).

See [[Debug/Fix-Customer-Portal-Phantom-Unassigned-Department]].


## 20:50 — Stuck Assignments "Release" rejected error (K3 / Jai Singh)

Count Settings → Zone Rules → Stuck Assignments showed `1 reserved · last seen 52m ago`. Clicking Release / + Unassign produced repeated `Count is not currently assigned / in progress` toasts. Bulk Release All silently no-op'd.

Root cause: migrations 231 (soft auto-release) + 233 (view exposes reserved rows) shipped, but the release RPCs were never widened. Their precondition still required `status IN ('in_progress','recount')` so soft-released `pending+assigned` rows were unreachable from the UI.

Fix: migration 290 (`290_release_stuck_includes_reserved.sql`) widens both `release_stuck_cycle_count_assignment` and `release_all_stuck_cycle_count_assignments` to accept `pending` rows with `assigned_to NOT NULL`. Soft path is idempotent on already-pending rows; hard `+ Unassign` now actually clears stale reservations. UI `StuckZonesCard` split to render `[Release][+ Unassign]` for active ids and `[Reserved badge][+ Unassign]` for reserved ids so the user isn't tempted to keep clicking a no-op Release.

Applied via Supabase MCP. `npx tsc -b --noEmit` clean. Vitest baseline unchanged (228/24 pass/fail — pre-existing).
