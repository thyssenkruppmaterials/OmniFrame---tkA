# Session 2026-05-06 — Fix UTC-midnight PATCH bug in `_update_putaway_status` (v1.8.3)

## Goal

Fix the production bug: 19 `rf_putaway_operations` rows that the agent successfully confirmed in SAP between 00:21–00:31 UTC May 6 stayed at `to_status='Completed'` and `confirmed_at=NULL` because they were created on May 5 22:00–23:00 UTC and the legacy PATCH filter `created_at >= 2026-05-06` rejected all of them.

User had already manually backfilled the 19 rows; the task was to fix the bug so the same scenario can't happen again.

## Root cause

`_update_putaway_status` filtered by `created_at >= today` (UTC). When the agent processed a TO across the UTC midnight boundary, the PATCH silently no-op'd. PostgREST returned 200 OK with empty body, the agent logged "patched 0 rows", and the OVERLAY columns (set by the v1.6.8 `_apply_trigger_post_patch` which uses `id=eq.<row_id>`) made the row LOOK like it had been confirmed — except the legacy 3 fields (`to_status` / `confirmed_at` / `confirmed_by`) silently stayed unset.

The v1.6.9 backfill poller + v1.7.0 drain mode increased the surface area: the agent now processes up-to-24h-old straggler rows in fast bursts that can cross midnight UTC.

## What shipped — v1.8.3

### `omni_agent/agent.py` (+~70 LOC, banner update)

- `_update_putaway_status(to_number, warehouse, row_id=None)` — when `row_id` is provided, PATCHes by `id=eq.<row_id>&to_status=neq.TO%20Confirmed` (exact row, no date guessing); when missing, falls back to a 48-hour `created_at` window so a same-day retry still hits AND the UTC-midnight cliff is no longer fatal.
- `confirm_transfer_order(req, row_id=None)` — accepts the optional kwarg and forwards it to BOTH `_update_putaway_status` call sites (already-confirmed branch + post-Save success branch).
- `_dispatch_job(job)` — extracts `row_id` from `payload.__omni_trigger_meta.post_success_patch.row_id` and passes it as a kwarg ONLY to endpoints in `_ROW_ID_AWARE_ENDPOINTS = ("/sap/confirm-to",)` — narrow allowlist so other handlers are opt-in.
- New diagnostic: when the PATCH affects 0 rows the agent now logs `[lt12]  WARN _update_putaway_status PATCHED 0 rows for TO {to_number} WH {warehouse} (row_id={row_id}, cutoff={cutoff}). Possible UTC-midnight crossing OR row already TO Confirmed OR RLS hid it from this user.` so future regressions are visible immediately.
- `AGENT_VERSION = "1.8.3"` with v1.8.3 banner.
- New capability `putaway-update-by-rowid` advertised in `/health.capabilities` (purely informational, no frontend gating).

### `_apply_trigger_post_patch` is UNCHANGED

Already uses `id=eq.<row_id>` since v1.6.8 — was unaffected by this bug. Verified by re-reading the function before shipping.

## Session 2 — Realtime Tenant Overload Fix (v1.8.4)

### Goal

URGENT user-facing fix. OmniFrame Agent v1.8.2 was contributing to a Supabase Realtime Presence overload that was blocking users from loading the OmniFrame web app. Diagnosis confirmed:

1. Supabase Realtime Presence GenServer crashing for tenant `c9d89a74` (Presence_shard112 timeouts on `:track` calls from the customer portal).
2. GoTrue `/user` requests at 2.2 seconds (should be <100ms). Users stuck at sign-in.
3. Agent's 5s clean-close Realtime reconnect cycle (12+ reconnects/min) compounded the tenant Realtime load.
4. Empirically: closing the agent immediately restored web-app loading.

The v1.8.0 circuit breaker was too lenient (5 closes in 60s = trip, then 5min auto-retry that re-tripped on a chronically degraded tenant — forever cycle).

### What shipped — v1.8.4

#### `omni_agent/agent.py` (+~190 LOC, banner update)

- **Tighter clean-close circuit breaker**: window 60→30s, threshold 5→2 closes, min-connect-age 30→15s. Trips after 2 spurious closes in 30s instead of 5 in 60s.
- **Exponential auto-retry cooldown**: replaced fixed 5min retry with a doubling ladder (30min → 60 → 120 → 240 → 360min cap = 6h) keyed off a new `_realtime_reset_state["consecutive_trips"]` counter that only resets on a 60s+ stable connection. New `_compute_realtime_cooldown_seconds(consecutive_trips)` helper.
- **`OMNIFRAME_DISABLE_REALTIME=1` escape hatch**: new `_is_realtime_disabled_via_env()` helper. When set at startup, `_start_realtime_subscription()` returns immediately, `state.realtime_disabled = True` is set, and the boot prints `[boot]   Realtime: DISABLED via OMNIFRAME_DISABLE_REALTIME=1 — using polling-only mode (job poller 5-15s, backfill poller 60s).` The reset loop is also skipped (no point spinning when Realtime is intentionally off).
- **Slower reconnect ladder**: initial delay 5→15s, additive +5s per attempt (was multiplicative ×2), max stays at 60s, resets to 15s only after `connect_age >= 60s` of stable connection. Removed the unconditional `backoff = min_backoff` reset on subscribe success.
- **Extended `/realtime/status` payload**: three new stable-contract fields — `consecutive_trips: int`, `next_retry_seconds: int`, `recommended_action: str` — plus `realtime_disabled_via_env: bool`. `recommended_action` switches between four states (env-disabled / chronic-trips / single-trip / healthy) so dashboards can show a useful next-step instead of just "degraded".
- **`_disable_realtime_subsystem` updated**: increments `consecutive_trips`, computes new cooldown, stores `next_retry_at` timestamp, prints the new banner with countdown + ISO timestamp + the `OMNIFRAME_DISABLE_REALTIME=1` hint.
- **`_start_realtime_circuit_reset_loop` rewritten**: polls every 30s and respects `next_retry_at` (was fixed 5min sleep). Skips spawning entirely if `OMNIFRAME_DISABLE_REALTIME=1` is set.
- **`AGENT_VERSION = "1.8.4"`** with v1.8.4 banner prepended to the giant inline comment string.
- **New capability** `realtime-aggressive-degradation` advertised in `/health.capabilities`.
- **Boot banner updated**: shows new clean-close thresholds, exponential cooldown ladder, additive reconnect delay, and the env-var hint. Also shows the disabled-via-env state when applicable.

## Browser-side companion fix — Customer Portal Presence hardening

Follow-up to the v1.8.4 agent fix above. The agent fix removed the agent's contribution to the `Presence_shard112` GenServer crash on tenant `c9d89a74`, but the customer-portal browser sessions were the ORIGINAL named source in the Supabase logs. Audited `src/lib/presence/*` end-to-end and shipped a six-fix surgical hardening pass so a future flare-up doesn't re-wedge the shard.

### Findings

The Customer Portal page itself doesn't use Presence directly — the load comes from the always-on `<PresenceProvider>` in `AuthenticatedLayout`, which wraps EVERY authenticated page. CSRs leave the customer portal tab open all day, so its sessions accumulate the most pressure on the shared org-wide channel `presence-org-{org_id}` (which consistently hashes to shard 112 for that tenant).

Nine specific issues identified — see [[Debug/Fix-CustomerPortal-Presence-Tenant-Overload]] for the full audit.

### What shipped (Phase A)

1. **`scheduleTrack()` debouncer/coalescer** — every `.track()` / `.untrack()` routes through a 1500ms coalescer. Reconnect storms (supabase-js firing `SUBSCRIBED` repeatedly on a flaky tenant) collapse from N RPCs to ~1 RPC per 1.5s. Always sends the LATEST payload.
2. **Channel-error circuit breaker** — mirror of agent v1.8.4: 3 errors in 60s → trip → untrack + `removeChannel` → 5→30 min exponential cooldown → auto re-join. `consecutive_trips` resets to 0 once a connection survives 60s.
3. **Visibility-aware DB heartbeat** — 60s foreground / 5min hidden / fully off when "Appear Offline" selected. Cuts CSR session writes from ~480/day to ~95/day.
4. **`VITE_PRESENCE_DISABLED` env-var kill switch** — browser-side equivalent of `OMNIFRAME_DISABLE_REALTIME=1`. Build-time inlined; service no-ops at boot when set.
5. **Kiosk/RF route opt-out** — `/rf-*`, `/timeclock(app)?`, `/customer-portal*` routes skip channel join + heartbeat. Provider stays mounted (consumer hooks don't crash); service stores identity but no-ops the network work.
6. **Sync throttle 500ms → 1000ms** — a logistics app does not need sub-second "who's online" freshness. Halves re-render churn at zero UX cost.

## Roadmap Sprint — SAP fleet WS migration + cross-tenant fix + ops metric

Sprint after the Phase A/B2/B3 presence work. Three deliverables from [[Roadmap-Rust-WS-Unlocks]] sequenced lowest-to-highest risk so the security fix lands cleanly even if the bigger migration breaks. All three shipped same-day.

### Shipped

1. **Cross-tenant filter fix** — [[Fix-ScheduledJobsTab-Cross-Tenant-Filter]]. The `sap-agent-schedules-tab` Realtime channel had no `organization_id` filter, leaking schedule UPDATEs across tenants as a metadata side-channel (the row list itself was always re-fetched org-scoped via `eq('organization_id', oid)`, but channel timing/fan-out leaked). Added `filter: organization_id=eq.${orgId}` + `if (!orgId) return` guard. Pure FE; no Rust delta; no migration. Independent of the bigger migration so it ratchets the security floor regardless.

2. **`RecvError::Lagged` metric** — [[Add-WsEvent-Lagged-Metric]]. The per-socket recv loop in `handle_socket()` previously absorbed `tokio::sync::broadcast::error::RecvError::Lagged(n)` silently via `while let Ok(…)`. ## Phase 5 — Material Master mutations behind defence-in-depth wrapper (rust-work-service full-integration plan)

After Phase 7's queue-claim path centralization, shipped the comprehensive plan's **Phase 5**: server-side defence-in-depth wrapper for the highest-risk SAP Testing surface. Material Master mutations (MM02 storage-bin / storage-types) used to flow `browser → agentFetch('localhost:8765/sap/material-master-bin') → SAP GUI` directly, with the FE writing the audit row via `logSapAudit(...)` after the agent returned. That left five gaps: no server-side role gate, no concurrency guard, no rate limit, audit row could be skipped on browser crash, and the audit writer was the same JWT the user holds. Phase 5 closes all five at the rust-work-service boundary.

### What shipped

**New endpoint** — `POST /api/v1/sap-mutations/material-master`. Five-step pipeline before the `sap_agent_jobs` INSERT:

1. **Role gate** — `admin` / `superadmin` / `super_admin` / `sap_mutator` (or service-key caller). Today's RBAC only carries `admin` + `superadmin`; `sap_mutator` is reserved for a future fine-grained role.
2. **Per-material concurrency lock** — `presence:lock:material:{org}:{material}` SET NX EX 300. Two admins racing on the same material → 409 for the second.
3. **Per-org rate limit** — `ratelimit:sap-mutations:{org}` INCR + EXPIRE 60s on first hit. > 10 in window ⇒ 429 with `Retry-After` header (uses remaining counter TTL when available).
4. **Pre-flight `sap_audit_log`** — INSERTed at `status='pending'` BEFORE the job. Captures the full intent (material, plant, fields, endpoint) so a forensic auditor sees "who tried what when" even if the agent crashes.
5. **`sap_agent_jobs` INSERT + audit row link** — same DB transaction. Job payload carries `_audit_log_id`; audit row's `job_id` column set in the same tx so the listener can resolve via either path.

**Migration 277** — extends `sap_audit_log_status_check` with `pending` / `completed` / `failed` / `canceled` (note: `canceled` matches `sap_agent_jobs.status`'s single-l spelling). Legacy `success` / `error` / `warning` triplet stays accepted so existing FE writers don't have to migrate in lock-step. Adds `idx_sap_audit_log_job_id` partial index.

Errors are logged at `warn!` and swallowed; the listener task NEVER dies on a patch failure. Patch awaits BEFORE the WS broadcast so a single tick is serial.

### Lock release semantics

- 5-min TTL — matches the upper bound of an MM02 transaction with the agent.
- Released early on rate-limit overflow (so a 429 doesn't accidentally lock the user out for 5min on a recoverable error).
- Left in place on success — expires naturally so a second admin can't immediately fire another MM02 on the same material. That's the intended defence-in-depth behaviour.
