## Session 2 — SAP Session Pinning (v1.7.9)

### Focus
User ran into a class of bug where multiple SAP GUI sessions were open on the same Citrix box (different system / client / user) and the OmniFrame Agent would greedily attach to whichever session it scanned first via `_auto_select_valid_session()`. Manual SAP work in the OTHER session would get hijacked by the agent's job poller as soon as a queued job fired. Built session-pinning so the agent can be dedicated to ONE specific SAP session while the rest of the user's SAP windows are safe.

### Actions
1. **`omni_agent/agent.py` (v1.7.8 → v1.7.9)**
   - New `AgentState.pinned_session: dict | None` (shape: `{conn_idx, sess_idx, system, client, user, pinned_at, by_criteria}`).
   - Persisted to `%APPDATA%\OmniFrameAgent\config.json` via `load_config()` / `persist_config()` so the pin survives EXE rebuild + restart.
   - New `_read_session_identity()` and `_find_pinned_session()` helpers — two-strategy lookup (exact indexes, then criteria scan).
   - `_auto_select_valid_session()` now returns `(None, None)` when pinned and the pinned session is unavailable — the agent stays disconnected rather than silently attaching to a different session. Default behaviour (no pin) unchanged.
   - `_restore_pinned_session_indexes()` primes the SAP scripting globals from a restored pin so `sap_connect()` tries the pinned coordinates first.
   - Two new endpoints: `POST /sap/select-session` (validates + pins + persists) and `POST /sap/unpin-session` (clears + persists).
   - `GET /sap/sessions` augmented — each session entry now carries `system`, `client`, `user`, `transaction`, `pinned: bool`, `is_active: bool`; top-level response also echoes `pinned_session` so the picker can render the pinned criteria even when the underlying SAP session isn't currently visible.
   - New capability `sap-session-pinning` advertised in `/health.capabilities` so the frontend can capability-gate the new picker.
   - New boot banner: `[boot] Pinned session config loaded: sys=PRD client=800 user=U8206556 ...` or `[boot] No session pin — auto-select mode active.`

2. **`src/features/admin/sap-testing/components/sap-session-picker.tsx` (NEW)**
   - Compact pill + dropdown shared between Inventory Management and Agent Triggers tabs. Lock icon + env-coloured criteria pill when pinned; open-lock + neutral pill in auto mode.
   - Dropdown lists every available SAP session with `system / client / user` slug + connection name + transaction code; currently-pinned row gets a checkmark + 'pinned' badge; currently-active row gets an 'active' badge.
   - Amber callout when the pinned session isn't currently visible ("Open it in SAP Logon or unpin to resume auto-select").
   - Footer: "Unpin (return to auto-select)" when something is pinned.

3. **`inventory-management-tab.tsx`** — `AgentStatusBar` renders the picker when `hasCapability(health, 'sap-session-pinning')`; older agents fall back to the v1.7.8 inline `<select>` (no functionality regression).

## v1.8.0 Rebuild + Upload (Realtime resilience)

User reported v1.7.9 was in an infinite "clean close" Realtime reconnect cycle (~100+ cycles in their console.txt over 10 minutes, ~5s per cycle). Confirmed via the log pattern `[realtime] connected ... [realtime] listen() returned cleanly — socket closed without exception; reconnect in 5.0s` repeating endlessly. v1.7.1's circuit breaker only counted exceptions so it never tripped despite the agent being functionally wedged. Most likely cause: Citrix corporate proxy (Netskope/ZScaler) idle-closing WebSocket connections faster than Supabase Realtime's default 30s heartbeat.

### v1.8.0 fix (3 layered defenses)
1. **Tighter WebSocket heartbeat** — passes `hb_interval=10` to AsyncRealtimeClient (verified achievable via `inspect.signature`; wrapped in TypeError fallback for older bundled wheels). 10s sits well under typical corporate-proxy idle windows. May eliminate the cycle on its own.
2. **Clean-close circuit breaker** — parallel to the v1.7.1 exception breaker. Tracks "spurious closes" (clean close <30s after connect, before any heartbeat could plausibly have kept it alive). 5 in a 60s window trips the same `_disable_realtime_subsystem` path → polling-only fallback for 5min → auto-retry. At ~5s cycling this trips after ~25s instead of forever.
3. **`/realtime/status` endpoint** — read-only diagnostic with `{connected, circuit_tripped, fallback_mode, spurious_close_count_60s, exception_count_60s, last_event_at, uptime_seconds, agent_uptime_seconds, version, details}`. Token-exempt. Frontend can use to render an accurate "Realtime: degraded" pill instead of flickering "agent disconnected".
4. **Escalation log** — at 3+ spurious closes in 30s, ONE WARN per minute names the proxy hypothesis: `[realtime] WARN — N spurious clean closes in last 30s (likely corporate proxy idle-closing WebSocket faster than heartbeat)`.
5. New capabilities: `'realtime-clean-close-detection'`, `'realtime-status-endpoint'`.

## Session 3 — Realtime Clean-Close Reconnect Cycle (v1.8.0)

### Focus
User's freshly-deployed v1.7.9 agent was healthy on SAP but the agent console at `MacWindowsBridge/console.txt` showed a non-stop reconnect cycle of 100+ pairs over ~10 minutes:
```
[realtime] connected to wss://...realtime/v1 (subscribed to public.sap_agent_jobs + public.rf_putaway_operations + public.shipment_queue + public.work_tasks for org ...)
[realtime] listen() returned cleanly — socket closed without exception; reconnect in 5.0s
```
User-visible symptom: *"SAP session is not persistent and it's losing connectivity. I try to refresh. It will show for a second, then disconnect."* — but SAP itself was fine on the agent side. The Realtime storm was monopolizing the GIL on the asyncio thread (re-handshaking + re-subscribing 4 channels every 5s), starving the synchronous heartbeat thread + flooding `/sap/sessions` polls so the frontend's agent-detection mis-rendered the agent as flaky.

### Root cause
Corporate proxy on the Citrix VDA (Citrix → Netskope → internet → Supabase) idle-closing the WebSocket faster than the realtime library's 25s default `hb_interval` heartbeat. The library observes a clean WebSocket close, returns `listen()` without raising, and our outer reconnect loop reconnects every 5s — forever. The v1.7.1 `_RealtimeCircuitBreaker` only counts EXCEPTIONS (asyncio task crashes, suppressed `ValueError("Set of Tasks/Futures is empty.")`, `ConnectionClosedError`); a clean return is silent to it, so the cycle never tripped the polling-only fallback.

### Actions
1. **`omni_agent/agent.py` (v1.7.9 → v1.8.0)**
   - **`_RealtimeCleanCloseTracker`** — new sliding-window counter, parallel to `_RealtimeCircuitBreaker`. 60s window, 5-spurious-close threshold, ignores closes with `connect_age >= 30s` (those are normal heartbeat-mediated drops). 5+ in 60s trips the SAME `_disable_realtime_subsystem` path the exception breaker uses, so polling-only fallback (5min, then auto-retry) takes over.
   - **`hb_interval=10` on `AsyncRealtimeClient`** — overrides the library default (25s) to 10s so a typical Netskope/ZScaler/Citrix idle timer in the 10-30s range gets reset before it fires. Wrapped in `try/except TypeError` so older bundled `realtime` wheels that don't accept the kwarg still construct.
   - **Connect-age tracking** — `_realtime_state["connected_at"]` is set after `subscribe()` succeeds, cleared on every disconnect path. The `[realtime] listen() returned cleanly …` log line now includes `… after Xs` so the user can see how long the connection actually lived without consulting the new endpoint.
   - **Escalate WARN log** — when 3+ spurious closes accumulate within 30s, emit ONE WARN per minute pointing the user at the corporate-proxy hypothesis. Throttled, so a sustained storm only emits one WARN per minute, not per cycle.
   - **`GET /realtime/status`** — read-only diagnostic endpoint returning `{connected, circuit_tripped, fallback_mode, spurious_close_count_60s, exception_count_60s, last_event_at, uptime_seconds, agent_uptime_seconds, version, details{...}}`. Token-exempt (added to `_TOKEN_EXEMPT_PATHS`) so a frontend status pill can render before /supabase/login completes. Read-only — never mutates state.
   - **Version banner bump** — `AGENT_VERSION = "1.8.0"` with the v1.8.0 description prepended to the existing carry-over chain. Two new capabilities: `realtime-clean-close-detection`, `realtime-status-endpoint`.
   - **Tracker reset on circuit-reset** — when the 5-min reset daemon re-arms Realtime, also `_realtime_clean_close_tracker.reset()` so a stale 4-close window from before the trip can't insta-trip again.
   - **NO existing handler touched. NO trigger semantics changed. NO migration. NO RLS. NO frontend logic touched beyond `LATEST_AGENT_VERSION`.**
   - LOC delta: ~+260 lines in `agent.py`.

2. **`src/features/admin/sap-testing/lib/agent-fetch.ts`** — `LATEST_AGENT_VERSION = '1.8.0'`. No other frontend changes — a "Realtime: degraded" status pill that consumes `/realtime/status` is deferred to next session.

3. **Build + copy** — AST parsed clean (`python3 -c 'import ast; ast.parse(...)'` exit 0); `pnpm build` exit 0 in 11.08s; `agent.py` copied to `/Users/jaisingh/Downloads/MacWindowsBridge/Omni-Agent/agent.py`; copy re-AST-parsed clean.

### Verification
- `python3 -c 'import ast; ast.parse(open("omni_agent/agent.py").read())'` — exit 0
- `python3 -c "from realtime import AsyncRealtimeClient; import inspect; print(inspect.signature(AsyncRealtimeClient.__init__))"` — confirmed `hb_interval=25` default exists in 2.x signature

### Why
The v1.7.1 fix correctly identified that *exception-driven* Realtime failure modes need a circuit breaker. v1.8.0 closes the *clean-close* gap that v1.7.1 didn't address: corporate-proxy idle-close is the most common production failure mode we'll see beyond v1.7.1's library-bug case, and it produces NO exceptions for the breaker to count. ### Root Cause of Bug 1 — Not the Proxy, A Missing Table

```sql
SELECT pubname, schemaname, tablename FROM pg_publication_tables
WHERE pubname = 'supabase_realtime' ORDER BY tablename;
-- returns sap_agent_jobs, rf_putaway_operations, work_tasks (but NOT shipment_queue)

SELECT table_schema, table_name FROM information_schema.tables
WHERE table_schema='public' AND table_name='shipment_queue';
-- returns zero rows
```

`shipment_queue` was referenced in `_HARDCODED_TRIGGERS` (`builtin-shipment-queue`, v1.7.2) and in `_start_realtime_subscription` (channel 3 subscribe), but the table **does not exist in the DB at all** — not in the publication, not in `information_schema.tables`. Subscribing to it caused Supabase Realtime to clean-close the WebSocket immediately after `subscribe()`.

Full analysis + verification steps appended to [[Debug/Fix-Realtime-CleanClose-Cycle]].
