## Inventory Management — Fleet routing gating bug fix

Same-day follow-up to the Local / Fleet routing toggle above. User reported that the round-6 toggle correctly flipped the **dispatch** path but missed the **availability gating**: with toggle on Fleet + a valid online agent picked (`USINDPR-CXA106V v2.0.0 — 77 caps`), three local-mode-centric surfaces still surfaced as if the local agent was the blocker — yellow "Agent not detected" banner at top, "Agent offline" status row, and per-query "Start the SAP agent" pill on the LT10 form. The Run Query button was disabled too.

Root cause: every gate keyed off `agentStatus === 'connected'` (which reads `agentDetection.available + .authenticated`, i.e. the LOCAL agent on `localhost:8765`). Fleet mode bypasses the local agent entirely — the gates didn't know.

Details: see [[Implement-Inventory-Management-Fleet-Routing]] § "2026-05-09 follow-up — Readiness gating bug fix".

### What landed

- **New `ready(capability?: string)` function** on `useExecutionMode()` returning `{ ok, reason, surface }`. Local mode checks `useAgentDetection().available + .authenticated + .hasCapability`; fleet mode checks `fleetAgent != null` (live-resolved) + capability advertisement. Reason copy ALWAYS names BOTH knobs the user can turn (e.g. "switch to Fleet Agent mode" in a local-mode error, "switch to Local Agent mode" in a fleet-mode error) so the user always has an out — the previous "Start it from the One Click Ship tab" copy was misleading in fleet mode.
- **Distinct from `canDispatch`** which keeps its existing semantics (fleet-only validator consumed by `dispatch()` for routing decisions; local mode short-circuits to true so dispatch falls through to `agentFetch`). `ready()` is the UI gating signal — its job is to decide WHETHER TO RENDER warnings / disable buttons / show pills.
- **Migrated every gating site** in `inventory-management-tab.tsx`:
  - `<AgentNotDetectedBanner>` — only renders in local mode (`!executionMode.isFleet`).
  - Auto-update banner — only renders in local mode (it's about the LOCAL agent's installed version).
  - `<AgentStatusBar>` `missing` / `unauthenticated` branches — neutral copy in fleet mode ("Local agent offline · fleet routing active." instead of "Agent offline — start it from the One Click Ship tab.").
  - Detail Card per-query status pill — uses `queryReady.reason` (mode-aware) instead of hard-coded local-only copy.
  - `canRun` — `queryReady.ok && !isRunning` (was `agentStatus === 'connected' && !isRunning`).
  - Run / Preview / Refresh button `disabled` + `title` — `!canRun` and `queryReady.reason` (single source of truth; dropped the parallel `capMissing` ladder).
  - LT01 chip — gated on `executionMode.ready('transfer-inventory')` instead of local-only `hasCapability`.
  - `dryRunCapAvailable` — uses `executionMode.ready(dryRunCapability).ok` (consults local OR picked fleet).
  - `<RowActionsMenu>` per-action gate — new `checkActionReady` prop threaded from `executionMode.ready` through `<ResultsCard>` so dropdown items honour the toggle (previously every row action showed "needs update" in fleet mode).
  - `<BatchModePanel queueAvailable>` — `executionMode.isFleet || hasCapability(...)` so the queue checkbox is enabled in fleet mode (queue is the fleet path).

### Fix 1 — Dark page background cuts off below the Query Library

**Symptom (dark mode).** On the LT24 Timeline view, scrolling the right pane past the Query Library's sticky height left the area below painted in the html canvas default (light grey) instead of the dark theme. Visually it looked like the page background "ended" at the Query Library height.

## LT24 Journey/Timeline — compact + elegant pass

User feedback on the new `<TransferOrderHistoryView />` (`src/features/admin/sap-testing/components/transfer-order-history-view.tsx`):

> The card columns are very large, and I believe this can look much more robust with a bit more of a compact layout but elegant at the same time.

The round-1 visuals (per [[Implement-LT24-History-Trail]]) used full Card chrome per TO + heavy hover lifts + stacked stat ribbon — vertically generous (~80–120px per movement), low information density. This pass tightens the same surfaces toward a Linear/Vercel-style robust analytics feel without dropping any data.

### What changed

| Surface | Before | After |
|---|---|---|
| **Stat ribbon** | `<Card>` p-4, `text-2xl` numbers, `gap-3`, hover-lift transform, `h-9 w-9` icon-tile | bordered tile (`border-border/40 bg-card/50`) `px-3 py-2`, `text-xl` numbers, `gap-2`, color-only hover, plain icon |
| **Toolbar** | full Card chrome, `text-xs` description, `h-8` toggle, `min-w-[220px]` search | `border-border/40 bg-card/50 shadow-none` Card, `line-clamp-1 text-[11px]` description, `h-7` toggle (`h-6` triggers), `max-w-xs` search |
| **Filter chips** | `px-2.5 py-0.5` | `h-6 px-2` with `hover:bg-accent` (was `hover:bg-muted`) |
| **Journey card** | Card with hover lift+shadow, stacked header (TO+status+mvmt-pills row, then material row, then qty column on right), per-leg trail with status-pill on each, footer block with icons + bordered separator | Hairline `border-b border-border/40` row inside a single bordered wrapper. Single-line header with `·` separators (status dot · TO badge · mvmt pills · material · description · ml-auto qty + leg count). Tight bin trail row (`size='sm'` chips + 12px arrows).
