# Session 2026-06-06

## Fix: kit number truncated to type on the cover sheet

The Kit Build / Cover Sheet `KIT :` title showed `KIT : Stack` for kit number
`431, 432 LF Stack 3 (A)`. Cause: the title ran the kit number through
`getKitTypeLabel`, which collapsed any STACK/SEAL/BEARING/GASKET-containing
value to that single word. Fixed to render the full `kitData.kitNumber`;
removed the now-unused helper. `tsc` + ESLint clean; FE-only. See
[[Debug/Fix-Build-Sheet-Kit-Number-Collapsed-To-Type]].

## Kit Build Plans: Expedites tab + tab renames

Renamed the grid tabs to **Open Work Kits** / **Completed Kits** and added an
**Expedites** tab for stand-alone single-part expedites (`engine_program =
'EXPEDITE'` — mode 2 of `addExpediteToKit`). Threaded `engine_program` through
`getKitGridData` → `KittingGridRow`; manager splits into expedite / open / completed
memos (expedites excluded from both kit tabs, any status). Expedites tab is
read-only. Fixed `handlePriorityChange` to merge the reordered open-work subset
back by **id** (was keeping only completed rows — would have dropped expedites).
`tsc` + ESLint clean; FE+service only, no schema change. See
[[Kit-Build-Plans-Expedites-Tab]].

## Add Expedite Part — TO import flow

Reworked the expedite dialog (renamed **Add Expedites to Kit Build Plan** →
**Add Expedite Part**) to use the same clipboard TO import as the normal Add
to Kit Build Plan. Dropped the manual part/qty/description fields — each
imported TO row becomes one stand-alone expedite part in the Expedites tab
(material → part number, qty, TO number stored). Only delivery time (+ reason
/ date) is entered, applied to all imported parts. New service
`addExpeditePartsFromTOs` loops `addExpediteToKit` (now accepts
`transferOrderNumber`). Removed the attach-to-Kit-PO option. `tsc` + ESLint
clean; FE+service only. See [[Add-Expedite-Part-TO-Import]].

## Fix: Kit Status stat cards counted parts, not kits

`getStatistics` counted `RR_Kitting_DATA` rows per status, but that table has
one row per TO line (part), so Pending/In Progress/Completed were inflated
(parts, not kits). Changed it to count **distinct `kit_serial_number` per
status** (`countKitsByStatus`) — status is replicated across a kit's rows so
this is exact. `totalRecords` left as a row count (the "Kit PO Numbers" card
already shows kit count). `tsc` + ESLint clean; service-only. See
[[Debug/Fix-Kit-Status-Stats-Counting-Parts-Not-Kits]].

## Fix: kitting grid + stats capped at 1000 rows

`getKitGridData` and `getStatistics` each issued a single unscoped select,
hitting Supabase's 1000-row cap — with one RR_Kitting_DATA row per part, large
queues were truncated (missing kits, wrong counts). Applied the same
range-pagination loop the inbound-scan / putaway-log services use: 1000-row
`.range(from, to)` pages until a short page, with a secondary `id` sort for
stable paging. `tsc` + ESLint clean; service-only. See
[[Debug/Fix-Kitting-Grid-Stats-1000-Row-Cap]].
