# Session 2026-05-11

## Summary

Closed the May 7 PgListener resilience audit's open follow-up by
mirroring the rust-side keepalive + watchdog pattern onto the Python
agent's WebSocket client. Net effect: the agent's WS connection to
`rust-work-service /ws` is no longer susceptible to Citrix VDA / corp
proxy half-open close — worst-case recovery time is now bounded at
~75s on defaults instead of "until the user presses something in the
agent console".

## Work completed

### Resilient work-service WS client (Python agent)

Detail: [[Implementations/Implement-Resilient-Work-Service-WS-Client]].

- Full rewrite of `omni_agent/work_service_ws.py` (281 LOC → 650 LOC,
  +369 LOC mostly docstring + watchdog + new public methods + env
  parsing) preserving the entire existing public API
  (`WorkServiceWsClient` ctor signature, `start/stop/is_connected/`
  `last_event_at/fallback_reason`, `WORK_SERVICE_URL` constant).
- **Two-layer keepalive stack**:
  1. Library-level via `websockets.connect(ping_interval=20s,
     ping_timeout=10s)` — protocol-level ping every 20s, library
     raises `ConnectionClosedError` on no pong in 10s.
  2. App-level via sibling asyncio `_watchdog_loop` task — wakes
     every 15s, force-closes the socket via `ws.close(code=1011,
     reason="agent watchdog timeout")` if
     `_last_message_received_at` gap exceeds 60s.
- **Reconnect ladder** retightened from "5s additive +5s, 60s cap" to
  bounded exponential `1s → 30s` (matches rust-side
  `RECONNECT_BACKOFF_MAX`).
- **Exception handling** tightened — `(OSError,
  asyncio.TimeoutError)` now listed explicitly so reconnect log
  carries precise class names; watchdog-driven closes get rebadged
  from `ConnectionClosedError` to `WatchdogTimeout` (cycle-scoped
  `state["tripped"]` dict survives into every `except` arm).
- **Self-reporting surface**: 4 new public methods (`reconnect_count`,
  `watchdog_trips`, `last_message_received_at`,
  `last_reconnect_reason`) ready for a future `/health` JSON
  extension. No agent metrics endpoint today — logged as open
  follow-up.
- **Tunable env vars** with sensible production defaults:
  `OMNIFRAME_WS_PING_INTERVAL_SEC` (20),
  `OMNIFRAME_WS_PING_TIMEOUT_SEC` (10),
  `OMNIFRAME_WS_WATCHDOG_INTERVAL_SEC` (15),
  `OMNIFRAME_WS_WATCHDOG_TIMEOUT_SEC` (60). Garbage input falls back
  to defaults with a WARN. Sanity check at import time WARNs if
  `watchdog_timeout < 2 × ping_interval` (would spuriously trip
  before the library can surface its own close).
- **Boot banner**: one-line resilience-posture summary printed once
  per process at `WorkServiceWsClient.start()`. Mirrors the rust-side
  `info!("resilient PgListener starting")` log line shape.
- **Reconnect log shape**: every reconnect emits `[work-ws] reconnect
  #<N> (last message <X>s ago, reason=<class>, watchdog_trips=<M>);
  sleeping <backoff>s`.
- **NO** `AGENT_VERSION` bump (still 2.0.0). NO message-protocol
  changes. NO changes to `agent.py` or `rust-work-service`. NO new
  external deps (`websockets>=11.0` already pinned;
  `inspect.signature(websockets.connect)` confirms `ping_interval` /
  `ping_timeout` available in the production v15.0.1).

## Capacity-ceiling analysis (~19:30 UTC)

Strategic follow-on to the morning's tactical fix. User asked "I thought I was able to handle 2,000 concurrent users and it doesn't seem to be the case." Answered with a read-only capacity analysis. Full ADR: [[Decisions/ADR-Capacity-Ceiling-2k-Users]]. Companion workspace canvas at `analysis/capacity-2k-users.canvas.tsx` (mirrored to the IDE-rendered canvas under the workspace `canvases/` folder).

### Headline answer

- **2,000 concurrent users is NOT reachable on the current shape** (Pro Small + single replicas + single uvicorn worker + production defaulting to Supabase Realtime presence).
- Realistic platform ceiling today: **~1,200–1,500** in theory, but the FIRST wall hits at **~80–150 concurrent tabs** because production presence still routes through Supabase Realtime even though the Rust replacement shipped 2026-05-06.

### Critical live finding

The May 6 [[Decisions/ADR-Presence-Architecture-Next-Steps]] Option 2 is shipped end-to-end ([[Implementations/Implement-Presence-On-Rust-Option-2]]) — `PresenceServiceRust`, REST routes, evictor, three `WsEvent` variants, FE facade in `src/lib/presence/index.ts`, build-time selector via `VITE_PRESENCE_MODE`. **But the env var is NOT set on `onebox-ai-logistics`**, so the build defaults to `'supabase'` and production still hits the wedging shard. Today's Realtime logs show `Presence_shard112` GenServer.call timeouts and `ClientPresenceRateLimitReached` events firing 14:25–16:30 ET — the wedge is back five days after the May 6 ADR shipped.

### Bind order (post-DB-cap fix)

1. **80–150 tabs**: Supabase Realtime presence shard wedge (TODAY)
2. **300–500 active users**: FastAPI single uvicorn worker (`start.py` calls `uvicorn.run(app, ...)` with no `workers` arg; Railway env has no `WEB_CONCURRENCY`)
3. **150–200 tabs**: Supabase Realtime concurrent socket cap (200/tenant on Pro × tabs/user) once grandfathered channels are counted
4. **600 RPS / 1,200 users**: PostgREST authenticator pool exhaustion
5. **1,500+ users**: Postgres login storm — same SQLSTATE 53300 we just fixed, returns under load even with the new 30-cap on rust-core

### Top three actions to reach 2k (zero of these landed today — out of scope for read-only analysis)

1. Set `VITE_PRESENCE_MODE=rust` on `onebox-ai-logistics` Railway env and redeploy. 5 min, zero code, removes the binding wall today.
2. `WEB_CONCURRENCY=4` env + 1-line `start.py` patch (`workers=int(os.environ.get('WEB_CONCURRENCY', 1))`). 4× per-replica capacity.
3. Supabase Pro → Team tier (raises `max_connections` 120→200/400, Realtime sockets 200→500–10k, enables read replicas).

### Discovery — why setting `VITE_PRESENCE_MODE=rust` on Railway alone is a no-op

`onebox-ai-logistics`'s Dockerfile only declared ARGs for 8 specific VITE_* vars (`VITE_SUPABASE_URL`, `VITE_API_URL`, `VITE_RUST_CORE_*`, `VITE_WORK_SERVICE_*`, `VITE_STREAMING_SERVICE_URL`). Without `ARG VITE_PRESENCE_MODE`, Railway's build context never received it, so `pnpm run frontend:build` saw `import.meta.env.VITE_PRESENCE_MODE === undefined` and the IIFE in `src/lib/presence/constants.ts:149-157` resolved `PRESENCE_MODE` to `'supabase'` regardless of what was set on the runtime env. The `3e4e1c2` patch closes this gap.

### Crash-loop — `25a270f6` FAILED on `WEB_CONCURRENCY=4`

The deployed `start.py` in Railway's source (different from `main` HEAD because the user pushes via `railway up`, not GitHub auto-deploy) is a partial-WIP version that already reads `WEB_CONCURRENCY` and passes `workers=N` to `uvicorn.run(...)` — but still passes `app` as an object instead of `"api.main:app"` as a string. With `workers > 1` + an app object, uvicorn emits the warning `"You must pass the application as an import string to enable 'reload' or 'workers'"` and the worker subprocesses cannot bootstrap; the parent exits and Railway's restart policy cycles ten times before the deploy goes FAILED. Production rolled back to `9697adf0` (which Railway kept the original container of from before `WEB_CONCURRENCY=4` was set). Setting `WEB_CONCURRENCY=1` makes any future container restart safe under the deployed code as well.

### Asyncpg pool sanity check (the user asked for this explicitly)

`api/config/connection_pool.py` defaults to `MIN_CONNECTIONS=10` / `MAX_CONNECTIONS=100` per Python process. With 4 uvicorn workers the worst case is 4×100 = 400 conns vs. Postgres `max_connections=120` — **dangerous if active**. **It is NOT active in production**: `onebox-ai-logistics` does not set `DATABASE_URL`, and the lifespan handler in `api/main.py:46-52` skips the pool init when `settings.database_url` is None (a `try/except` swallows the absence). 4 workers therefore add **0 connections** from this pool. No code change to `connection_pool.py` was needed today — option (c) from the ADR "options if pool is too big" list, but for the right reason (pool is dormant, not the multiplier being too aggressive). If `DATABASE_URL` ever gets enabled, follow the `rust-core-service` `DB_MAX_CONNECTIONS=30` precedent and add an env-var override to `connection_pool.py` first.

## rust-work-service Redis Round 2 — v0.1.40 lazy pool (~21:15 ET / 01:15 UTC May 12)

v0.1.39's retry-with-backoff fix from earlier this evening shipped via `railway up` but DID NOT solve the crash-loop — the retry IS firing (Round 2 deploy logs show the new `WARN attempt=1..5 max_attempts=5` lines I added) but every attempt deterministically fails with the same `Multiplexed connection driver unexpectedly terminated- IoError` (~41 s wall-clock per attempt; ~4 min total before the panic at `main.rs:334:21` — `Failed to create Redis pool after 5 attempts`). Two failed deploys: `19f06d71` and `f8b8fbfc` (the latter is still cycling under Railway's 10-restart policy as I write this). The retry loop turned out to be the wrong tool: the failure is **structural, not transient** — Redis Cloud accepts the TCP handshake then kills the multiplexed connection, hitting every new container the same way while the v0.1.36 container's 36 h-old pool keeps serving on its long-lived sockets. Verified Cargo.lock and src/main.rs are byte-identical between v0.1.36 (works) and v0.1.37+ (broken); only Postgres connectivity works from the new container, only Redis is rejected — confirms infrastructure-level root cause (most likely Redis Cloud max-clients hit, OR IP allowlist drift on Railway egress IP rotation across container respawns).

Switched to **`bb8::Pool::builder().build_unchecked()`** (Option B from the prompt) so the pool returns synchronously without touching the network. Boot succeeds regardless of Redis state; first request that calls `pool.get().await` takes the connection hit. Spawned a 5 s best-effort connectivity probe at boot that only logs the result (`Redis connectivity probe OK` / `... failed at boot — service will continue in degraded mode (Redis-dependent endpoints will return 500). Check Redis Cloud max-conns / allowlist.`) and never panics. The presence/entity-focus/trigger-evaluator background tasks already handle Redis errors gracefully (`warn!` + continue per `presence::evictor::run` doc), and route handlers propagate `bb8::RunError` as 500 — same shape as a Redis blip mid-op, which FE already retries. `/health` does not touch Redis (Railway healthcheck always passes); `/health/detailed` still surfaces Redis status as `degraded` 503 if it's down. Bumped Cargo.toml 0.1.39 → **0.1.40**.

`cargo build --release` clean (1m 12 s; same 7 pre-existing dead-code warnings; zero new lints). User runs `railway up` next — v0.1.40 will boot cleanly even if Redis is still down, AND the **larger half of the fix is investigating Redis Cloud directly** (admin console: max-conns vs tier, IP allowlist for `europe-west4-drams3a` Railway egress) — out of scope for this code change.
