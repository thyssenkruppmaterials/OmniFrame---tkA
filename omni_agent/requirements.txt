# Phase D #16 — Agent-Direct Supabase Realtime subscription. The agent
# subscribes to INSERTs on sap_agent_jobs over WebSocket so newly-queued
# work is claimed sub-second instead of waiting on the 60s polling fallback.
# Lightweight standalone (we don't need the full `supabase` package; REST
# + JWT calls are still done via `requests`). Soft import in agent.py —
# missing this dep transparently degrades to polling-only mode.
#
# v1.7.1 — pin floor to 2.29.0 (latest as of 2026-04-24). Older 2.x
# releases (2.0.x – 2.4.x roughly) had a `_reconnect()` bug that
# called `asyncio.wait([])` on an empty pending-tasks set after a
# WebSocket drop, raising `ValueError: Set of Tasks/Futures is empty.`
# The library didn't catch it, so the exception escaped the listen
# task → asyncio default handler → multi-line `Task exception was
# never retrieved` traceback per drop. After ~25min idle on a Citrix
# VDA hibernate or corporate-proxy idle close, agents flooded stderr
# with thousands of tracebacks per minute, drowning the heartbeat
# thread (sap_agents.last_seen_at silently stalled) and the job
# poller. The bug was refactored away around v2.5+ when the listen
# task management moved into `_on_connect` and `_reconnect` stopped
# calling `asyncio.wait()` at all. Pinning to 2.29.0 picks up the
# fix; the v1.7.1 in-agent containment layers (asyncio loop
# exception handler + circuit breaker + 5min auto-recovery) stay as
# defense-in-depth so a future 2.x regression is bounded. NOT v3.x
# (alpha as of 2026-04-09). See [[Debug/Fix-Realtime-Library-CrashLoop]]
# + [[Patterns/Async-Library-Circuit-Breaker]].
realtime>=2.29.0,<3.0
